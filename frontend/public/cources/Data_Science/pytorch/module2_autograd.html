01 — Computation Graph Intuition

What is a Computation Graph?

Every time you do math with PyTorch tensors that have requires_grad=True, PyTorch silently builds a directed acyclic graph (DAG) in the background. Each node in this graph is either a tensor (the data) or a function (an operation like +, *, sin, matmul).

When you call .backward(), PyTorch traverses this graph in reverse — from the output back to the inputs — applying the chain rule of calculus at each step to compute gradients.

Real-World Analogy — A Factory Assembly Line (in Reverse): Imagine a factory: raw materials go in, a product comes out. To figure out how much each raw material contributed to the final product's quality, you trace backwards through the assembly steps — that's exactly what backprop does.

x (input)
→
x² (square)
→
x² + b
→
w(x²+b) = loss

Forward pass: left → right  ·  Backward pass: right → left (chain rule)

Python
import torch

# Simple computation graph
x = torch.tensor(3.0, requires_grad=True)   # Leaf node
w = torch.tensor(2.0, requires_grad=True)   # Leaf node
b = torch.tensor(1.0, requires_grad=True)   # Leaf node

# Forward pass — graph is built as we compute
y = x ** 2              # y = x² = 9
z = w * y + b           # z = 2x² + 1 = 19

print("z =", z.item())
print("z.grad_fn:", z.grad_fn)   # Shows the last operation
Output
z = 19.0
z.grad_fn: <AddBackward0 object at 0x...>
Python
# Backward pass — traverses graph, accumulates gradients
z.backward()

print("dz/dx =", x.grad.item())  # d(2x²+1)/dx = 4x = 12
print("dz/dw =", w.grad.item())  # d(2x²+1)/dw = x² = 9
print("dz/db =", b.grad.item())  # d(2x²+1)/db = 1
Output
dz/dx = 12.0
dz/dw = 9.0
dz/db = 1.0
02 — Controlling Gradient Flow

When NOT to Track Gradients

Tracking the computation graph has a memory and compute cost. There are two common scenarios where you want to turn it off:

Two Gradient Control Patterns
torch.no_grad()  ·  tensor.detach()
no_grad() — context manager that disables gradient tracking for a block of code (inference, evaluation)  ·  .detach() — returns a new tensor that shares data but is detached from the graph
Python
# During inference — don't track gradients (saves memory & speed)
model_output = torch.tensor([0.9, 0.1], requires_grad=True)

with torch.no_grad():
    prediction = model_output.argmax()  # No graph built
    print("Predicted class:", prediction.item())

# .detach() — break the graph connection
x = torch.tensor(2.0, requires_grad=True)
y = x ** 2                  # y is part of graph
y_detached = y.detach()    # y_detached has same value but no grad_fn
print("y.grad_fn:", y.grad_fn)           # PowBackward0
print("y_detached.grad_fn:", y_detached.grad_fn)  # None

# requires_grad flag
frozen = torch.randn(3, requires_grad=False)
print("frozen.requires_grad:", frozen.requires_grad)  # False
Output
Predicted class: 0
y.grad_fn: <PowBackward0 object at 0x...>
y_detached.grad_fn: None
frozen.requires_grad: False

Gradient Accumulation Trap: PyTorch accumulates (adds) gradients by default. If you call .backward() twice without zeroing gradients in between, they will be summed. Always call optimizer.zero_grad() before each backward pass in a training loop.

03 — Linear Regression From Scratch

Putting It All Together

Now let's implement the full gradient descent loop using only tensors and autograd — no nn.Module, no optimisers. This builds deep intuition before moving to higher-level abstractions.

Python
import torch

# --- Synthetic data: y = 2x + 1 + noise ---
torch.manual_seed(42)
X = torch.linspace(-1, 1, 100)
y = 2 * X + 1 + 0.2 * torch.randn(100)

# --- Parameters (what we want to learn) ---
w = torch.tensor(0.0, requires_grad=True)
b = torch.tensor(0.0, requires_grad=True)

lr = 0.1
n_epochs = 200

# --- Training loop ---
for epoch in range(n_epochs):
    # 1. Forward pass: compute predictions
    y_pred = w * X + b

    # 2. Compute loss (Mean Squared Error)
    loss = ((y_pred - y) ** 2).mean()

    # 3. Backward pass: compute gradients
    loss.backward()

    # 4. Update parameters (gradient descent step)
    with torch.no_grad():
        w -= lr * w.grad
        b -= lr * b.grad

    # 5. Zero the gradients for next iteration
    w.grad.zero_()
    b.grad.zero_()

    if (epoch + 1) % 50 == 0:
        print(f"Epoch {epoch+1:3d} | Loss: {loss.item():.4f} | w={w.item():.3f}, b={b.item():.3f}")

print(f"\nLearned: y ≈ {w.item():.2f}x + {b.item():.2f}  (true: y = 2.00x + 1.00)")
Output
Epoch  50 | Loss: 0.0563 | w=1.812, b=0.976
Epoch 100 | Loss: 0.0421 | w=1.934, b=0.996
Epoch 150 | Loss: 0.0405 | w=1.972, b=0.999
Epoch 200 | Loss: 0.0402 | w=1.986, b=1.000

Learned: y ≈ 1.99x + 1.00  (true: y = 2.00x + 1.00)

The 5-Step Loop: Every neural network training loop follows this same pattern: (1) Forward pass → (2) Compute loss → (3) .backward() → (4) Update parameters → (5) Zero gradients. Understanding it at this level makes debugging frameworks like PyTorch Lightning trivial.

Why zero_() has an underscore?

In PyTorch, methods with a trailing underscore _ are in-place operations. They modify the tensor directly without creating a copy. zero_() sets all elements to zero in-place. Other examples: fill_(), add_(), clamp_().

04 — Advanced Graph Concepts

Leaf Nodes, Retain Graph, and Second-Order Gradients

Python
# retain_graph — call backward() more than once
x = torch.tensor(2.0, requires_grad=True)
y = x ** 3

y.backward(retain_graph=True)  # Keep graph alive
print("First backward:", x.grad)   # 3x² = 12

x.grad.zero_()
y.backward()                      # Now free the graph
print("Second backward:", x.grad)  # 12 again

# is_leaf — distinguish learnable params from intermediate results
a = torch.tensor(1.0, requires_grad=True)
b = a * 2
print("a.is_leaf:", a.is_leaf)   # True  — user-created with requires_grad
print("b.is_leaf:", b.is_leaf)   # False — result of an operation
Output
First backward: tensor(12.)
Second backward: tensor(12.)
a.is_leaf: True
b.is_leaf: False
Chain Rule — The Math Under the Hood
∂L/∂x = (∂L/∂z) · (∂z/∂y) · (∂y/∂x)
PyTorch applies this automatically at every node of the computation graph. ∂L/∂x is what you get in x.grad after calling loss.backward().
