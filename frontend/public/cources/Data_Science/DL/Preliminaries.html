Data Manipulation:

Data Manipulation with Tensors

When I first started diving into deep learning, I thought I could just stick with the NumPy arrays I knew and loved. But I quickly realized that to build effective deep learning models, we need something more powerful. We need efficient ways to store and manipulate data that can scale to millions of parameters and run on GPUs.

The foundation of this entire process involves two key steps: acquiring data and processing it once it’s inside the computer’s memory. At the core of modern deep learning frameworks like PyTorch lies the tensor. Think of a tensor as an n-dimensional array on steroids—it extends traditional NumPy arrays with two critical capabilities that changed the game for me:

  1. Automatic Differentiation: They track their own history, allowing us to compute gradi- ents automatically (crucial for backpropagation).
  2. GPU Acceleration: They can live on the GPU, speeding up matrix math by 50x-100x com- pared to CPUs.

Tensor Anatomy

Tensors represent multidimensional arrays of numerical values. Depending on how many ”axes” or dimensions they have, we give them different names. I like to think of them this way:

      • Scalar (0-D): A single point, like a temperature reading (25.0).
      • Vector (1-D): A line of points, like a row of data ([1, 2, 3]).
      • Matrix (2-D): A grid, like a black-and-white image (28 × 28 pixels).
      • Higher-order tensor (n-D): A cube or hypercube, like a batch of color images (Batch ×

Height × Width × Color).

Broadcasting: The Magic Trick

Broadcasting was the hardest concept for me to grasp, but it’s the most powerful. It enables operations between tensors of different shapes.

➔ Example: Broadcasting Shape Rules — Step-by-Step Worked Examples

PyTorch broadcasts by aligning shapes from the right and expanding any dimension of size 1. Here are three cases that trip up most beginners:

import torch

# ── CASE 1: Adding a bias vector to a batch of activations ──

activations = torch.randn(32, 256) # 32 samples, 256 features

bias = torch.randn(256) # one bias per feature

out = activations + bias # bias broadcast along dim-0 → (32,256)

print(out.shape) # torch.Size([32, 256])

# ── CASE 2: Row-normalisation (divide each row by its max) ──

scores = torch.randn(4, 10) # 4 samples, 10 class scores

row_max = scores.max(dim=1, keepdim=True).values # shape (4, 1)

normalised = scores / row_max # (4,10) / (4,1) → (4,10)

print(normalised.shape) # torch.Size([4, 10])

# ── CASE 3: Outer product via broadcasting ──

a = torch.tensor([1., 2., 3.]) # shape (3,)

b = torch.tensor([10., 20.]) # shape (2,)

outer = a.unsqueeze(1) * b.unsqueeze(0) # (3,1) * (1,2) → (3,2)

print(outer)

# tensor([[10., 20.],

# [20., 40.],

# [30., 60.]])

# ── COMMON ERROR: shape mismatch ──

x = torch.randn(3, 4)

y = torch.randn(3) # (3,) aligns to last dim → needs (4,) not (3,)

# x + y --> RuntimeError: shapes (3,4) and (3,) cannot be broadcast

# Fix: y.unsqueeze(1) → (3,1) broadcasts correctly with (3,4)

Memory tip: broadcasting never copies data — it is a virtual view. The only cost is the arithmetic itself, so always prefer broadcasting over explicit torch.repeat() or torch.tile() to save memory.

The mechanism works in two virtual steps:

  2. Perform operation: It does the math on these expanded arrays.

Saving Memory: In-Place Operations

Memory is the bottleneck in deep learning. We update millions of parameters thousands of times. If every update created a new copy of the model weights, we’d run out of RAM in seconds. Standard operations like Y = Y + X allocate new memory for the result. In-place opera-

tions modify the tensor directly in memory.

1

# In-place operations (Good for parameter updates) Z = torch.zeros_like(Y)

before = id(Z)

Z[:] = X + Y # Slice assignment forces in-place update id(Z) == before # Returns True

# Alternative syntax using += before = id(X)

X += Y

Data Preprocessing in Deep Learning

The Critical Role of Data Preprocessing

In the real world of machine learning, the adage ”garbage in, garbage out” holds profound truth. While textbooks often present clean, perfectly formatted datasets, actual data is messy, inconsistent, and full of surprises. Data preprocessing—the transformation of raw data into a format suitable for machine learning models—is not merely a preliminary step; it’s often the difference between a model that works and one that fails spectacularly.

Consider this sobering statistic: data scientists spend approximately 80% of their time on data preprocessing and only 20% on actual model building and evaluation. This imbalance exists for good reason. Raw data typically contains:

  • Missing values scattered throughout the dataset
  • Inconsistent formatting and encoding schemes
  • Outliers and extreme values that distort statistical properties
  • Features on vastly different scales that confuse optimization algorithms
  • Categorical variables that numerical models cannot interpret directly

A poorly preprocessed dataset can lead to model instability, poor generalization, biased pre- dictions, and computational inefficiency. The goal of data preprocessing is to transform raw data into a representation that preserves meaningful information while removing noise and irrelevant details.

Theoretical Framework for Handling Missing Values

Understanding Missing Data Mechanisms

Not all missing data is created equal.

  1. Missing Completely at Random (MCAR): Probability of missingness is uniform. Safe to remove without bias.
  2. Missing at Random (MAR): Missingness depends on observed data (e.g., income missing for younger people). Requires imputation.
  3. Missing Not at Random (MNAR): Missingness depends on the unobserved value itself (e.g., high earners hiding income). Introduces bias difficult to correct.

➔ Example: MCAR vs MAR vs MNAR — Real-World Examples from a Medical Dataset

The three missing-data mechanisms require different handling strategies. Here is a concrete medical scenario illustrating each:

# Dataset: patient records with columns:

# [age, blood_pressure, income, cholesterol, diabetes_diagnosis]

# ── MCAR: Missing Completely At Random ──────────────────────────

# A lab technician accidentally drops 5% of blood samples.

# Which patients lose results is random — unrelated to their health.

# Test: compare demographics of missing vs non-missing groups → no difference

# Action: safe to drop rows or use simple imputation (mean/median)

# ── MAR: Missing At Random ──────────────────────────────────────

# Income is missing more often for younger patients (who declined to share).

# Missingness depends on 'age' (observed) but not on 'income' itself.

# Test: logistic regression of 'income_missing ~ age' → significant

# Action: use multiple imputation (MICE) conditioned on age

# ── MNAR: Missing Not At Random ─────────────────────────────────

# Cholesterol is missing because patients with very high readings

# refused re-testing (the missing VALUE drives the missingness).

# Test: no statistical test can confirm MNAR from observed data alone

# Action: domain expertise + sensitivity analysis; imputation will be biased

# Quick Python check for potential MAR pattern:

import pandas as pd

df = pd.read_csv('patients.csv')

df['income_missing'] = df['income'].isna().astype(int)

print(df.groupby('income_missing')['age'].mean())

# income_missing

# 0 52.3 (non-missing income — older patients)

# 1 31.7 (missing income — younger patients) -> likely MAR

Why this matters for deep learning: if you blindly drop rows with missing values from a MNAR dataset, you systematically exclude the highest-risk patients — your model learns on a biased sample and will under-predict risk for exactly the patients who need it most.

Imputation Strategies

Simple Imputation

      • Mean/Median: Replace numerical missing values with the mean/median. Reduces vari- ance.
      • Mode: Replace categorical missing values with the most frequent category.

Advanced Imputation

      • K-Nearest Neighbors (KNN): Find k nearest samples and impute based on them. Pre- serves local structure.
      • MICE (Multiple Imputation by Chained Equations): Uses regression models to estimate missing values iteratively.

Best Practices and Common Pitfalls

Prevent Data Leakage

A common mistake is applying preprocessing to the entire dataset before splitting. Correct Workflow: 1. Split into Train/Test. 2. Fit preprocessors (calculate mean/std) on Train data only.

3. Transform both Train and Test data using Train statistics.

Outlier Handling

Calculus

Introduction: Why Calculus Matters

In deep learning, we frequently need to understand how small changes in model parameters affect the output. This is where calculus becomes essential. It provides the mechanism to opti- mize our models by telling us exactly how to adjust weights to minimize error. Without calculus, specifically derivatives, training neural networks would be a blind search.

Derivatives and Differentiation

The Rate of Change

The derivative measures the instantaneous rate of change of a function with respect to its input. Formally, for a function f : R → R, the derivative at point x is defined as:

fr(x) = lim f (x + h) − f (x)

h→0 h

This limit represents the slope of the tangent line to the function at point x. When this limit exists, we say the function is differentiable at x.

Example: Consider f (x) = 3x2 − 4x. To find fr(1), we can compute numerically:

As h approaches zero, the difference quotient approaches 2, confirming that fr(1) = 2. Ana- lytically: fr(x) = 6x − 4, so fr(1) = 6(1) − 4 = 2.

Visualizing Derivatives

For the function f (x) = 3x2 −4x, the tangent line at x = 1 has slope fr(1) = 2 and passes through the point (1, f (1)) = (1, −1). The equation of this tangent line is:

y = fr(1)(x − 1) + f (1) = 2(x − 1) − 1 = 2x − 3

This linear approximation is accurate near x = 1 but becomes less accurate as we move away. Such local linear approximations are the basis for many optimization algorithms used in machine learning.

Automatic Differentiation (Autograd)

Deep learning frameworks like PyTorch do not require us to manually derive these gradients. They use Automatic Differentiation (Autograd).

Computational Graphs

Autograd builds a **computational graph** (specifically, a Directed Acyclic Graph or DAG) as we perform operations.

      • **Nodes**: Operations (add, multiply, ReLU, etc.)
      • **Edges**: Tensors flowing between operations

When we compute the forward pass (calculating the loss), the graph is built dynamically. When we call ‘.backward()‘, the framework traverses this graph in reverse order to compute gradients using the chain rule.

Forward vs. Reverse Mode

There are two main modes of automatic differentiation:

  1. **Forward Mode**: Computes derivatives alongside the forward pass. Efficient when in- put dimension < output dimension.
  2. **Reverse Mode (Backpropagation)**: Computes the output first, then propagates deriva- tives backward. Efficient when input dimension (parameters) output dimension (scalar loss).

Deep learning almost exclusively uses **Reverse Mode** because we typically have millions of parameters but a single scalar loss value.

Numerical Stability

Calculus in deep learning is performed using floating-point arithmetic, which is subject to pre- cision errors.

Techniques like Gradient Clipping, Batch Normalization, and residual connections (ResNets) are designed to mitigate these calculus-based issues.

Automatic Differentiation: The Engine of Deep Learning

Computational Foundations

While manual differentiation works for simple functions, deep learning models involve com- plex composite functions with millions of parameters. Computing gradients by hand becomes impractical and error-prone. Automatic differentiation (autograd) solves this problem by algo- rithmically computing exact derivatives through computational graphs.

The core idea is straightforward: any computation can be represented as a directed graph where nodes represent operations and edges represent data flow. For example, the function f (x) = 2x2 can be decomposed into:

  1. u = x2
  2. v = 2u

Automatic differentiation applies the chain rule systematically through this graph. Two pri- mary modes exist:

    • Forward mode: Computes derivatives alongside the forward pass. Efficient when input dimension < output dimension.
    • Reverse mode (backpropagation): Computes derivatives in reverse order. Efficient for functions with many inputs and single output (common in deep learning).

Forward vs. Reverse Mode Automatic Differentiation.

Implementation in Practice

Modern deep learning frameworks implement reverse-mode automatic differentiation. When we define a computation, the framework builds a dynamic computational graph tracking oper- ations. During the backward pass, it computes gradients by traversing this graph in reverse.

Key terminology:

  • Leaf nodes: Input variables (typically model parameters).
  • Root node: Final output (typically loss function).
  • Gradient flow: Direction of derivative propagation from root to leaves.

Theoretical Significance

Automatic differentiation provides the mathematical backbone for training deep neural net- works through gradient-based optimization. By efficiently computing exact gradients of com- plex composite functions, it enables:

  • Training models with millions of parameters
  • Implementing sophisticated optimization algorithms
  • Building custom architectures without manual derivative calculations

This automation has been instrumental in the rapid advancement of deep learning, allowing researchers and practitioners to focus on model design rather than mathematical implemen- tation details. The combination of automatic differentiation with GPU acceleration creates the foundation for modern deep learning systems.

Best Practices and Pitfalls

In-Place Operations

In-place operations (like x += y) can break the computational graph because they modify data needed for gradient computation. PyTorch will raise an error if an in-place operation invalidates required history.

Gradient Accumulation

By default, .backward() accumulates gradients in the .grad attribute. This is useful for RNNs but requires manually zeroing gradients (optimizer.zero_grad()) before each training step to avoid summing gradients from previous batches.

A More Formal Treatment

Sample Spaces and Events

The coin toss example introduces us to several key concepts in probability theory, but we need a more formal framework to handle complex scenarios.

Sample Space (Ω): The set of all possible outcomes of an experiment.

      • Coin toss: Ω = {H, T }
      • Die roll: Ω = {1, 2, 3, 4, 5, 6}
      • Two coin tosses: Ω = {(H, H), (H, T ), (T, H), (T, T )}

Event: A subset of the sample space representing a specific outcome or collection of out- comes.

      • First coin heads in two tosses: A = {(H, H), (H, T )}
      • Die equals 1: B = {1}
      • Die shows odd number: C = {1, 3, 5}

Probability Axioms

The mathematical foundation of probability rests on three fundamental axioms established by Andrey Kolmogorov:

  1. Non-negativity: For any event A, P (A) ≥ 0.
  2. Normalization: The probability of the entire sample space is 1: P (Ω) = 1.
  3. Countable Additivity: For mutually exclusive events Ai (where Ai ∩ Aj = ∅ for i /= j):

P (∪iAi) = P (Ai)

    • Complement Rule: P (A) + P (Ac) = 1
    • Empty Set: P (∅) = 0

The HIV Test Example

Let’s apply Bayes’ Theorem to a classic diagnostic problem.

P (D = 1) = P (D = 1|H = 1)P (H = 1) + P (D = 1|H = 0)P (H = 0)

P (D = 1) = 0.99 · 0.001 + 0.01 · 0.999 = 0.01098

Now, calculate posterior:
