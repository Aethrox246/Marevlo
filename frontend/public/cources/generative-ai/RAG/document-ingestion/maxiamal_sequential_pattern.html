Module 0 — The Big Picture: Why the MFP Algorithm Exists

This is all about a clever, fast algorithm that automatically spots Maximal Frequent Phrases (MFPs) basically the longest, most commonly repeating contiguous blocks of text across thousands or millions of documents.

By the end of this module, you'll see exactly why this problem matters in real Document AI pipelines, web crawling, search engines, and data cleaning, and why older methods fall short.

Goal: Grasp the problem and motivation (from Abstract, Introduction, and Related Work).

Boilerplate noise in documents: Headers, footers, disclaimers inflating indices and degrading search

Imagine you're building a search engine or feeding documents into an AI model (like training a large language model on web data). Most pages on the same website look very different at first glance, but they share huge chunks of identical text:

- Navigation menus ("Home | About | Contact | Privacy Policy")

- Legal footers ("© 2025 Company Name. All rights reserved. Terms of Service.")

- Cookie banners

- Sidebars with "Related Articles" or "Sign up for newsletter"

- Site-wide disclaimers ("This is not financial advice…")

We call this boilerplate or invariant fragments.

Why is it a problem?

- Index bloat: Every time you index these repeating blocks, your inverted index grows massively, wasting disk space and slowing queries.

- Search quality crash: If someone searches "privacy policy", suddenly millions of pages from the same site rank high because the footer matches, drowning out pages that actually talk about privacy.

- AI training noise: When you feed raw web crawls to models, they learn that Accept Cookies is super important, wasting compute on garbage patterns.

Real-world pain: This article mentioned modern web architecture relies on template-based content generation - unique content is wrapped in standardized blocks. Without removing them, downstream tasks (retrieval, summarization, entity extraction) suffer badly.

Limitations of heuristics / DOM analysis on dynamic / modern sites

Old-school ways to clean boilerplate:

- Rule-based heuristics ("remove anything in <footer> tag")

- DOM-tree analysis ("if this block appears on every page of the site, drop it")

These work okay for simple static sites… but fail hard on:

- JavaScript-heavy dynamic pages (React, Vue) where menus load via API and don't live in consistent DOM spots.

- Mobile vs desktop versions, different templates.

- Sites that randomize class names or shuffle elements for A/B testing.

- Single-page apps where everything is in one giant div.

The article explains it:

> "localized approaches are frequently insufficient for modern dynamic sites where template blocks may mimic the linguistic structure of the primary content or appear inconsistently across different subdirectories."

You need a corpus-wide, statistical view - not per-page rules. That's where recurrence across documents becomes the signal.

SPM for text: Finding invariant fragments via recurrence across documents

This is where Sequential Pattern Mining (SPM) comes in, but specialized for text. Instead of looking inside one document, we look across the entire corpus and ask: "Which exact contiguous phrases appear in many different documents?"

If a long string (e.g., a full privacy policy paragraph) shows up in 10,000+ docs from the same domain (or even across domains), it's almost certainly boilerplate.

The paper builds on SPM classics (PrefixSpan, SPAM, VMSP, BIDE+) but adds:

- Contiguous-only constraint (no gaps — templates don't have holes)

- Focus on maximal patterns (longest version only — no redundant sub-phrases)

- Seed-and-expand greedy strategy (much faster than recursive projection)

This gives you clean, non-overlapping invariant blocks to remove or flag.

Why maximal / contiguous patterns: Avoid redundancy, focus on longest frequent sequences

If you find "This site uses cookies" frequent, you'll also find every substring ("uses cookies", "cookies", etc.) frequent, millions of useless overlaps.

Maximal Frequent Phrases (MFPs) solve this:

- A phrase is maximal if no longer super-phrase containing it is also frequent (support ≥ threshold σ).

- Contiguous = no gaps (important for rigid templates like disclaimers).

The paper's scoring: Ψ(P) = length × document frequency — longer + more frequent = better. This compresses the output dramatically — instead of millions of n-grams, you get hundreds of clean, long boilerplate blocks.

Real-world examples

- Yandex-style web cleaning: (paper cites Zelenkov & Segalovich): Search engines identify site-wide navigation / footers via global recurrence → remove them before indexing → better precision.

- Gutenberg book deduplication: Project Gutenberg has tons of books with repeating prefaces, license texts, headers ("This eBook is for…"). MFPs catch the full license block across editions → strip once, save storage.

-Enron email signature removal: Corporate emails have repeating signatures ("Best regards, John Doe | VP Sales | Phone: …"). Algorithm finds the longest signature variants → clean dataset for NLP tasks.

-Amazon product template stripping: Every product page has "About this item", bullet lists, "Customer reviews" boilerplate. MFPs across millions of listings → extract clean product descriptions for recommendation models.

In an Indore context: Imagine cleaning e-commerce listings from local sites, every seller copies the same "Cash on Delivery | Free Shipping | 7-day Return" block. MFPs spot it instantly → better product search for customers.

Outcome: You see why MFPs solve real scalability issues in Document AI.

Boilerplate isn't just annoying, it's a silent killer of scale in search, crawling, and AI training. Traditional heuristics break on modern web; per-document rules miss cross-site patterns.

By mining maximal contiguous frequent phrases across a corpus, we get a clean, statistical way to detect and remove invariant junk, faster and more reliably than older SPM methods. This sets the stage for the algorithm itself. In the next modules we'll build it step by step, from seeds to greedy expansion to bitmask magic.

Module 1 — Foundations of Sequential Pattern Mining

In Module 0 we saw why boilerplate is a silent killer in large document collections, now it's time to understand the toolbox used to hunt it down: Sequential Pattern Mining (SPM). By the end, you'll have the SPM basics in your pocket, see why text needs special treatment, and even run a tiny Python toy example to feel the pain of redundancy.

Goal: Build core SPM knowledge (combines basic and advanced concepts from Related Work and Problem Formalization).

FIM vs SPM: Sequences, support (df, document frequency), contiguity (no gaps for text)

First, let's compare apples to oranges:

- Frequent Itemset Mining (FIM): - classic market-basket analysis: "People who buy bread & butter often buy milk. Order doesn't matter, just co-occurrence in a transaction.

- Sequential Pattern Mining (SPM): - order matters a lot. It's about finding repeating sequences (ordered lists) across many sequences (documents).

Example: In shopping — <bread → butter → milk> is a sequence. In text — <"This site uses" → "cookies" → "to improve"> is a sequence.

Key terms that are used in this article:

- Support (df, document frequency): Not how many times a phrase appears inside one doc, but how many unique documents contain it at least once.

df(P) = |{d ∈ corpus : P appears in d}|

Threshold σ = minimum number of docs needed to call it "frequent" (e.g., σ = 1000).

- Contiguity (no gaps): For boilerplate like disclaimers or headers, we want exact contiguous substrings, no interruptions. The paper stresses: "contiguous substrings" because templates are rigid. If you allow gaps (subsequences), you get nonsense like "Privacy Policy ... Accept Cookies" missing the middle junk.

Real talk: In natural language, allowing gaps would match unrelated phrases — useless for template detection.

Classic algorithms: GSP/Apriori, PrefixSpan (pattern-growth), SPAM (bit-vectors)

The article gives a nice history, here's the quick tour:

1. GSP (Generalized Sequential Patterns): - Apriori-style for sequences.

Generate candidates → scan database → prune infrequent → repeat.

Problem: Multiple full scans = slow on millions of docs.

2.PrefixSpan (pattern-growth):- ig leap forward (2001).

No candidate generation. Instead: Find frequent prefixes → project database into smaller "projected" sub-databases → recurse.

Much faster — but projection still costs memory/time when corpus is huge.

3. SPAM (Sequential PAttern Mining): - Vertical format hero.

Represents sequences as bit-vectors (one bit per document).

Frequent patterns found via bitwise AND intersections, super-fast on sparse data.

Later: FAST, CM-SPAM add pruning.

These are powerful, but for text corpora with millions of docs and long phrases, they still choke on redundancy.

Redundancy issues: Sub-fragments explosion

Here's the killer problem in text: If a long phrase is frequent (support ≥ σ), then every substring of it is also frequent!

Example:

- Full disclaimer: "This website uses cookies to enhance user experience and for analytics purposes." (appears in 50,000 docs)

- Then "uses cookies", "cookies to", "enhance user", ..., single words — all frequent too.

Output explodes: One long template → millions of overlapping n-grams.

Useless for boilerplate removal — you want the longest complete block, not its pieces.

Closed patterns (CloSpan, BIDE+) vs Maximal patterns (MaxSP, VMSP)

To fight redundancy:

- Closed patterns — A pattern is closed if no super-pattern has the exact same support.

CloSpan, BIDE+ (bi-directional extension check, clever pruning without storing all patterns).

- Maximal patterns — Even stricter: No super-pattern is frequent at all (even with lower support).

MaxSP (early), VMSP (current SOTA), focus on longest only.

VMSP uses vertical bit-vectors → very efficient.

The article chooses maximal because for boilerplate, the longest version is semantically richest (full header/footer).

Definitions: Vocabulary Σ, Document d, Phrase P, Seed S, df(P), MFP

Straight from the paper's Problem Formalization:

- Vocabulary Σ — All unique tokens (words), mapped to integers (for speed).

- Document d — Sequence of token IDs: d = (t₁, t₂, ..., tₙ)

- Phrase P — Contiguous subsequence: P = (tᵢ, tᵢ₊₁, ..., tᵢ₊ₖ₋₁)

- Seed S — Short frequent n-gram (e.g., 3–5 grams) used as starting point for expansion.

- df(P)— Number of unique docs containing P at least once.

- MFP (Maximal Frequent Phrase)— P is maximal if df(P) ≥ σ and no longer contiguous super-phrase P' has df(P') ≥ σ.

Math: df(P), maximality condition, scoring Ψ(P) = |P| × df(P)

From paper:

df(P) = |{d ∈ {1, ..., m} : ∃ position p s.t. (d, p) ∈ Occurrences(P)}|

Maximality: No P' ⊃ P (contiguous) with df(P') ≥ σ.

Scoring (Dominant Path): Ψ(P) = |P| × df(P)

→ Longer + more frequent = better candidate for the "dominant" (longest useful) template.

Real-world analogy: Mining shopping sequences (ordered items) but for words in docs

Think of supermarket baskets as documents, items as words.

Classic FIM: "Bread & butter often together" (no order).

SPM: "<Bread → Butter → Milk>" — people buy in that order.

For boilerplate: Treat each web page as a "shopping sequence" of words.

We want repeating "purchase paths" like "<Home → About → Contact → Privacy Policy>" that appear in thousands of "baskets" (pages).

But we only care about the **longest complete path** — not every sub-path like "<About → Contact>".

In Indore terms: Imagine scanning thousands of local e-commerce listings.

Many sellers copy-paste the same block: "Cash on Delivery | Free Shipping in Indore | 7 Days Return Policy | GST Invoice Provided".

SPM finds this long string repeating across shops → maximal version is the full block → strip it once → clean product descriptions.

Code: Python — Basic n-gram support calculator on toy corpus + simple maximality pruning

Let's feel it hands-on. Tiny corpus of 5 "documents" (strings).

python

from collections import defaultdict, Counter

import re

# Toy corpus: 5 short "documents"

corpus = [

"This site uses cookies to improve your experience. Privacy Policy.",

"Welcome! This site uses cookies to improve your experience. Contact us.",

"Our company. This site uses cookies to improve your experience. Terms.",

"Footer text. This site uses cookies to improve your experience. About.",

"Header. This site uses cookies to improve your experience. Privacy."

]

# Normalize: lower + split into words

docs = [re.findall(r'\w+', d.lower()) for d in corpus]

# Compute df for every possible contiguous n-gram (n=3 to 8)

min_n, max_n = 3, 8

ngram_df = defaultdict(int)

ngram_locations = defaultdict(list)

for doc_id, tokens in enumerate(docs):

for n in range(min_n, max_n + 1):

for i in range(len(tokens) - n + 1):

ngram = tuple(tokens[i:i+n])

ngram_df[ngram] += 1 # naive count — in real: unique docs

ngram_locations[ngram].append((doc_id, i))

# Filter frequent (support >= 3 docs)

sigma = 3

frequent = {ng: df for ng, df in ngram_df.items() if df >= sigma}

# Simple maximality check: keep only those not substring of longer frequent ones

maximal = []

for ng, df in sorted(frequent.items(), key=lambda x: -len(x[0])): # longest first

is_max = True

for longer in maximal:

longer_str = ' '.join(longer[0])

ng_str = ' '.join(ng)

if ng_str in longer_str:

is_max = False

break

if is_max:

maximal.append((ng, df))

print("Frequent n-grams (support >= 3):")

for ng, df in sorted(frequent.items(), key=lambda x: -df):

print(f"{' '.join(ng):<60} df={df}")

print("\nMaximal Frequent Phrases:")

for ng, df in maximal:

print(f"{' '.join(ng):<60} df={df} length={len(ng)}")

🧩 Module 2 — Probabilistic Filtering and Seed Generation

In Module 1 where we unpacked SPM basics and felt the redundancy pain firsthand, now let's get into the smart prep work that makes the whole MFP algorithm fly. This module is all about Phase I from the paper (Probabilistic Seed Discovery in Section 2, plus details from Sections 3 & 4). It's the "filter before you dig" stage: turning a massive messy corpus into a clean list of promising starting points (seeds) for expansion. By the end, you'll know how to tokenize, use clever probabilistic tricks to prune junk, and generate ranked seeds, ready for greedy growth in later modules.

Goal: Prepare corpus for mining (from Phase I: Probabilistic Seed Discovery and Sections 3 & 4).

Tokenization: Integer encoding, vocabulary building, prune low-df tokens

First step: Turn raw text into something computable.

- Vocabulary building: Scan all docs → collect unique words (tokens) → map each to a unique integer ID (e.g., "cookies" → 42).

Paper uses OpenMP for parallel scanning, fast on big corpora.

- Integer encoding: Replace words in each doc with their IDs.

Doc becomes a list of ints: ["this", "site", "uses", "cookies"] → [10, 20, 30, 40]

- Prune low-df tokens: While building vocab, count df (document frequency) for each token.

If a word appears in < σ docs (e.g., rare proper names), drop it, it can't be part of frequent phrases.

Paper: This "immediate pruning" skips whole sequences with rare words.

Why? Text corpora are noisy, typos, URLs, one-off names. Pruning early saves tons of compute.

Counting Bloom Filter: FNV-1a hash, 8-bit counters, atomic increments for parallelism

Here's the magic for estimating n-gram frequencies without exploding memory.

-What is it? A Bloom Filter is a probabilistic set, "probably in" or "definitely not in".

Counting version adds counters (here 8-bit = 0–255) to estimate frequency.

- How it works: For each n-gram (e.g., 4-gram tuple of IDs):

Hash it multiple times (paper uses FNV-1a, fast, low collisions).

Each hash points to a slot in a big array (e.g., 2GB → ~2^31 slots).

Increment the counters at those slots atomically (thread-safe for parallel processing).

- Probabilistic estimate: To check freq of an n-gram → hash it → take the minimum of those counters (avoids overcount from collisions).

Paper: 8-bit counters cap at 255 (enough for df estimates), FNV-1a for hashing, atomic increments via OpenMP.

Why probabilistic: Estimate n-gram frequency, prune 90%+ candidates (e.g., 2GB memory cap)

Exact n-gram counting on millions of docs? You'd need insane RAM for hash tables.

- Probabilistic wins: Bloom gives "good enough" estimates with fixed memory (paper caps at 2GB).

False positives ok, we over-estimate some, but prune 90%+ rare n-grams early.

- Pruning power: Only promote n-grams where min-counter ≥ σ to full verification.

Paper: This filters out junk before expensive exact df computation.

In numbers: For 1M docs, exact 4-gram counting might need 100GB+, Bloom does it in 2GB, pruning 90% candidates.

Seed gen: Fixed n (e.g., 4-5), in-memory/external sort-merge, rank by Ψ(S)

Now, generate seeds, short frequent n-grams as anchors.

- Fixed n: Pick n=4 or 5 (long enough to be specific, short enough to enumerate).

Slide window over all docs → extract all n-grams → estimate frequency via Bloom.

- Sort-merge: If too many candidates:

In-memory: Use heaps/priority queues.

External: Write to disk shards → sort-merge (paper's choice for massive scale).

- Rank by Ψ(S): Ψ = length × df (but since length fixed, mostly by df).

Keep top-K seeds (e.g., millions) for expansion.

Adapt to hardware: RAM-rich? In-memory. Disk-bound? External merge. Paper: Balances I/O vs memory.

Real-world example: Filtering rare phrases in massive logs before full processing

Picture you're at an Indore startup analyzing server logs from 1M e-commerce transactions daily. Logs full of repeating errors: "Connection timed out due to high load. Retry after 5 seconds." But mixed with rare one-offs: "User raman123 failed login from IP 192.168.1.1".

Before deep analysis (e.g., anomaly detection):

- Tokenize logs → build vocab (prune rare IPs/usernames).

- Bloom filter: Estimate 4-gram frequencies like ("timed", "out", "due", "to"), high df → seed.

- Prune 90% junk → generate ranked seeds: Full timeout message as candidate.

- Result: Focus compute on common issues → faster alerting, less storage waste.

Like cleaning chai spills before brewing, filter mess early!

Code: CountingBloomFilter class (add/estimate with FNV hash) + Tokenize corpus, generate/sort seeds (use pandas/numpy for efficiency)

Let's implement a simple version. (Note: Real paper uses C++ for speed — here Python for clarity. Use numpy for arrays.)

python

import numpy as np

import hashlib # For FNV-1a simulation (Python doesn't have built-in FNV)

from collections import defaultdict

class CountingBloomFilter:

def __init__(self, size=2**31 // 8, num_hashes=4): # ~2GB cap (size in bytes / 8-bit counters)

self.size = size

self.num_hashes = num_hashes

self.array = np.zeros(size, dtype=np.uint8) # 8-bit counters

def _hash(self, item, seed):

# Simulate FNV-1a (simple hash)

h = hashlib.sha256(str(item).encode() + seed.to_bytes(4, 'big')).digest()

return int.from_bytes(h[:8], 'big') % self.size

def add(self, item):

for i in range(self.num_hashes):

idx = self._hash(item, i)

if self.array[idx] < 255: # Cap at 255

self.array[idx] += 1 # Atomic in real: use locks or OpenMP

def estimate(self, item):

min_count = 255

for i in range(self.num_hashes):

idx = self._hash(item, i)

min_count = min(min_count, self.array[idx])

return min_count

# Toy corpus

corpus = [

"this site uses cookies to improve experience privacy policy",

"welcome this site uses cookies to improve experience contact us",

"our company this site uses cookies to improve experience terms",

"footer text this site uses cookies to improve experience about",

"header this site uses cookies to improve experience privacy"

]

# Tokenization: Build vocab, encode, prune low-df (sigma=3)

tokens = [doc.split() for doc in corpus]

flat = [w for doc in tokens for w in doc]

vocab = {w: i for i, w in enumerate(sorted(set(flat)))}

df = Counter(w for doc in tokens for w in set(doc)) # unique per doc

pruned_vocab = {w: i for i, (w, c) in enumerate(df.items()) if c >= 3}

encoded_docs = [[pruned_vocab[w] for w in doc if w in pruned_vocab] for doc in tokens]

# Seed gen: n=4, use Bloom to estimate

n = 4

bloom = CountingBloomFilter(size=10000, num_hashes=4) # Small for toy

candidates = defaultdict(int)

for doc in encoded_docs:

for i in range(len(doc) - n + 1):

ngram = tuple(doc[i:i+n])

bloom.add(ngram)

candidates[ngram] += 1 # Exact for toy; real: estimate first

sigma = 3

seeds = [(ng, bloom.estimate(ng)) for ng, count in candidates.items() if count >= sigma]

# Rank by Ψ = len * df (len fixed → by df)

ranked_seeds = sorted(seeds, key=lambda x: -x[1]) # Descending df

print("Ranked Seeds (ngram IDs, est_df):")

for ng, est in ranked_seeds[:5]: # Top 5

print(ng, est)

Explanation: Bloom adds/estimates n-gram freqs probabilistically. Tokenize: Build pruned vocab → encode docs. Generate candidates → filter by sigma → rank. In real scale: Use pandas for df calc, numpy arrays for speed, external sort for huge candidates.

Outcome: Create filtered seeds ready for expansion.

You've now prepped a corpus: Tokenized, pruned rare, estimated n-gram frequencies with Bloom (90%+ junk gone in 2GB), generated ranked seeds. This probabilistic filter is why the algorithm scales, exact methods would OOM on web-scale text.

Module 3 — Greedy Expansion and Global Occupancy Bitmask

In Module 2 where we prepped our corpus with tokenized docs, pruned junk, and generated ranked seeds via that clever Bloom filter, now we're at the heart of the algorithm. This module dives into Phase II from the paper (Priority-Based Expansion in Section 2, with details from Section 4 on the bitmask). It's where the magic happens: taking short seeds and growing them into full Maximal Frequent Phrases (MFPs) without the redundancy explosion we saw in classic SPM.

By the end, you'll grasp the greedy core and even code up a simple expander with bitmask — ready to hunt boilerplate in real corpora.

Goal: Core algorithm for MFPs (from Phase II: Priority-Based Expansion, Sections 4 & global bitmask details).

Greedy strategy: For P, select t* = arg max df(P ∪ {t}) if ≥ σ, commit to dominant path (linear, no branching)

The article's innovation: Greedy expansion instead of exhaustive branching.

Start with a seed P (e.g., 4-gram tuple from Module 2).

-Forward/Backward extend: Look for successors (next token t after P) or predecessors (prev token before P).

For each possible t: Compute df(P ∪ {t}) — how many docs have the extended phrase?

- Select t: Pick the t that maximizes df (if df ≥ σ).

Commit: Add it to P → now P is longer. Repeat until no more extensions ≥ σ.

- Dominant path: It's linear — no branching into trees.

If ties, pick one (e.g., highest df). This "commits to the dominant" longest path.

Math from paper: t* = arg max_t df(P ∪ {t}) s.t. df ≥ σ.

Why? Greedy approximates the optimal longest frequent super-phrase without exploring every combo.

Why greedy: Avoid recursive projections (faster than PrefixSpan/BIDE)

Remember PrefixSpan/BIDE from Module 1? They recurse into projected sub-databases for every branch, memory hog, slow on millions of docs.

- Greedy skips that: No projections, just query successors via inverted index (pre-built map: phrase → list of (doc, pos) where it appears).

Extend linearly along one path per seed.

Paper: At least 10x faster than baselines, no recursive overhead, focuses on "dominant" (highest coverage) paths.

Bitmask M: Track (doc, pos) coverage, check/mark to ensure maximality/non-redundancy

To avoid redundancy (e.g., expanding overlapping sub-phrases multiple times):

- Global bitmask M: A giant 2D bit array: rows = docs, columns = token positions in that doc.

Each bit: 1 if covered by some discovered MFP, 0 otherwise.

- Check before expand: For a candidate extension at (doc, pos), if any bit in that range is already 1, skip (already covered by a longer phrase).

- Mark after commit: Once you finalize an MFP, set all bits in its occurrences to 1.

This "suppresses" subsets, ensures maximality globally.

Paper: Bitmask uses sparse/compressed storage (e.g., bit-vectors), efficient even for huge corpora.

Path compression: Suppress overlapping sub-phrases

Tied to bitmask: During expansion, if a path overlaps a marked region, compress/skip. After finding a long MFP, its sub-parts are "compressed" (not expanded separately) because their positions are marked. This is path compression in action, like in union-find, but for phrases: Merge into longest.

Full pipeline: Seeds → expand → mask update

Tie it together:

1. Load ranked seeds from Module 2 (sorted by Ψ = len × df).

2. For each seed: Expand greedy (forward/backward) until stuck.

3. Check bitmask: If new MFP's positions uncovered → accept, mark bits.

4. Output MFPs.

Paper: Processes seeds in priority order (highest Ψ first), grabs longest first.

Real-world example: Growing "Copyright ©" into a full footer, skipping subsets

Imagine you're cleaning a corpus of 10,000 Indore business websites (e.g., local shops' terms pages).

Seed: ("copyright", "©", "2026"), high df from footers.

- Greedy expand backward: Add "All rights reserved." if df ≥ σ → commit.

- Forward: Add "Company Name." → full: "All rights reserved. Copyright © 2026 Company Name."

- Bitmask: Mark all positions where this full footer appears.

- Later seeds like ("©", "2026") hit marked bits → skipped (subset suppressed).

Result: You get the complete footer once — not 100 overlapping variants.

Saves time when indexing for a local search engine — no redundant "copyright" hits.

Code: Expansion loop (successor query simulation via inverted index) + Bitmask class (numpy/bitarray) + Integrate with seeds from Module 2

Build on Module 2's encoded_docs and pruned_vocab. Use bitarray for bitmask (pip install bitarray — but since no install, assume numpy bool arrays for toy).

python

import numpy as np

from collections import defaultdict

from bitarray import bitarray # For efficiency; fallback to np.bool

# Assume from Module 2: encoded_docs (list of int lists), pruned_vocab

# Build inverted index: ngram → list of (doc_id, start_pos)

inv_index = defaultdict(list)

for doc_id, tokens in enumerate(encoded_docs):

for i in range(len(tokens) - 1): # Up to bigrams for successors

bigram = tuple(tokens[i:i+2])

inv_index[bigram].append((doc_id, i))

# Bitmask: For toy, list of bitarrays per doc (1 bit per token pos)

class GlobalBitmask:

def __init__(self, doc_lengths):

self.masks = [bitarray(length) for length in doc_lengths]

for mask in self.masks:

mask.setall(False)

def is_covered(self, doc_id, start, length):

return any(self.masks[doc_id][start:start+length]) # Any bit set?

def mark(self, doc_id, start, length):

self.masks[doc_id][start:start+length] = True

bitmask = GlobalBitmask([len(d) for d in encoded_docs])

# Expansion loop: Greedy for one seed

def greedy_expand(seed, sigma=3):

P = list(seed) # Mutable list of token IDs

occurrences = inv_index.get(tuple(P[-2:]), []) # Start with last bigram

while True: # Forward expand

candidates = defaultdict(int)

for doc_id, pos in occurrences:

if pos + len(P) < len(encoded_docs[doc_id]): # Room for next

next_t = encoded_docs[doc_id][pos + len(P)]

candidates[next_t] += 1

if not candidates:

break

t_star = max(candidates, key=candidates.get) # arg max df

if candidates[t_star] >= sigma:

P.append(t_star)

# Update occurrences: Filter to those with t_star next

new_occ = [(d, p) for d, p in occurrences if encoded_docs[d][p + len(P) - 1] == t_star]

occurrences = new_occ

else:

break

# Similar for backward (prepend) — exercise for you!

# Check/mark bitmask

mfps = []

for doc_id, start in occurrences:

if not bitmask.is_covered(doc_id, start, len(P)):

mfps.append((P, occurrences)) # Collect if new

bitmask.mark(doc_id, start, len(P))

return mfps

# Integrate: Take top seed from Module 2's ranked_seeds

top_seed = ranked_seeds[0][0] # e.g., (10, 20, 30, 40) from prev

mfps = greedy_expand(top_seed)

print("Expanded MFPs:", mfps)

Module 4 — Hierarchical Tree Construction and Ladder Compression

By now you've got the raw MFPs from the seed-and-expand core (Modules 2–3). But raw lists are messy, especially when the same template has slight variations (e.g., different company names, dates, or legal clauses).

This module covers the post-processing step from Section 5 of the paper: turning flat MFPs into a hierarchical tree for interpretability, then applying ladder compression to collapse boring linear chains into compact, readable structures.

The goal? Make the output human-friendly and useful for analysts, not just a dump of long strings.

Goal: Post-process for interpretability (from Section 5).

Build tree: Identify parent-child (prefix/suffix matches)

Once you have a set of MFPs (long maximal phrases), group them hierarchically:

- Parent-child relation: Phrase A is parent of B if:

- A is a proper prefix or suffix of B, and

- df(B) ≈ df(A) (or very close, meaning B is a small variation of A), and

- No longer phrase sits between them.

- Tree construction:

Start with longest/root MFPs.

For each shorter phrase, check if it prefixes/suffixes a longer one → attach as child.

Use string matching (startswith/endswith on tokenized phrases) + df similarity threshold (e.g., |df(A) - df(B)| ≤ 5% or fixed delta).

Paper hints: This creates a DAG (Directed Acyclic Graph), not always strict tree, because one phrase can branch into multiple variants.

Result: A tree/DAG where root = most general long template, leaves = specific variants.

Ladder compression: Collapse linear chains (one child → merge with tag for new tokens)

Many chains are linear:

Root: "All rights reserved. Copyright © 2025"

Child: "All rights reserved. Copyright © 2025 Company XYZ"

Grandchild: "All rights reserved. Copyright © 2025 Company XYZ. No warranty…"

Ladder compression (paper's term): Collapse these chains into one node with tagged differences.

- If a node has exactly one child → merge them.

- Record the delta tokens (added/removed) as a tag/label on the edge or node.

Example: "Company XYZ" added at end → tag: "+ [Company Name]"

- Repeat recursively → turn long chains into compact ladders with variation points.

Why? Reduces visual clutter. A 10-level chain becomes one node with 3–4 variation tags.

Output: Interactive HTML for browsing branches/divergences

Paper suggests: Generate interactive HTML (or Graphviz DOT → SVG) for analysts to explore.

- Nodes: Phrase text (truncated + tooltip full).

- Edges: Labeled with deltas ("+ date", "- old clause").

- Colors: Green = high df, red = low variation.

- Click to expand variants or see occurrence stats.

Tools: Use networkx + pyvis (interactive HTML) or simple Jinja2 templates.

Why: Visualize variants (e.g., signature blocks with slight changes)

Raw MFPs give you the blocks — but variants tell the story:

- "Privacy Policy last updated [DATE]" → branches on dates.

- Email signatures: Core text same, but names/titles/phone change.

- Legal disclaimers: Base clause identical, but jurisdiction or company-specific add-ons branch off.

Tree + compression makes it easy to spot:

- Core invariant template (trunk)

- Common customizations (branches)

- Rare outliers (leaves)

Analysts can decide: "Strip the whole trunk + common branches" vs. "Keep branches with company names".

def build_phrase_dag(mfps, df_threshold=0.95):

G = nx.DiGraph()

phrase_to_node = {}

# Sort by length descending

sorted_mfps = sorted(mfps, key=lambda x: -len(x[0]))

for phrase, df, _ in sorted_mfps:

node_id = ' '.join(phrase[:10]) + '...' if len(phrase) > 10 else ' '.join(phrase)

G.add_node(node_id, label=' '.join(phrase), df=df, full=phrase)

# Find parents: longer phrases that prefix/suffix this one

for parent_id, data in G.nodes(data=True):

parent_phrase = data['full']

if len(parent_phrase) <= len(phrase):

continue

# Prefix match

if parent_phrase[:len(phrase)] == phrase:

similarity = df / data['df'] if data['df'] > 0 else 0

if similarity >= df_threshold:

G.add_edge(parent_id, node_id, label="prefix")

# Suffix match (similar check)

return G

def ladder_compress(G):

# Recursive collapse: nodes with indegree=1 & outdegree=1 → merge

changed = True

while changed:

changed = False

for node in list(G.nodes):

if G.in_degree(node) == 1 and G.out_degree(node) == 1:

pred = list(G.predecessors(node))[0]

succ = list(G.successors(node))[0]

delta = f"+ {G.nodes[node]['label']}" # Simplified tag

G.add_edge(pred, succ, label=delta)

G.remove_node(node)

changed = True

return G

Goal: Understand advantages (from Sections 6, 7, and Complexity Analysis).

Innovations: Bloom pruning, greedy commitment, bitmask vs bidirectional checks

The paper's key breakthroughs:

1. Bloom Filter pruning (Phase I)

Probabilistic counting with 8-bit counters + FNV-1a → estimate n-gram freqs in fixed memory (2GB cap).

Prunes 99.9999% of candidates before exact df computation — huge early filter.

2. Greedy commitment to dominant path

Instead of branching/exploring all possibilities (like PrefixSpan/BIDE), pick the single best extension (max df) and commit linearly.

No recursion → no projected databases → massive speed-up.

3.Global occupancy bitmask

Tracks every (doc, position) covered by discovered MFPs.

Ensures maximality globally (no redundant sub-phrases) without storing previous patterns in memory (unlike BIDE+'s bidirectional checks).

Bitmask is sparse/compressed → scales to 1M+ docs.

Vs baselines:

- BIDE+ / CloSpan: Clever pruning but still recursive + memory-heavy for long text patterns.

- VMSP / SPAM: Vertical bit-vectors great for sparse data, but redundancy explosion on text → output millions of overlapping n-grams.

- Our method: Sacrifices theoretical completeness (might miss rare branches) for practical speed + clean maximal output.

Vs baselines: 266% speedup, 100% precision (3 vs 11 patterns on synthetic data)

Theoretical breakdown (paper's analysis):

- Seed generation (Phase I):

Tokenization + Bloom adds: O(N) where N = total tokens across corpus.

Sorting candidates or external merge: O(N log N) worst-case.

→ Dominated by sort/merge → O(N log N).

- Expansion (Phase II):

For each seed: Successor queries via inverted index → O(1) amortized per extension.

Number of extensions bounded by longest phrase length (~100–200 tokens max in practice).

Bitmask checks/marks: O(1) per position with sparse bit-vectors.

Total across all seeds: Roughly O(N) — each token position checked a constant number of times.

Overall: Near-linear in corpus size — huge win vs recursive O(N²) worst-case in PrefixSpan on dense text.

Trade-offs: Sacrifices completeness for speed/maximality in text

Critical thinking:

- Not complete: Greedy might miss rare-but-maximal branches if a dominant path overshadows them.

Paper admits: Sacrifices theoretical exhaustiveness for practical maximality + speed.

- Why ok for text? Boilerplate templates are usually dominant (high df, long, rigid).

Rare variants (e.g., one-off typo in footer) aren't useful for stripping anyway.

- Bitmasks scale better for 1M+ docs?

Yes — bit-vectors are compact (1 bit per token position).

For 1M docs × avg 1000 tokens = 1 billion bits ≈ 125 MB compressed.

Vs storing full pattern lists (BIDE+ needs to remember previous patterns) → exponential memory.

Bitmask is global + sparse → check/mark in O(1) amortized, no explosion.

# Re-used classes from earlier modules (simplified)

# -------------------------------

class CountingBloomFilter:

def __init__(self, size=2**28, num_hashes=5): # ~32MB

self.size = size

self.num_hashes = num_hashes

self.array = np.zeros(size, dtype=np.uint8)

def _hash(self, item, seed):

h = hash(str(item) + str(seed)) & 0xFFFFFFFF # 32-bit

return h % self.size

def add(self, item):

for i in range(self.num_hashes):

idx = self._hash(item, i)

if self.array[idx] < 255:

self.array[idx] += 1

def estimate(self, item):

return min(self.array[self._hash(item, i)] for i in range(self.num_hashes))

# Global bitmask (per doc: bitarray of positions)

class GlobalBitmask:

def __init__(self, doc_lengths):

self.masks = [bitarray(length) for length in doc_lengths]

for m in self.masks:

m.setall(False)

def is_covered(self, doc_id, start, length):

return self.masks[doc_id][start:start+length].any()

def mark(self, doc_id, start, length):

self.masks[doc_id][start:start+length] = True

def tokenize_file(file_path):

"""Tokenize one file"""

text = Path(file_path).read_text(encoding='utf-8', errors='ignore').lower()

tokens = re.findall(r'\w+', text)

return tokens

def build_vocab_and_encode(corpus_files, min_df=10):

"""Parallel vocab + encoding"""

with mp.Pool() as pool:

tokenized = pool.map(tokenize_file, corpus_files)

flat_unique = Counter(w for doc in tokenized for w in set(doc))

vocab = {w: i for i, (w, c) in enumerate(flat_unique.items()) if c >= min_df}

encoded = [[vocab.get(w, -1) for w in doc if w in vocab] for doc in tokenized]

encoded = [doc for doc in encoded if doc] # remove empty

return encoded, vocab

print(f"Encoded {len(encoded_docs)} docs")

bloom = CountingBloomFilter()

candidates = defaultdict(int)

for doc_id, tokens in enumerate(encoded_docs):

for i in range(len(tokens) - n + 1):

ngram = tuple(tokens[i:i+n])

bloom.add(ngram)

candidates[ngram] += 1 # approx count

seeds = [(ng, bloom.estimate(ng)) for ng, c in candidates.items() if c >= sigma]

seeds.sort(key=lambda x: -x[1]) # rank by est df

seeds = seeds[:max_seeds]

print(f"Generated {len(seeds)} seeds")

Goal: Design next-gen variants.

Multimodal: Add DOM/layout info, embeddings (e.g., integrate with DiT/LayoutLM)

The original MFP is pure text - it only sees token sequences, no visual/layout cues.

Future extensions:

- Incorporate DOM tree / HTML structure:

When mining web pages, keep track of XPath or CSS selectors where phrases appear.

Example: If the same long phrase appears in <footer> across 10K pages → stronger boilerplate signal than if it’s scattered in <p> tags.

- Fuse with vision embeddings (DiT-style):

Run DiT (or LayoutLMv3) on rendered page images → get patch embeddings for each text block.

Cluster similar visual regions → only mine MFPs within same visual cluster (e.g., all footers together).

Or: Use cosine similarity between DiT embeddings to group near-identical visual templates even if OCR tokenizes slightly differently.

- LayoutLM integration:

Feed MFP-cleaned text + bounding boxes into LayoutLM pre-training → cleaner multimodal corpus. Or reverse: Use LayoutLM’s layout-aware attention to guide which phrases to expand (prefer phrases in same layout zone). Why? Modern web is multimodal, text alone misses that "this disclaimer is always at bottom in gray font" signal.

Scaling: Beyond 1M docs, distributed (Spark/Hadoop)

This article already handles 1.7M docs in minutes on one machine, but web-scale is billions.

Next steps:

- Distributed seed generation:

Spark → partition docs → build local Bloom filters → merge counters (approximate union).

Or Hadoop MapReduce for external sort-merge of candidates.

- Distributed expansion:

Assign top seeds to workers → each expands independently → central reducer checks bitmask conflicts (use distributed bitmap like RoaringBitmaps).

- Sharded bitmask:

Shard by doc_id → each worker owns bitmask for its shard → gossip or reduce conflicts only when needed.

Goal: Mine MFPs from Common Crawl snapshots (petabytes) → feed cleaner data to next-gen LLMs.

Variants: Gapped patterns, fuzzy matching for OCR errors

Pure contiguous is great for rigid templates, but noisy data needs flexibility.

- Allow small gaps (max_gap=1–2):

Catch templates broken by dynamic ads ("Privacy Policy [ad] Terms of Service").

- Fuzzy matching / edit distance:

During expansion, allow Levenshtein distance ≤1–2 → catch OCR errors ("C0pyright" vs "Copyright", "2026" vs "2O26").

- Embedding-based successors:

Instead of exact token match, use word embeddings (fastText/BERT) → extend if cosine sim > 0.9 (handles synonyms or typos).

Trade-off: More recall → more false positives → need stronger bitmask / clustering.

Applications: ML integration (e.g., fine-tune BERT on cleaned data), real-time web crawling

Practical downstream uses:

- Cleaner pre-training corpora: Strip MFPs before feeding to BERT/RoBERTa/GPT → less noise, better downstream NLP.

- Real-time crawler filtering: During crawl, maintain rolling Bloom + recent MFPs → skip boilerplate pages early → save bandwidth.

- Dataset curation for RAG: Clean knowledge base → better retrieval precision.

- Ad/template blocker: Browser extension flags site-wide navigation/disclaimers.

Code: Prototype extension — Add embedding-based successor (cosine sim instead of exact)

Let’s prototype one idea embedding-guided expansion (instead of exact token match, use fastText-like vectors).

Python

import numpy as np

from collections import defaultdict

# Dummy fastText-style embeddings (in real: load gensim fasttext)

embedding_dim = 50

np.random.seed(42)

vocab_embeddings = {i: np.random.randn(embedding_dim) for i in range(10000)} # mock

def cosine_sim(a, b):

return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b) + 1e-8)

def embedding_guided_expand(P, encoded_docs, inv_index, sigma=100, sim_threshold=0.85):

"""

Extend P using embedding similarity instead of exact match

"""

current_vec = np.mean([vocab_embeddings[t] for t in P], axis=0)

candidates = defaultdict(float) # next_token → avg sim

counts = defaultdict(int)

# Find positions where P ends

for doc_id, start in inv_index.get(tuple(P[-min(5,len(P)):]), []): # last few for context

pos = start + len(P) - 1

if pos + 1 < len(encoded_docs[doc_id]):

next_t = encoded_docs[doc_id][pos + 1]

next_vec = vocab_embeddings[next_t]

sim = cosine_sim(current_vec, next_vec)

if sim > sim_threshold:

candidates[next_t] += sim

counts[next_t] += 1

if not candidates:

return P # stuck

# Pick best: highest avg sim * count

best_t = max(candidates, key=lambda t: candidates[t] * counts[t])

if counts[best_t] >= sigma:

P.append(best_t)

return embedding_guided_expand(P, encoded_docs, inv_index, sigma, sim_threshold)

return P
