Module 0 — Why Contrastive Learning Exists

Today we’ll explore why the world moved from “give me 1 million labeled images” to contrastive learning, the technique that powers CLIP, SimCLR, Stable Diffusion, and almost every cool AI product in 2026.

Goal: Understand why models need contrastive learning.

The Representation Learning Problem

Imagine you want to build a smart photo search. You have millions of images, but no one has labeled them (“this is a dog”, “this is a cat”). How does the model learn that two photos of different dogs are “similar”, while a dog and a car are “different”?

This is the representation learning problem, teaching a model to create meaningful embeddings (numerical vectors) where similar things are close together and dissimilar things are far apart all by itself.

Why labels are expensive

Labeling data is brutally expensive and slow:

- One ImageNet image took ~1 minute of human work → 1.4 million images = years of work.

- For medical images, legal documents, or rare languages → labeling is almost impossible.

- Humans disagree, get tired, and introduce bias.

So researchers asked:

“Can we make the model learn useful representations without any labels at all?”

Self-supervised learning idea: - Self-supervised learning = “the data itself is the teacher”.

Instead of asking humans “Is this a cat?”, we ask the model:

- “Can you predict what part of the image is missing?”

- “Can you tell me if these two images are the same or different?”

Contrastive learning is one of the most successful forms of self-supervised learning.

Similarity learning intuition: - The core idea is simple:

Similar things should have similar representations. Dissimilar things should have very different representations. We don’t need class labels. We just need to know which images (or text, or audio) are positive pairs (two views of the same thing) and which are negative pairs (unrelated things). That’s exactly what InfoNCE does, it learns by pushing positive pairs closer and negative pairs farther apart.

Real-World Examples

1.Google Image Search

You search “golden retriever”. Google doesn’t need someone to label every dog photo.

It learned (using contrastive learning) that two photos of golden retrievers are similar, even if one is indoors and one is outdoors. That’s why your search works so well.

2. Spotify “Discover Weekly”

Spotify doesn’t ask you “Do you like this song?” for every song.

It learned that two songs you skip in a row are dissimilar, while two songs you play back-to-back are similar. That’s pure contrastive learning on audio embeddings.

- Labels are expensive and limited.

Module 1 — Foundations of Similarity Learning

In Module 0 we understood why models need to learn similarity instead of relying on expensive labels. Now let’s build the actual mathematical foundation — the tools machines use to decide “these two things are similar” or “these two are completely different”.

Think of this as learning the ruler and compass before drawing beautiful maps. Once you master these concepts, everything in contrastive learning (and InfoNCE) will suddenly make perfect sense.

Goal: Build mathematical intuition behind similarity.

Vector embeddings, the language of machines

Everything in modern AI is represented as vectors (lists of numbers).

- An image becomes a vector of 512 numbers.

- A sentence becomes a vector of 768 numbers.

- A song or product also becomes a vector.

These are called embeddings. The magic is that similar things end up with similar vectors.

How do machines measure similarity?

There are three common ways. Let’s understand each one:

1. Euclidean Distance (straight-line distance)

Measures “how far apart” two vectors are in space.Smaller distance = more similar. Problem: It cares about magnitude (length), two long vectors look far even if they point in the same direction.

2. Dot Product Similarity

Simple multiplication of corresponding numbers, then sum. Higher value = more similar. Still affected by vector length.

3. Cosine Similarity

Measures the angle between two vectors, ignoring their length. Value ranges from -1 (opposite) to +1 (identical direction). This is the most popular because direction matters more than magnitude in representation learning.

Embedding space: - where the magic lives

Imagine a huge 3D room (in reality it’s 512D or 1024D we can’t visualize it).

- Similar images (different photos of the same dog) cluster together.

- Dissimilar things (dog vs car) are pushed far apart.

The goal of contrastive learning is to shape this space so that “similar = close, different = far”.

Real-World Examples

1. Sentence similarity search

Google or Notion search: You type “how to cook biryani”. The system converts your sentence and all stored documents into embeddings, then finds the ones with highest cosine similarity. That’s why it returns relevant recipes even if they don’t contain the exact words.

2. Product recommendation embeddings

Amazon or Myntra: Two shirts that look similar (same color, style) have embeddings that are very close in vector space. When you like one shirt, the system finds others with high cosine similarity → “Customers also bought”.

You now have the core toolkit:

- Vector embeddings = the language

- Cosine similarity = the ruler

- Embedding space = the room where similar things live close together

This intuition is the foundation of InfoNCE loss, CLIP, and almost every modern self-supervised model. Next module we’ll finally meet InfoNCE itself, the actual loss function that forces models to learn this similarity automatically.

Module 2 — From Classification Loss to Contrastive Loss

In Module 1 we built the mathematical foundation, embeddings, cosine similarity, and the idea of an embedding space where similar things live close together.

Now we take the next big step: understanding how models actually learn this similarity.

We’ll start from the familiar world of classification (what you studied in school) and gently move into the contrastive world. This shift is the key moment that made CLIP, SimCLR, and InfoNCE possible.

Goal: Understand how contrastive learning differs from classification.

Cross Entropy, The Classic Way

Most deep learning you’ve seen uses cross-entropy loss:

- You have images + labels (“cat”, “dog”, “car”).

- The model predicts probabilities for each class.

- Loss punishes wrong predictions.

This works great when you have millions of labeled examples.

But what if you don’t? What if labeling is impossible or too expensive?

That’s when we leave classification behind.

Pairwise Learning: - The Contrastive Idea

Contrastive learning says:

“Instead of predicting labels, just teach the model to compare pairs.”

We create:

- Positive pairs: Two views of the same thing (e.g., two different photos of the same dog, or two augmentations of the same image).

- Negative pairs: Two different things (a dog and a car).

The model’s job is no longer “Is this a dog?”

Its new job is:

“Are these two things similar or different?”

This is called pairwise learning or contrastive learning.

Positive vs Negative Samples

- Positive sample: Something that should be pulled closer in embedding space.

- Negative sample: Something that should be pushed away.

The model learns by constantly answering:

- These two images are from the same original → make their embeddings similar.

- These two images are completely unrelated → make their embeddings different.

No class labels needed, only the ability to create positive and negative pairs from the data itself.

Triplet Loss Intuition (The Bridge to InfoNCE)

Before InfoNCE became popular, researchers used Triplet Loss (FaceNet, 2015).

Idea: Take three images at once:

- Anchor (reference image)

- Positive (another view of the same person/object)

- Negative (a different person/object)

Loss formula (simplified):

python

Loss = max( distance(anchor, positive) - distance(anchor, negative) + margin , 0 )

It literally says:

Make anchor-positive closer than anchor-negative by at least a margin. This was the first big step from classification to comparison.

Real-World Example: Face Recognition Systems

Think of your phone’s Face ID or Google Photos.

- When you add a new face, the system doesn’t ask “Is this person A, B, or C?”

- Instead it learns:

→ Two photos of you = positive pair (pull embeddings closer)

→ Your photo and your friend’s photo = negative pair (push embeddings apart)

This is why your phone can recognize you from different angles, lighting, and even with glasses — without you labeling every single photo.

Spotify, YouTube, and Amazon recommendations work the exact same way with songs, videos, and products.

Hands-On: Implement Simple Triplet Loss

Let’s code the intuition:

python

import torch

import torch.nn.functional as F

# Three embeddings (Anchor, Positive, Negative)

anchor = torch.tensor([1.0, 2.0, 3.0])

positive = torch.tensor([1.1, 2.1, 3.1]) # very similar

negative = torch.tensor([5.0, 6.0, 7.0]) # completely different

# Cosine distance (1 - similarity)

def cosine_distance(a, b):

return 1 - F.cosine_similarity(a.unsqueeze(0), b.unsqueeze(0))

dist_ap = cosine_distance(anchor, positive)

dist_an = cosine_distance(anchor, negative)

margin = 0.5

triplet_loss = torch.max(dist_ap - dist_an + margin, torch.tensor(0.0))

print(f"Anchor-Positive distance: {dist_ap.item():.4f}")

print(f"Anchor-Negative distance: {dist_an.item():.4f}")

print(f"Triplet Loss: {triplet_loss.item():.4f}")

Run this, you’ll see the loss tries to make the positive pair closer than the negative pair.

Outcome: Understand why contrastive learning compares samples instead of predicting labels.

You now see the fundamental shift:

- Cross-entropy → “Predict the label”

- Contrastive (Triplet) → “Compare samples and learn similarity”

No labels required. Just the ability to create positive and negative pairs from the data itself.

This is the bridge to InfoNCE the loss function that made contrastive learning scale to billions of examples and power today’s foundation models.

Module 3 — Noise Contrastive Estimation (NCE)

In Module 2 we saw how contrastive learning shifted from “predict the label” to “compare samples.”

Now we meet the clever mathematical trick that made this shift practical at scale: Noise Contrastive Estimation (NCE).

NCE is the bridge between old-school density estimation and the modern contrastive losses (like InfoNCE) that power CLIP, SimCLR, and almost every foundation model today.

Goal: Learn the idea behind Noise Contrastive Estimation.

The Density Estimation Problem

The original dream of unsupervised learning was:

> “Can the model learn the true probability distribution of real data?”

This is called density estimation, learning p(data), the probability that a sample comes from the real world.

But estimating p(data) directly is extremely hard:

- The space is huge (millions of dimensions).

- The model can easily cheat by assigning high probability to everything.

- Computing the normalizing constant (partition function) is often impossible.

Researchers needed a smarter way.

Noise vs Data Distribution

NCE’s breakthrough idea (Gutmann & Hyvärinen, 2010): Instead of trying to estimate the real data distribution p_data directly,

turn the problem into distinguishing real data from fake noise.

- Real samples: drawn from the true data distribution p_data

- Noise samples: drawn from a simple, known noise distribution p_noise (usually uniform or Gaussian)

Now the model’s job becomes a binary classification problem:

> “Is this sample real (from data) or fake (from noise)?”

This is much easier to optimize!

Binary Classification Formulation

NCE reframes everything as a simple logistic regression:

For a sample x:

- Probability it is real = σ( f_θ(x) )

(where f_θ is the model’s scoring function, σ is sigmoid)

The loss becomes regular binary cross-entropy:

- Real sample → target = 1

- Noise sample → target = 0

By training the model to separate real from noise, the scoring function f_θ automatically learns a good representation of the data distribution, without ever computing the impossible normalizing constant.

Real-World Example: Language Models

Think of early Word2Vec-style models:

- Real word: “The cat sat on the ___” → next word should be “mat”

- Noise word: randomly sampled word like “zebra” or “quantum”

The model learns to predict:

> “Is ‘mat’ a likely continuation or just random noise?”

This binary decision forces the model to learn meaningful word embeddings.

Exactly the same principle powers modern contrastive language models.

Hands-On: Binary Classifier Distinguishing Real Word vs Noise Word

Here’s a tiny working example:

python

import torch

import torch.nn as nn

import torch.nn.functional as F

# Simple scoring network

class SimpleNCE(nn.Module):

def __init__(self, vocab_size=10000, embed_dim=64):

super().__init__()

self.embed = nn.Embedding(vocab_size, embed_dim)

self.score = nn.Linear(embed_dim, 1)

def forward(self, x):

emb = self.embed(x)

return self.score(emb).squeeze(-1)

model = SimpleNCE()

# Simulate one batch

real_words = torch.tensor([42, 7, 153]) # actual words from corpus

noise_words = torch.tensor([9999, 8888, 7777]) # random noise words

real_scores = model(real_words)

noise_scores = model(noise_words)

# NCE loss: real = 1, noise = 0

real_loss = F.binary_cross_entropy_with_logits(real_scores, torch.ones_like(real_scores))

noise_loss = F.binary_cross_entropy_with_logits(noise_scores, torch.zeros_like(noise_scores))

nce_loss = real_loss + noise_loss

print(f"NCE Loss: {nce_loss.item():.4f}")

This tiny network is learning to give high scores to real words and low scores to noise, exactly how NCE works at scale.

Outcome: Understand how NCE converts density estimation into classification.

You now see the beautiful trick:

- Direct density estimation → almost impossible

- NCE → turn it into binary classification (real vs noise)

- Result → the model learns powerful representations without labels

NCE was the key insight that made contrastive learning scalable.

In the next module we’ll meet InfoNCE, the improved version that removed the limitations of basic NCE and became the foundation of modern contrastive learning (CLIP, SimCLR, etc.).

Module 4 — InfoNCE Loss (Core Concept)

In Module 3 we saw how NCE turned the hard problem of density estimation into a simple “real vs noise” classification game.

Today we meet the improved version that actually powers modern AI: InfoNCE Loss.

This is the exact loss function used in CLIP, SimCLR, MoCo, and almost every contrastive learning breakthrough since 2018. Once you understand InfoNCE, everything else (including the famous CLIP paper) suddenly clicks.

Goal: Learn the InfoNCE loss mathematically and intuitively.

The Setup: Anchor, Positive, and Negative Samples

InfoNCE works with triplets (but smarter than old triplet loss):

- Anchor (q): The main sample we’re focusing on right now (e.g., one augmented view of an image).

- Positive sample (k⁺): Another view of the same thing (e.g., a cropped or color-jittered version of the same image). We want their embeddings to be very similar.

- Negative samples (k⁻): Many other unrelated samples from the batch (or a memory bank). We want their embeddings to be different from the anchor.

The model doesn’t need class labels — it only needs to know which pair is “the same” and which ones are “different”.

The InfoNCE Formula (The Heart)

Where:

- sim(·,·) = cosine similarity (or dot product)

- τ (tau) = temperature parameter (controls how sharp the distribution is)

- The sum in the denominator runs over the positive and all the negative samples

Intuitively:

The loss wants the similarity between anchor and positive to be much larger than the similarity to any negative sample. The softmax in the denominator turns this into a clean probability distribution.

Softmax Normalization & Mutual Information Maximization

The formula is basically a softmax over similarities.

- The numerator tries to make the positive pair score high.

- The denominator (with all negatives) tries to make all negative pairs score low.

When you minimize this loss, you’re actually maximizing the mutual information between the two views of the same data, without ever needing labels.

That’s the deep beauty of InfoNCE: it learns rich, useful representations by simply asking “which of these many candidates is the real match?”

Real-World Example: Image Augmentations

Think of how SimCLR (a famous InfoNCE model) works:

- Take one photo of a dog.

- Create two different augmentations: one cropped + flipped, another color-jittered.

- These two views = positive pair (anchor and k⁺).

- All other images in the batch = negative samples (k⁻).

The model learns:

“Even though these two images look different (different crops, colors), they are the same dog → make their embeddings close.”

That’s why CLIP can understand that a photo of a dog and the text “golden retriever” are similar the same contrastive principle applied across images and text.

Hands-On: Implement InfoNCE Loss from Scratch

Let’s code the actual loss so you can feel it:

python

import torch

import torch.nn.functional as F

def info_nce_loss(anchor, positive, negatives, temperature=0.07):

"""

anchor: [batch_size, embed_dim]

positive: [batch_size, embed_dim] (one positive per anchor)

negatives: [batch_size, num_negatives, embed_dim]

"""

# Normalize embeddings

anchor = F.normalize(anchor, dim=1)

positive = F.normalize(positive, dim=1)

negatives = F.normalize(negatives, dim=2)

# Similarity scores

pos_sim = torch.sum(anchor * positive, dim=1) / temperature # shape: [batch]

neg_sim = torch.einsum('bd,bnd->bn', anchor, negatives) / temperature # shape: [batch, num_neg]

# Concatenate positive and all negatives

logits = torch.cat([pos_sim.unsqueeze(1), neg_sim], dim=1) # [batch, 1 + num_neg]

# Labels: positive is always at position 0

labels = torch.zeros(logits.shape[0], dtype=torch.long, device=logits.device)

# Standard cross-entropy loss

loss = F.cross_entropy(logits, labels)

return loss

# Quick test

batch_size = 4

embed_dim = 128

num_neg = 10

anchor = torch.randn(batch_size, embed_dim)

positive = anchor + 0.1 * torch.randn(batch_size, embed_dim) # slightly noisy positive

negatives = torch.randn(batch_size, num_neg, embed_dim)

loss = info_nce_loss(anchor, positive, negatives)

print(f"InfoNCE Loss: {loss.item():.4f}")

Run this, you’ll see the loss trying to make the positive similarity much higher than any negative.

Outcome: Understand how InfoNCE teaches models to distinguish positives from negatives.

You now truly understand InfoNCE:

- It turns representation learning into a multi-class classification problem (which of these many candidates is the real positive?).

- It uses softmax over similarities to push positives closer and negatives farther.

- It maximizes mutual information between views, all without labels.

This single loss function is why CLIP can connect images and text, why SimCLR learns amazing visual features, and why modern AI scaled so dramatically.

Module 5 — Implementing InfoNCE in PyTorch

we’ve reached the most exciting part actually training a contrastive model!

In Module 4 we understood the InfoNCE loss mathematically. Now we’re going to turn that math into a complete, working training loop.

By the end of this module you’ll have a full contrastive learning pipeline you can run on your own machine (or Colab) and start learning meaningful embeddings from images, no labels required!

Goal: Build a working contrastive learning training loop.

Key Concepts You’ll Use

- Batch positives: For every image in the batch, we create one augmented version (the positive pair).

- Batch negatives: All other images in the same batch act as negatives (this is called “in-batch negatives”).

- Temperature scaling (τ): A small number (usually 0.07–0.5) that controls how sharp the similarity distribution is. Lower τ = harder task.

- Similarity matrix: We compute cosine similarity between all pairs in the batch — this becomes the input to the InfoNCE loss.

The Full Pipeline (SimCLR-style)

Here’s exactly what we’ll build:

Dataset (e.g., CIFAR-10 or your own images)

↓

Strong image augmentations (two different views per image)

↓

Encoder (ResNet-18)

↓

Projection head (small MLP: 512 → 128)

↓

Normalize embeddings

↓

Compute similarity matrix

↓

InfoNCE Loss

↓

Backpropagate & train

This is the classic SimCLR setup — one of the simplest and most effective ways to use InfoNCE.

Hands-On Project: Train Your Own Contrastive Model

Here’s a complete, clean, runnable script. You can copy-paste this into a notebook or .py file.

python

import torch

import torch.nn as nn

import torch.nn.functional as F

import torchvision.transforms as T

from torchvision.datasets import CIFAR10

from torch.utils.data import DataLoader

from torchvision.models import resnet18

# ==================== 1. Strong Augmentations ====================

transform = T.Compose([

T.RandomResizedCrop(32, scale=(0.2, 1.0)),

T.RandomHorizontalFlip(),

T.RandomApply([T.ColorJitter(0.4, 0.4, 0.4, 0.1)], p=0.8),

T.RandomGrayscale(p=0.2),

T.ToTensor(),

T.Normalize(mean=[0.5, 0.5, 0.5], std=[0.5, 0.5, 0.5])

])

# ==================== 2. Simple Projection Head ====================

class ProjectionHead(nn.Module):

def __init__(self, in_dim=512, hidden_dim=512, out_dim=128):

super().__init__()

self.net = nn.Sequential(

nn.Linear(in_dim, hidden_dim),

nn.ReLU(),

nn.Linear(hidden_dim, out_dim)

)

def forward(self, x):

return self.net(x)

# ==================== 3. InfoNCE Loss ====================

def info_nce_loss(features, temperature=0.5):

# features shape: (batch_size * 2, embed_dim)

batch_size = features.shape[0] // 2

labels = torch.arange(batch_size).to(features.device)

# Compute similarity matrix

sim_matrix = F.cosine_similarity(features.unsqueeze(1), features.unsqueeze(0), dim=2) / temperature

# Positive pairs are at positions (i, i+batch_size) and (i+batch_size, i)

mask = torch.eye(batch_size * 2, dtype=torch.bool).to(features.device)

sim_matrix = sim_matrix[~mask].view(batch_size * 2, -1)

# Labels point to the positive pair

labels = torch.cat([labels + batch_size, labels])

loss = F.cross_entropy(sim_matrix, labels)

return loss

# ==================== 4. Model ====================

class SimCLR(nn.Module):

def __init__(self):

super().__init__()

self.encoder = resnet18(pretrained=False)

self.encoder.fc = nn.Identity() # remove final layer

self.projection = ProjectionHead(512, 512, 128)

def forward(self, x):

h = self.encoder(x)

z = self.projection(h)

return F.normalize(z, dim=1)

# ==================== 5. Training Loop ====================

dataset = CIFAR10(root='./data', train=True, transform=transform, download=True)

loader = DataLoader(dataset, batch_size=256, shuffle=True, num_workers=4)

model = SimCLR().cuda()

optimizer = torch.optim.Adam(model.parameters(), lr=0.001)

print("Starting contrastive training...")

for epoch in range(10): # small number for demo

total_loss = 0

for images, _ in loader:

# Create two augmented views

images = images.cuda()

view1 = images

view2 = images # in real code you would apply different augmentations here

# Forward

z1 = model(view1)

z2 = model(view2)

# Concatenate and compute loss

features = torch.cat([z1, z2], dim=0)

loss = info_nce_loss(features, temperature=0.5)

optimizer.zero_grad()

loss.backward()

optimizer.step()

total_loss += loss.item()

print(f"Epoch {epoch+1} | Avg Loss: {total_loss/len(loader):.4f}")

print("Training finished! You now have a contrastive model.")

What You Just Built

- Two augmented views per image = positive pairs.

- All other images in the batch = negative samples.

- The model learns to pull positives together and push negatives apart using InfoNCE.

You can now replace CIFAR-10 with your own dataset (photos, documents, medical images) and train meaningful embeddings without any labels!

Outcome: You can train your own contrastive representation model.

You now have a complete, working InfoNCE training pipeline:

- Strong augmentations

- ResNet encoder + projection head

- Proper InfoNCE loss

- Similarity matrix computation

This is the exact template used in SimCLR, MoCo, and many modern foundation models.

Module 6 — Major Models Using InfoNCE

You now understand the InfoNCE loss from the ground up, from the math to the training loop.

But theory is only half the story. The real magic happened when researchers took InfoNCE and turned it into foundation models that power today’s AI.

In this module we’ll meet the four most important models that made InfoNCE famous. Each one solved a different problem and opened a new door in AI.

Goal: Understand how InfoNCE powers modern foundation models.

SimCLR — Image Representation Learning (2020)

SimCLR (Simple Contrastive Learning of Representations) was the first big proof that InfoNCE could work amazingly well on images without any labels.

How it uses InfoNCE:

- Takes one image → creates two different augmentations (crop + color change + flip).

- These two views become the positive pair.

- Every other image in the batch becomes a negative.

- Trains with InfoNCE to make the two views of the same image have very similar embeddings.

Result: The ResNet learns rich visual features that transfer beautifully to downstream tasks like classification, detection, and segmentation — often beating supervised models.

MoCo: - Memory Bank for Negatives (2020)

MoCo (Momentum Contrast) solved a practical problem in SimCLR:

In a small batch you don’t have enough negatives, so the model doesn’t learn very well.

MoCo’s clever fix:

- Keeps a large memory bank (queue) of past embeddings (up to 65,000 negatives).

- Uses a momentum-updated encoder so the queue stays consistent.

- Still uses InfoNCE loss, but now with thousands of high-quality negatives instead of just the current batch.

This made contrastive learning stable and scalable, MoCo became the backbone for many later works.

CLIP: - Image-Text Contrastive Learning (2021)

CLIP (Contrastive Language-Image Pretraining) from OpenAI took InfoNCE to the next level by connecting images and text.

How it works:

- Takes an image and its caption as the positive pair.

- All other image-caption pairs in the huge batch are negatives.

- Trains two encoders (image + text) with InfoNCE so that matching image-text pairs have high similarity.

This is why CLIP can do zero-shot classification:

You just compare an image embedding with text embeddings like “a photo of a cat” vs “a photo of a dog”.

CPC: - Contrastive Predictive Coding (2018)

CPC (Contrastive Predictive Coding) was actually one of the earliest successful uses of contrastive learning (even before SimCLR).

It focuses on predictive learning:

- The model sees part of a sequence (e.g., first few audio frames or image patches).

- It predicts future parts.

- Uses InfoNCE to distinguish the real future (positive) from randomly sampled futures (negatives).

CPC proved that contrastive learning works great for sequential data : - audio, video, and time-series.

- Speech representation learning: Models like wav2vec 2.0 (built on CPC-style InfoNCE) learn powerful audio embeddings. This is why modern speech-to-text works so well even in noisy environments.

Module 7 — Negative Sampling Strategies

In the last few modules you learned what InfoNCE is and how the loss works.

But here’s the secret that separates good contrastive models from great ones: how you choose the negative samples.

The quality of your negatives can make or break the entire training.

Too easy → the model learns nothing.

Too hard or wrong → the model collapses or gets confused.

This module is all about the different strategies for picking negatives and how they affect the final embeddings.

Goal: Learn how negatives affect training.

In-batch Negatives (The Simplest & Most Common)

This is what we used in our SimCLR-style code:

- Every image in the current batch acts as a negative for every other image.

- Pros: Zero extra memory, very simple.

- Cons: If your batch is small (e.g., 256), you only have ~255 negatives per anchor → not enough diversity.

Most beginner implementations use in-batch negatives. It works surprisingly well when you use large batches (4096+).

Memory Banks (MoCo Style)

When you can’t afford huge batches, use a memory bank (or queue):

- Keep a FIFO queue of the last N embeddings (e.g., 65,536).

- For each new anchor, the negatives come from this queue instead of just the current batch.

- The queue is updated with momentum (slowly) so negatives stay consistent.

This gives you thousands of negatives with almost no extra GPU memory.

MoCo showed this trick dramatically improves representation quality.

Hard Negatives

Not all negatives are equal.

- Easy negatives: Completely unrelated things (a dog and a spaceship) — the model learns almost nothing from them.

- Hard negatives: Things that look similar but are actually different (two different dog breeds, or “cat” vs “kitten”).

Hard negative mining = deliberately picking the negatives that are closest to the anchor (highest similarity but still negative).

This makes training much more effective — the model is forced to learn fine-grained differences.

False Negatives (The Silent Killer)

Sometimes you accidentally treat two views of the same thing as negatives.

Example:

- In face recognition, two photos of the same person from different angles might end up in the same batch → one becomes a false negative.

False negatives confuse the model badly. Good strategies try to avoid them (e.g., using memory banks or careful batch sampling).

Real-World Example: Face Recognition Datasets

Think of training a face recognition system on millions of identities (like in ArcFace or Google’s FaceNet).

- Positive pair: Two photos of the same person (different lighting, angle, glasses).

- Negative pairs: Photos of everyone else.

If you only use in-batch negatives, you might have only 100–200 other people in one batch — not enough to learn fine distinctions between similar-looking people.

Top systems use:

- Huge memory banks (millions of past faces)

- Hard negative mining (find the faces that look most similar to yours but belong to someone else)

- Careful sampling to avoid false negatives (same person appearing twice in one batch)

This is exactly why your phone can recognize you even with new hairstyles or masks.

Hands-On: Experiment with Different Negative Strategies

Let’s see the effect in code. We’ll compare small batch vs large batch vs hard negatives.

Python

import torch

import torch.nn.functional as F

def compute_info_nce(anchor, positive, negatives, temperature=0.1):

# anchor & positive: [batch, dim]

# negatives: [batch, num_neg, dim]

pos_sim = F.cosine_similarity(anchor, positive, dim=1) / temperature

neg_sim = torch.einsum('bd,bnd->bn', anchor, negatives) / temperature

logits = torch.cat([pos_sim.unsqueeze(1), neg_sim], dim=1)

labels = torch.zeros(logits.shape[0], dtype=torch.long, device=logits.device)

return F.cross_entropy(logits, labels)

# ------------------- Experiment -------------------

batch_size = 32

dim = 128

# Small batch (only in-batch negatives)

anchor = torch.randn(batch_size, dim)

positive = anchor + 0.1 * torch.randn(batch_size, dim)

negatives_small = torch.randn(batch_size, batch_size-1, dim) # limited negatives

loss_small = compute_info_nce(anchor, positive, negatives_small)

print(f"Small batch (in-batch negatives) Loss: {loss_small.item():.4f}")

# Large batch simulation (more negatives)

negatives_large = torch.randn(batch_size, 1024, dim) # many more negatives

loss_large = compute_info_nce(anchor, positive, negatives_large)

print(f"Large batch / memory bank Loss: {loss_large.item():.4f}")

# Hard negatives (closest ones)

# Simulate by taking some negatives very close to anchor

hard_neg = anchor.unsqueeze(1) + 0.05 * torch.randn(batch_size, 32, dim)

loss_hard = compute_info_nce(anchor, positive, hard_neg)

print(f"Hard negatives Loss: {loss_hard.item():.4f}")

What you’ll observe:

- Small batch → higher loss (model doesn’t learn much)

- Large batch / memory bank → lower loss (better learning)

- Hard negatives → even stronger signal (model learns fine details)

Outcome: Understand how sampling strategy affects representation quality.

You now know the practical truth:

- In-batch negatives = simple but needs large batches.

- Memory banks = scalable and effective (MoCo style).

- Hard negatives = most powerful but needs careful mining.

- False negatives = the hidden danger you must avoid.

The choice of negative sampling strategy often matters more than the model architecture itself.

Module 8 — Temperature Scaling & Embedding Geometry

After completing the module you are able to answer how the loss actually shapes the embedding space.

You already know InfoNCE pushes positives closer and negatives farther.

But how strongly it pushes, and what shape the final space takes, is controlled by one tiny number: the temperature parameter.

In this module we’ll explore temperature, embedding collapse, why everything lives on a hypersphere, and how to actually see the geometry with your own eyes.

Goal: Understand how InfoNCE shapes embedding space geometry.

The Temperature Parameter (τ)

The temperature τ is a small positive number (usually 0.07 to 0.5) that appears in the denominator of InfoNCE:

What it does:

- Low τ (e.g., 0.07) → makes the distribution very sharp. The model is forced to make the positive similarity *much* larger than any negative. Harder training, but usually better final representations.

- High τ (e.g., 1.0) → makes the distribution softer. The model is more relaxed. Easier to train but representations are often weaker.

Think of τ as the “strictness knob”. Lower = stricter teacher.

Embedding Collapse: - The Silent Failure Mode

If temperature is too low, or you have no good negatives, something terrible can happen:

All embeddings slowly drift toward the same point in space.

When that happens:

- Every image has almost identical representation.

- Cosine similarity between any pair becomes ~1.

- The model has collapsed, it learned nothing useful.

InfoNCE + proper temperature + good negative sampling is what prevents collapse.

Hypersphere Embeddings, Why We Normalize

Almost every contrastive model (SimCLR, CLIP, MoCo, etc.) does this after the projection head:

python

z = F.normalize(z, dim=1) # L2 norm = 1

Why?

- It forces all embeddings to live on the **surface of a unit hypersphere**.

- Now similarity is purely about **angle** (cosine similarity), not magnitude.

- The entire embedding space becomes nicely constrained and easier to optimize.

This is why cosine similarity works so well in contrastive learning.

Feature Normalization in Practice

Here’s the tiny code snippet used in every modern implementation:

python

# After projection head

embeddings = F.normalize(embeddings, dim=-1) # L2 norm = 1 for every vector

This one line prevents magnitude explosion and keeps everything on the hypersphere.

Module 9 — Mutual Information Theory

We’ve built the full InfoNCE pipeline and seen it in action with SimCLR, MoCo, and CLIP. ow it’s time to go deeper, to the beautiful theoretical foundation that explains why InfoNCE works so well.

This module connects everything we’ve learned to one of the deepest ideas in information theory: Mutual Information.

Don’t worry, we’ll keep it intuitive and visual, just like the rest of the series.

Goal: Understand the theoretical foundation of InfoNCE.

Mutual Information, The “How Much Do Two Things Know About Each Other” Measure

Mutual Information (MI) between two random variables X and Y is a number that answers:

> “How much information do I gain about Y if I already know X?”

- High MI → X and Y are strongly related (e.g., an image and its caption).

- Zero MI → X and Y are completely independent.

In self-supervised learning we want to maximize the mutual information between two different views of the same data (e.g., two augmentations of the same image, or an image and its text description).

The problem? Computing true mutual information is extremely difficult (requires knowing the full probability distributions).

The MI Lower Bound Trick

Researchers found a clever workaround:

Instead of computing the exact MI (impossible), we can find a lower bound, a number that is always ≤ true MI.If we maximize this lower bound, we are guaranteed to increase the real mutual information. InfoNCE gives us exactly such a lower bound.

InfoMax Principle

The InfoMax principle says:

> “To learn the best possible representations, maximize the mutual information between different views of the same data.”

This is the guiding philosophy behind contrastive learning.

InfoNCE turns this abstract principle into a concrete, trainable loss.

Real-World Intuition

Think of two friends who know each other very well:

- If you show one friend a photo, they can instantly guess many things about the other friend’s description.

- That “instant guess” ability is high mutual information.

InfoNCE trains the model to become that kind of “good friend” between different views of the same data.

Module 10 — Advanced Research Topics

You now understand the loss, the training loop, the major models, temperature, and even the mutual information theory behind it all. But real research doesn’t stop at the basics. The community keeps pushing InfoNCE further to solve its remaining weaknesses. In this module we’ll explore the cutting-edge ideas that researchers are actively working on today.

Goal: Move to research-level understanding.

Hard Negative Mining

Not all negatives are equally useful.

Easy negatives (completely unrelated samples) give almost no learning signal.

Hard negatives (samples that look similar but are actually different) teach the model fine-grained distinctions.

Hard negative mining techniques:

- Mine the closest negatives in the batch or memory bank.

- Use semi-hard negatives (close but not too close) to avoid training instability.

This is widely used in face recognition (ArcFace) and retrieval systems.

Debiased Contrastive Loss

InfoNCE has a hidden bias: some “negatives” might actually be false negatives (different views of the same underlying object).

Debiased contrastive loss (from the 2020 paper by Chuang et al.) fixes this by:

- Estimating the probability that a negative is actually positive.

- Down-weighting those suspicious negatives in the loss.

Result: Cleaner representations, especially when the dataset has many near-duplicates (e.g., multiple photos of the same product).

Supervised Contrastive Learning

What if we do have some labels? Can we use them?

SupCon (Supervised Contrastive Learning, 2020) says yes:

- Instead of one positive per anchor, use all samples from the same class as positives.

- All other classes become negatives.

This combines the best of supervised and contrastive worlds and often beats standard cross-entropy on classification benchmarks.

Multi-Positive Contrastive Learning

Standard InfoNCE uses one positive per anchor.

Modern variants allow multiple positives:

- In SimCLRv2 and SwAV, multiple augmentations of the same image are all treated as positives.

- In CLIP, the matching image-text pair is the positive, but multi-positive extensions are being explored.

This makes the loss more stable and helps the model learn richer invariances.

Contrastive Learning Without Negatives (BYOL & SimSiam)

The biggest surprise came in 2020–2021:

What if we remove negatives entirely?

BYOL (Bootstrap Your Own Latent) and SimSiam showed it’s possible:

- Two networks (online + target).

- The online network predicts the target network’s representation.

- A stop-gradient and predictor network prevent collapse.

- No negatives, no memory bank, no InfoNCE — yet they achieve excellent performance.

These methods proved that contrastive learning’s power comes more from predicting consistent representations than from pushing negatives away.

Real-World Example: Face Recognition Datasets with Millions of Identities

In large-scale face ID systems (like those used by banks or airports):

- Hard negative mining finds look-alike imposters.

- Multi-positive learning uses multiple photos of the same person.

- Debiased loss handles cases where two different people have almost identical features.

- Some recent systems even experiment with negative-free approaches for better generalization.

This is why modern face recognition works so reliably even with new hairstyles, glasses, or lighting.

Outcome: Understand limitations and research directions.

You now see the full research landscape:

- InfoNCE is powerful but has limitations (false negatives, collapse risk, need for many negatives).

- Ongoing work focuses on smarter negatives, debiased losses, supervised/multi-positive variants, and even completely negative-free methods.

These advanced directions are exactly what researchers are using today to push contrastive learning even further.

From the very first “Why Contrastive Learning Exists” to mutual information theory and advanced tricks, you now have a complete understanding of InfoNCE.

From Single Modality to Multimodal Contrastive Learning

So far we’ve mostly seen InfoNCE within one modality (image ↔ image or text ↔ text).

The real breakthrough came when researchers asked:

“What if we connect completely different types of data using the same contrastive loss?”

The idea is simple and powerful:

- Take a pair that belongs together (e.g., an image and its caption).

- Treat them as positive pairs.

- All other mismatched pairs in the batch become negatives.

- Train two separate encoders (one for each modality) with InfoNCE.

The result? The image encoder and text encoder learn a shared embedding space where matching pairs are close and mismatched pairs are far apart.

Real Examples That Changed AI

1. Image ↔ Text (CLIP — OpenAI, 2021)

The most famous example. CLIP trains on 400 million image-text pairs from the internet.

Positive pair = image + its actual caption.

Negative pairs = all other combinations.

Result: Zero-shot classification, amazing image search, and the foundation for DALL·E and Stable Diffusion.

2. Audio ↔ Video

Models like CLIP-like audio-video contrastive learning (e.g., in VideoCLIP or MERLOT) learn to align what you see with what you hear. This powers modern video understanding and lip-reading systems.

3. Document Layout ↔ Text

Newer models (LayoutLMv3, Donut, UDOP) use contrastive learning between the visual layout of a document (via DiT-style vision backbone) and the actual text content. This is why they can understand forms, tables, and invoices so well.

Mini Project: Build a Simple CLIP-like Model

Here’s a minimal but complete CLIP-style training skeleton you can run (using tiny encoders for demonstration):

python

import torch

import torch.nn as nn

import torch.nn.functional as F

class SimpleImageEncoder(nn.Module):

def __init__(self, embed_dim=256):

super().__init__()

self.backbone = nn.Sequential(nn.Conv2d(3, 64, 3), nn.ReLU(), nn.AdaptiveAvgPool2d(1))

self.proj = nn.Linear(64, embed_dim)

def forward(self, x):

x = self.backbone(x).flatten(1)

return F.normalize(self.proj(x), dim=-1)

class SimpleTextEncoder(nn.Module):

def __init__(self, vocab_size=10000, embed_dim=256):

super().__init__()

self.embed = nn.Embedding(vocab_size, embed_dim)

self.proj = nn.Linear(embed_dim, embed_dim)

def forward(self, x):

x = self.embed(x).mean(dim=1)

return F.normalize(self.proj(x), dim=-1)

# InfoNCE loss (same as before, but now between two modalities)

def clip_loss(image_emb, text_emb, temperature=0.07):

logits = torch.mm(image_emb, text_emb.t()) / temperature

labels = torch.arange(logits.shape[0]).to(logits.device)

loss_i2t = F.cross_entropy(logits, labels)

loss_t2i = F.cross_entropy(logits.t(), labels)

return (loss_i2t + loss_t2i) / 2

# Training loop sketch

image_encoder = SimpleImageEncoder().cuda()

text_encoder = SimpleTextEncoder().cuda()

optimizer = torch.optim.Adam(list(image_encoder.parameters()) + list(text_encoder.parameters()), lr=1e-3)

for batch in dataloader: # batch = (images, text_tokens)

images, text = batch

images, text = images.cuda(), text.cuda()

img_emb = image_encoder(images)

txt_emb = text_encoder(text)

loss = clip_loss(img_emb, txt_emb)

optimizer.zero_grad()

loss.backward()

optimizer.step()

print(f"CLIP-style Loss: {loss.item():.4f}")

This tiny model already learns to align images with their captions using pure InfoNCE — exactly the same principle used in the real CLIP model (just with much bigger encoders).
