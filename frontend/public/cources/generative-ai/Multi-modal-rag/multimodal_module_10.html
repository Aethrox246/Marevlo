02 — ReAct

ReAct: The Classic Thought–Action–Observation Loop

ReAct stands for Reason + Act. Introduced by Yao et al. in 2022, it interleaves chain-of-thought reasoning with tool-using actions. The agent thinks about the next step, executes an action, observes the result, updates its reasoning, and repeats.

Animated ReAct Loop THOUGHT → ACTION → OBSERVATION → REPEAT
Phase 1

Thought

The controller LLM reasons about the next useful move.

Phase 2

Action

It chooses and parameterizes a tool call.

Phase 3

Observation

The tool returns evidence or a result from the environment.

Phase 4

Update

The next thought is conditioned on what was just observed.

When to Use

Use ReAct when tasks require iterative interaction with the outside world and the plan may need to change after every observation.

Why It Helps

It reduces hallucinations by grounding each next step in retrieved or observed evidence instead of reasoning in a vacuum.

TechVista example: a customer reports a malfunctioning laptop. The ReAct agent first reasons that it should verify warranty status, then queries the order database, observes that the purchase is eligible, next inspects the uploaded photos, observes no obvious physical damage, and finally reasons that this looks like a manufacturer fault and initiates a replacement.

Python · ReAct-Style Loop Sketch
while not done:
    thought = llm_reason(state)
    action = choose_tool(thought)
    observation = call_tool(action)
    state = update_state(state, thought, action, observation)
    done = should_finish(state)

ReAct is adaptive and transparent. Auditors can inspect the reasoning trace and see why the agent chose each action after each observation.

03 — Plan-and-Execute

Plan-and-Execute: Separate Strategy From Execution

Plan-and-Execute, also called Plan-and-Solve or Plan-Then-Act, divides the work into two phases. First the agent creates a complete ordered plan. Then an executor follows that plan and performs the steps.

Planner

Receives the task description and emits a structured sequence such as verify warranty, inspect damage, send replacement, and notify customer.

Executor

Follows the plan step by step, invoking tools or subroutines. If a step fails, it can retry or trigger replanning.

This style is best for long-horizon tasks where the sequence is mostly known in advance and interruptions are expensive. A full warranty workflow with verification, label generation, scheduling pickup, and follow-up notification fits this pattern well.

Core Trade-Off
more structure, less flexibility
Plan-and-Execute is efficient for predictable workflows, but less graceful than ReAct when unexpected new information appears midstream.

TechVista example: for a complicated warranty claim, the planner first outputs the entire sequence: retrieve order details, validate warranty eligibility, collect evidence, inspect damage, generate authorization, schedule pickup, notify customer. The executor then performs those steps in order. If photo inspection fails, the executor can pause and ask for replanning.

04 — Reflexion

Reflexion: Self-Critique and Revision for Higher Reliability

Reflexion extends the usual reasoning loop by adding a critique phase. Instead of treating each thought or action as final, the agent treats it as a draft, critiques it, and then revises before moving on.

Animated Reflexion Loop THOUGHT → ACTION → OBSERVATION → CRITIQUE → REVISE
Step 1

Thought

Produce an initial interpretation or next move.

Step 2

Action

Use the current idea to inspect evidence or trigger a tool.

Step 3

Critique

Ask whether something was missed, misread, or weakly supported.

Step 4

Revision

Refine the thought or redo the action with a better focus.

This is especially useful when you need reliability without expensive reinforcement learning. The reflections can be stored in memory, helping the agent avoid repeating the same mistakes later in the task and, in some designs, across future related tasks as well.

TechVista example: during photo inspection the agent first classifies the damage as minor. In Reflexion it then critiques itself: maybe scratches around the ports were missed. It zooms in, reruns analysis with a better crop, and updates the classification. That extra critique-revise loop can materially improve fairness in warranty decisions.

Reflexion costs extra time and tokens, so it is best used where trust and self-correction matter more than raw speed.

05 — Tree-of-Thoughts

Tree-of-Thoughts: Explore Several Hypotheses Before Deciding

Tree-of-Thoughts extends ordinary chain-of-thought by maintaining several candidate reasoning branches instead of one single line of thought. Each branch is a partial solution, and the agent evaluates which branches deserve further exploration.

Thought Generation

At each depth, generate several candidate thoughts or explanations.

Evaluation and Search

Score branches, then expand promising ones using BFS, DFS, or beam search while pruning weak branches that look unlikely to lead to the correct answer.

Python · ToT Branch Evaluation Sketch
branches = generate_candidate_thoughts(state)
scored = [(score_thought(branch), branch) for branch in branches]
scored.sort(key=lambda x: x[0], reverse=True)
active_branches = [branch for _, branch in scored[:beam_width]]

This style fits ambiguous problems. If the evidence could support several plausible stories, ToT forces the agent to consider alternatives before acting.

TechVista example: a blurry image of a cracked smartphone screen could suggest drop damage, manufacturing defect, or shipping damage. The agent creates several branches, calls different tools for each branch, evaluates the evidence, and then chooses the most plausible explanation. This makes the rejection or approval decision more defensible.

06 — Graph-of-Thoughts

Graph-of-Thoughts: Reason as a Network, Not Just a Tree

Graph-of-Thoughts generalizes Tree-of-Thoughts by representing reasoning as an arbitrary graph rather than a strict tree. Thoughts become nodes, edges capture dependencies, and the graph can support merging, feedback loops, reuse, and cross-branch interaction.

Thought Nodes

Each node stores an intermediate fact or conclusion, such as damage severity, purchase location, or recall eligibility.

Edges and Aggregation

Edges encode dependencies, and the agent can combine nodes, augment them with new evidence, or prune irrelevant ones.

GoT is powerful for interdependent tasks. A policy lookup may depend on time since purchase, damage severity, manufacturer recall history, and order channel all at once. A graph structure can represent those relationships more naturally than a simple tree.

That graph structure also allows merging and feedback loops. A policy node can influence a damage-decision node, which in turn can trigger a recall-history node, and the result can flow back into the main eligibility decision. This is why GoT fits complex, interdependent reasoning so well.

TechVista example: the agent constructs nodes for order history, damage classification, warranty policy, past complaints, and recall status. It then traverses and refines that graph until the evidence supports approve, repair, or reject.

The key structural difference from ToT is that GoT allows cross-branch merging and cycles. It is not just a search tree with more branches.

07 — ReWOO

ReWOO: Plan Tool Calls Up Front for Lower Latency

ReWOO stands for Reasoning Without Observation. Instead of alternating between reasoning and observations at every step, the agent plans all needed tool calls in advance, executes them, and only then integrates the results.

Animated ReWOO Flow PLAN → EXECUTE ALL → INTEGRATE
Stage 1

Planning

Generate the full list of tool calls before execution begins.

Stage 2

Execution

Run the planned calls, potentially in parallel, while the LLM waits.

Stage 3

Integration

Combine the returned results into the final answer or next decision.

Stage 4

Risk

Incorrect assumptions in the initial plan cannot be corrected until all calls complete.

ReWOO is useful when speed and token efficiency matter and the calls are mostly independent. TechVista example: checking replacement stock, expedited shipping cost, and warranty status can all be planned together and executed concurrently.

The trap is dependency risk. ReWOO is much less safe when later tool calls depend on facts that only earlier observations could have clarified. In a mixed dependent/independent warranty scenario, that upfront planning can bake in invalid assumptions, such as estimating the wrong next step before warranty status or purchase timing is actually confirmed.

08 — Self-Consistency

Self-Consistency: Sample Several Reasoning Paths and Vote

Self-consistency, also called majority voting, acknowledges that a single reasoning chain may go wrong by chance. Instead of trusting one path, the system samples several independent chains and then votes on the final answer.

Parallel Reasoning

Run the model several times with different sampling paths to produce independent reasoning trajectories.

Voting

Extract the final answers, count frequencies, and select the majority answer, optionally exposing agreement level as a confidence cue.

This works well when accuracy matters more than cost or latency and the task has a deterministic answer. The source example uses a refund calculation: if three chains produce ₹7,500, ₹7,500, and ₹7,000, the system returns ₹7,500 and can show the 2/3 agreement.

09 — Toolformer and Tool Calling

Toolformer and the Evolution From Sequential to Parallel Tool Use

Early tool-using agents relied on carefully hand-crafted prompts. Toolformer showed that models can instead be trained to decide which tools to call, when to call them, and how to integrate the results more natively.

Three Eras of Tool Calling

Sequential

Call one tool, wait for its result, then decide whether to call another. This is the default in ReAct and many Plan-and-Execute systems.

Parallel

Issue several independent calls at once, drastically reducing end-to-end latency when calls do not depend on one another.

Hybrid

Run independent calls in parallel and dependent calls sequentially, combining the best of both patterns.

Python · Hybrid Tool Orchestration Sketch
# First get the user/order identity because later calls depend on it
user = lookup_customer(query)

# Then run independent calls in parallel
stock_result = check_local_stock(user["city"])
shipping_result = estimate_shipping(user["city"])
warranty_result = verify_warranty(user["order_id"])

# Finally integrate the results into one answer
response = synthesize(stock_result, shipping_result, warranty_result)

TechVista example: the assistant may need to look up the order, check local stock, compute shipping, and verify warranty eligibility. A mature hybrid tool-calling agent runs the independent parts together while respecting dependencies where needed.

10 — Practical Comparison

How to Choose the Right Paradigm for a Real Task

ReAct

Best for: dynamic tasks that need step-by-step adaptation.

TechVista fit: open-ended troubleshooting and evolving support interactions.

Plan-and-Execute

Best for: structured long-horizon workflows with predictable sequences.

TechVista fit: full warranty claim processing from validation to customer notification.

Reflexion

Best for: high reliability and self-correction.

TechVista fit: borderline damage-classification or ambiguous evidence cases.

ToT / GoT

Best for: ambiguity, competing explanations, and interdependent reasoning.

TechVista fit: deciding among several plausible causes of damage or policy outcomes.

ReWOO

Best for: latency-sensitive tasks with many independent tool calls.

TechVista fit: fast status and eligibility checks where dependencies are minimal.

Self-Consistency

Best for: highest accuracy on deterministic decisions where cost is acceptable.

TechVista fit: high-stakes refund or eligibility decisions that need extra confidence.

The strongest production systems often combine paradigms. For a high-stakes warranty case, a structured long-horizon flow from Plan-and-Execute plus the self-critique of Reflexion can be a very strong pairing.

11 — Conclusion

Key Takeaways from Module 10

Dynamic and adaptive: choose ReAct.

Structured and long-horizon: choose Plan-and-Execute.

High reliability and self-correction: choose Reflexion.

Ambiguous multi-hypothesis reasoning: choose Tree-of-Thoughts or Graph-of-Thoughts.

Latency-sensitive independent actions: choose ReWOO with parallel calls.

Highest accuracy on deterministic tasks: choose Self-Consistency voting.

In 2026 production settings like TechVista, the practical defaults are clear: simple warranty photo checks can use ReAct or ReWOO, full claims with policy plus evidence plus follow-up can use Plan-and-Execute or Graph-of-Thoughts, and high-stakes eligibility decisions benefit from Reflexion plus Self-Consistency.

In production, these choices are really trade-offs between cost, latency, and reliability. ReWOO and parallel calls are attractive when speed dominates, Reflexion and Self-Consistency are worth the extra cost when accuracy and fairness dominate, and Plan-and-Execute or GoT are strongest when the task is long and structurally complex.
