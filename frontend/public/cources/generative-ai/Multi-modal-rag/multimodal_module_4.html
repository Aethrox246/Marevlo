A. Vision Encoder

This part takes raw pixels and turns them into rich, meaningful vectors. Most modern VLMs use Vision Transformers or ConvNeXt.

How it works
  • Split the image into 16×16 or 14×14 patches
  • Flatten each patch into a vector
  • Add position embeddings so the model knows where each patch came from
  • Run many transformer layers so each patch can attend to every other patch
  • Output a sequence of patch embeddings plus one special [CLS] token that acts as a compact summary of the whole image

Typical output: 256 or 576 patch embeddings plus the summary token, depending on image resolution and patch size.

Examples: CLIP uses ViT-L/14 with 1024-dimensional features, LLaVA uses CLIP’s ViT-L/336px, and SigLIP keeps a similar ViT backbone with a different loss.

Real-world example: Tesla Full Self-Driving style systems processing camera feeds to detect pedestrians and traffic signs.

PyTorch · Conceptual Patch Extraction
import torch

def extract_patches(image, patch_size=16):
    # image shape: [B, C, H, W]
    # patch_size is often 14 or 16 in modern ViT-style models
    B, C, H, W = image.shape
    patches = image.unfold(2, patch_size, patch_size).unfold(3, patch_size, patch_size)
    patches = patches.contiguous().view(B, C, -1, patch_size, patch_size)
    patches = patches.permute(0, 2, 1, 3, 4).reshape(B, -1, C * patch_size * patch_size)
    return patches

B. Language Backbone

This is usually a decoder-only LLM such as LLaMA-3, Mistral, Gemma, Qwen, or similar families.

It already knows how to chat, reason, and follow instructions. The VLM pipeline teaches it how to listen to images too.

Real-world example: vision-enabled chatbots like ChatGPT or Grok describing photos and answering questions about visuals.

Transformers · Decoder Stub
from transformers import AutoModelForCausalLM, AutoTokenizer

model = AutoModelForCausalLM.from_pretrained("gpt2")

C. Adapter / Projector

This is the smallest part, but often the most important one in modern VLMs, because it translates visual tokens into a form the language model can use.

In practice, this often means mapping something like 1024-dimensional vision features into a 4096-dimensional language-model hidden space. That translation step is exactly why the projector is frequently the most important trainable part.

Simple LLaVA-style version
  • Image features might arrive as 576 patches × 1024 dimensions
  • A two-layer MLP with GELU projects them into the language model’s hidden size
  • Example flow: 1024 → 4096 → GELU → 4096 for a LLaMA-7B hidden space
More advanced versions
  • Q-Former in BLIP-2 and InstructBLIP compressing 577 image tokens into only 32 smart tokens
  • Perceiver Resampler in Flamingo resampling arbitrary numbers of image tokens into fixed 64 to 128 tokens
  • Gated Cross-Attention in Flamingo and IDEFICS inserting cross-attention layers inside the LLM itself

These advanced connectors matter because they control how much visual detail survives the handoff into language. Q-Former is aggressive compression, Perceiver handles variable-length visual input, and gated cross-attention keeps visual access alive throughout generation.

Real-world example: medical imaging systems translating X-ray features into language tokens so an LLM can write a report such as a left-femur fracture description.

PyTorch · Simple MLP Projector
import torch.nn as nn

class Projector(nn.Module):
    def __init__(self, in_dim=1024, hidden_dim=4096):
        super().__init__()
        self.net = nn.Sequential(
            # Maps vision features into the LLM hidden space
            nn.Linear(in_dim, hidden_dim),
            nn.GELU(),
            nn.Linear(hidden_dim, hidden_dim),
        )

    def forward(self, image_features):
        # image_features could be [num_patches, 1024] from the vision encoder
        return self.net(image_features)
03 — Families

The Three Major VLM Architecture Families

Architecture Family Comparison RETRIEVAL · GENERATION · BRIDGING
Dual-Encoder

Retrieval-First

Separate image and text encoders project into the same embedding space and are trained contrastively.

Encoder-Decoder

Generation-First

A vision encoder plus a language decoder, with a smart connector such as Q-Former in the middle.

Vision-to-LLM

Bridging

Project visual tokens directly into an LLM or insert cross-attention so the model stays conversational.

Dual-Encoder (CLIP Style)

Flow: image goes to a vision encoder, text goes to a text encoder, and both are mapped into the same embedding size such as 768 dimensions.

Training: contrastive loss like InfoNCE, exactly as in the previous module.

Perfect for: reverse image search, product search, and zero-shot classification.

Main weakness: it is excellent for speed and retrieval, but it cannot generate rich textual descriptions of the image, only similarity scores.

Real-world example: Google reverse image search matching uploaded images to similar web visuals.

Pseudo-Code · CLIP-Like Dual Encoder
image_vec = vision_encoder(image)
text_vec = text_encoder(text)

image_vec = normalize(image_vec)
text_vec = normalize(text_vec)

similarity = image_vec @ text_vec.T
print(similarity)
Encoder-Decoder (BLIP / BLIP-2)

Flow: one vision encoder, one language decoder, and a connector in the middle.

BLIP-2 example: frozen ViT vision encoder, frozen Flan-T5 or OPT language model, and a trainable Q-Former with only about 188M parameters.

The Q-Former acts like a smart compressor asking the image which 32 things the language model really needs to know.

Trade-off: highly parameter-efficient, strong for generation, and much cheaper to adapt than a fully native multimodal giant.

Real-world example: caption generation for uploaded social-media images.

Vision-to-LLM Bridging

Flamingo: inserts gated cross-attention layers inside a frozen language model.

GPT-4o and Gemini: likely use native multimodal backbones internally, but from the outside the interface still looks like image tokens plus text tokens flowing into next-token prediction.

Trade-off: this is the strongest family for natural multi-turn conversation because the language model can stay in its native chat mode while still consuming visual tokens.

Real-world example: e-learning systems that explain diagrams, charts, and screenshots in conversational language.

Pseudo-Code · Forward Pass
vision_tokens = vision_encoder(image)
projected_tokens = projector(vision_tokens)

# projected_tokens now live in the same hidden space expected by the LLM
llm_input = concat(projected_tokens, text_tokens)
response = language_model.generate(llm_input)
print(response)
04 — Instruction Tuning

How VLMs Became Conversational

Before 2023, most VLMs were fixed-task systems. They could caption, answer a VQA prompt, or do a narrow image-language task. The major shift came when models like LLaVA moved into multimodal instruction tuning.

The key move: train on image-grounded conversations rather than only fixed question-answer templates.

  • User says: describe this photo in one sentence
  • Assistant responds in natural, conversational language
  • The conversation can continue over multiple turns

The source explicitly calls out that LLaVA generated about 150k multimodal conversations using GPT-4 plus images as synthetic instruction-following data. That is a crucial detail because it explains why these systems suddenly became chatty and flexible rather than rigid captioners.

Simple Rule
pretraining teaches alignment · instruction tuning teaches usefulness
Alignment teaches the model how image and language relate. Instruction tuning teaches it how to act like a helpful assistant.

Real-world example: customer-support bots that analyze uploaded product photos and provide tailored advice rather than just generic captions.

This is why modern VLMs feel like talking to a smart friend rather than a stiff captioning engine. They were not just trained to see. They were trained to answer, explain, and continue the conversation.

05 — Fashion App

Your Fashion App’s Journey Through VLM Architectures

The source uses a detailed scenario with a customer named Priya, and it is worth preserving because it shows how architecture choices connect to product design. Priya uploads a photo of a dress from a wedding and asks: Find me something similar, but in blue, and tell me how to style it for an office party.

Dual-Encoder: Best for Search

This is the quick matcher. It is like a fast librarian that instantly spots similarity without reading the whole book.

Step-by-step practical application
  • Collect catalog photos and descriptions such as handwoven silk with gold motifs
  • Encode both product images and product text into a shared space
  • Encode Priya’s uploaded photo into that same space
  • Retrieve the closest matching points, such as blue products with similar patterns

Why it is useful: instant visual search, fast recommendations, and less typing from the user. The source explicitly frames this as the kind of feature that can boost sales by roughly 20–30% in real e-commerce settings because users buy what visually matches their vibe.

For Priya, this means the app can quickly surface a blue version with similar border motifs instead of forcing her to browse manually through hundreds of items.

Encoder-Decoder + Q-Former: Best for Parameter Efficiency

This is the smart compressor. It keeps costs lower while still preserving important visual details.

Step-by-step practical application
  • The vision side extracts fabric texture, color palette, and drape details
  • The Q-Former asks a few smart questions and compresses the visual information into a small set of useful tokens
  • The decoder uses that compressed summary to explain why a product is a good match

Why it is useful: lower server and hardware cost, smoother behavior on basic devices, and better scalability when you are not a hyperscale company.

Vision-to-LLM: Best Chat Experience

This is the conversational star. It turns image understanding directly into multi-turn natural dialogue.

Step-by-step practical application
  • Project Priya’s image into language-friendly visual tokens
  • Feed those tokens into a large language model
  • Generate a natural response like: similar blue variant, styling ideas, and follow-up suggestions
  • Support follow-up questions such as whether the item is wrinkle-free

Why it is useful: this is the architecture family that builds trust and makes the shopping assistant feel human rather than transactional.

A concrete reply might be: This blue Banarasi-style option keeps the same pallu feel. Pair it with silver earrings and a neat bun for an office party. Better conversational advice like this can increase engagement and reduce returns because customers understand the recommendation, not just the match.

Instruction Tuning: Helpful Friend Mode

This is the personality trainer. It teaches a smart model how to answer in a warm, practical, instruction-following way.

Step-by-step practical application
  • Create sample conversations such as festive styling suggestions or out-of-stock alternatives
  • Tune the model to follow instructions precisely and politely
  • Add local context such as Odisha-specific jewelry or styling cues

Why it is useful: it reduces friction and builds emotional connection with users.

Tool-Use Readiness: The Next Frontier

This is the action-hero layer. The model does not just talk about what it sees. It can act using tools.

Step-by-step practical application
  • Connect the app to tools such as stock checkers, size guides, or return-policy systems
  • If Priya uploads a damaged-item photo, the agent can inspect the image and trigger the right support workflow
  • The system can compare the photo to the catalog or to past purchases automatically

Why it is useful: this is where VLMs stop being passive explainers and become active multimodal agents.

06 — Conclusion

Key Takeaways from Module 4

Vision-Language Models are the bridge between seeing the world and talking about it. By combining encoders, language backbones, adapters, and smart training, they power everything from retrieval to chat to tool-using multimodal agents.

Three-piece stack: vision encoder, language backbone, and adapter or projector.

Dual-encoder: best for fast retrieval and similarity search.

Encoder-decoder: strong generation with good parameter efficiency, especially with Q-Former style compression.

Vision-to-LLM bridging: best for natural multi-turn conversation and modern assistant experiences.

Instruction tuning: is what transformed rigid image models into useful conversational assistants.

Future direction: tool-integrated agents that not only understand visuals but act on them.

Grasping VLMs equips you to build or evaluate multimodal systems that are efficient, conversational, and practically useful in the real world.

In product terms, the choice is direct: choose Dual-Encoder when speed and search quality matter most, choose Encoder-Decoder when you want efficient generation, and choose Vision-to-LLM when you need a true conversational assistant.
