ReAct (Reason + Act)

Concept and intuition

  • What it is ΓÇô ReAct (Reasoning + Acting) was proposed by Yao et al. (2022) as a way to interleave chainΓÇæofΓÇæthought reasoning with toolΓÇæusing actions. During problem solving the agent alternates between thinking about the next step, executing an action (e.g., calling an external tool) and observing the outcome, then repeats. The reasoning trace keeps the agent focused and updates its plan; the action step allows it to interact with external data sources.
  • When to use ΓÇô Use ReAct when tasks require iterative interaction with the external world and the agent must adapt its plan on the fly. It is particularly useful for openΓÇæended tasks such as information lookup, troubleshooting or multiΓÇæstep operations where each action depends on observations. ReAct helps reduce hallucinations by grounding reasoning in retrieved information.

Architecture

The ReAct architecture forms a thoughtΓÇôactionΓÇôobservation loop. A controller (the LLM) generates a thought (Thought_i), uses it to choose and parameterize a tool call (Action_i), receives the toolΓÇÖs output (Observation_i), and then updates its internal state before proceeding to the next thought. The loop ends when the agent decides it has enough information to produce the final answer.

ΓÇïΓÇïImagine a customer contacts TechVista regarding a malfunctioning laptop. A ReActΓÇæbased agent would:

  1. Reason: ΓÇ£The laptop is under warranty; I should verify purchase details.ΓÇ¥
  2. Act: Query the order database tool for the customerΓÇÖs purchase record.
  3. Observe: Order details confirm a valid warranty.
  4. Reason: ΓÇ£I need to inspect the uploaded photos for physical damage.ΓÇ¥
  5. Act: Use an imageΓÇæanalysis tool to classify damage severity.
  6. Observe: The tool reports no obvious physical damage.
  7. Reason: ΓÇ£Issue seems like manufacturer fault; approve a replacement.ΓÇ¥
  8. Act: Initiate a replacement request and send a confirmation email.

Benefits

  • Adaptive ΓÇô can handle unexpected observations (e.g., missing receipt) by branching reasoning and actions.
  • Transparent ΓÇô reasoning trace helps auditors understand decisions.

PlanΓÇæandΓÇæExecute (PlanΓÇæandΓÇæSolve)

Concept and intuition

  • What it is ΓÇô Unlike ReActΓÇÖs stepΓÇæbyΓÇæstep reasoning, PlanΓÇæandΓÇæExecute (also called PlanΓÇæThenΓÇæAct or PlanΓÇæandΓÇæSolve) separates the process into two distinct phases. First, the agent plans a complete sequence of steps; then it executes them in order. Each phase can employ different prompts or even different models. The planning phase decomposes the task into subtasks, while the execution phase follows the plan and handles tool calls.
  • When to use ΓÇô Use PlanΓÇæandΓÇæExecute for longΓÇæhorizon tasks where the sequence of actions is mostly known in advance and interruptions are costly. It is well suited for tasks like processing a warranty claim (verification, shipping label generation, followΓÇæup) or composing long reports. Because execution follows a preΓÇædetermined plan, it is efficient but less flexible when new information emerges.

Architecture

  1. Planner: An LLM (or planning module) receives the task description and outputs an ordered list of steps (e.g., ΓÇ£Verify warranty,ΓÇ¥ ΓÇ£Inspect damage,ΓÇ¥ ΓÇ£Send replacementΓÇ¥).
  2. Executor: Another LLM or workflow engine executes each step in sequence. For each step it may still think/act internally but sticks to the original plan. If a step fails, the executor can adjust or replan.

This architecture decouples strategy (planning) from operations (execution) and reduces context switching.

For a complicated warranty case, the agent would first produce a complete plan:

  1. Retrieve order details.
  2. Validate warranty eligibility.
  3. Collect evidence (photos/videos).
  4. Inspect evidence for damage.
  5. Generate a replacement authorization.
  6. Schedule pickup and notify the customer.

After planning, the executor performs these steps using the appropriate tools. If the photo inspection step fails, the executor can reΓÇæinvoke the planner or adjust the remaining plan.

Reflexion / SelfΓÇæReflection

Concept and intuition

  • What it is ΓÇô Reflexion introduces a selfΓÇæcritique loop where the agent reflects on its own intermediate outputs and revises them. It treats each thought/action as a draft and adds a ΓÇ£critiqueΓÇ¥ phase that evaluates the output and suggests improvements. The Prompt Engineering Guide explains that Reflexion helps agents learn from trial and error by using verbal feedback instead of expensive reinforcement learning. SelfΓÇæreflections are stored in an explicit memory so the agent can avoid repeating mistakes.
  • When to use ΓÇô Reflexion is valuable when the agent needs to reduce hallucinations and improve reliability without access to large training data. It is suited for sequential decisionΓÇæmaking, reasoning and programming tasks. Use it when nuanced feedback is required and interpretability matters.

Architecture

The architecture extends ReAct by adding a critiqueΓÇôrevision loop

Thought ΓåÆ Action ΓåÆ Observation ΓåÆ Critique ΓåÆ Revised Thought ΓåÆ ΓÇª ΓåÆ Final Answer

After each observation, the agent generates a critique of its thought/action (ΓÇ£Did I miss something?ΓÇ¥). If the critique identifies issues, it revises its thought and possibly redoes the action. The reflections are stored in memory for later steps.

During photo inspection, the agent might initially classify the damage as minor. In Reflexion, it then critiques its own assessment: ΓÇ£I may have missed scratches around the ports; zoom in and reΓÇæinspect.ΓÇ¥ The agent calls the image tool again with a different focus and updates its classification. By iterating this critiqueΓÇæfix loop, the system reduces misclassification and improves the fairness of warranty decisions.

TreeΓÇæofΓÇæThoughts (ToT)

Concept and intuition

  • What it is ΓÇô TreeΓÇæofΓÇæThoughts extends chainΓÇæofΓÇæthought by maintaining a tree of candidate reasoning paths. Each ΓÇ£thoughtΓÇ¥ is a partial solution; the agent explores multiple branches in parallel. As described in the ToT framework, this approach lets the model evaluate intermediate thoughts and backtrack when a branch appears unpromising. Search algorithms like breadthΓÇæfirst or depthΓÇæfirst search can be used to systematically explore the tree.
  • When to use ΓÇô Use ToT for ambiguous or multiΓÇæsolution problems where exploring several candidate strategies improves success. For TechVista, this could involve diagnosing ambiguous damage photos (multiple possible causes) or suggesting product configurations.

Architecture

  1. Thought generation: At each depth, the agent generates multiple candidate thoughts (partial solutions).
  2. Evaluation: It evaluates each thought using heuristics or separate prompts (e.g., scoring how likely it leads to the correct answer).
  3. Search: Using a search strategy (BFS/DFS/beam search), it expands promising thoughts while pruning unpromising ones.
  4. Termination: When a satisfactory solution is found or the search budget is exhausted, the best thought path yields the final answer.

Suppose a customer submits a blurry image of a cracked smartphone screen. The agent generates multiple hypotheses: (1) screen cracked due to drop; (2) manufacturing defect; (3) packaging damage. It then performs branchΓÇæspecific actions: call the imageΓÇæanalysis tool to detect crack patterns; consult the manufacturing defect database; search shipping logs for damage reports. After evaluating these branches, it selects the most plausible cause and decides whether the warranty claim is valid. ToT ensures that alternate explanations are considered before rejecting a claim.

GraphΓÇæofΓÇæThoughts (GoT)

Concept and intuition

  • What it is: - GraphΓÇæofΓÇæThoughts generalizes the tree concept by representing thoughts as nodes in an arbitrary graph, with edges capturing dependencies. This framework allows thoughts to be combined, distilled or enhanced via feedback loops, approximating how human reasoning forms complex networks. GoT can integrate outputs from multiple reasoning branches and support recurrent connections.
  • When to use: - GoT is powerful for complex reasoning tasks involving multiple interrelated subtasks (e.g., policy lookup, image comparison and eligibility check in warranty processing). The ability to merge and reuse intermediate thoughts makes it suitable for problems that benefit from crossΓÇæbranch interactions.

Architecture

  1. Thought graph: Each thought node contains a piece of information or intermediate conclusion (e.g., ΓÇ£damage severity=minor,ΓÇ¥ ΓÇ£purchase location=onlineΓÇ¥).
  2. Edges: Directed edges encode dependencies or flows of information (e.g., an eligibility check depends on warranty status and damage severity). Cycles and feedback loops are allowed.
  3. Graph operations: The agent can combine thoughts (summarize multiple nodes), augment a thought with new information, or prune irrelevant nodes. The final answer is derived by traversing and aggregating across the graph.

In a complex warranty case, the agent builds a graph with nodes representing different facts: order history, damage classification, warranty policies, past user complaints, etc. Edges indicate how these facts influence each other. A ΓÇ£policy lookupΓÇ¥ node connects to ΓÇ£damage severityΓÇ¥ and ΓÇ£time since purchaseΓÇ¥ nodes. The agent iteratively refines this graph ΓÇô for example, adding a node for ΓÇ£manufacturer recallΓÇ¥ if the damage matches a known defect ΓÇô and uses it to decide whether to approve a claim, repair or replacement.

ReWOO (Reasoning Without Observation)

Concept and intuition

  • What it is: Existing augmented language models interleave reasoning and tool observations, which incurs significant token cost and latency. ReWOO (Reasoning WithOut Observation) decouples the reasoning process from tool calls: the LLM plans all tool calls in advance, then executes them without waiting for intermediate observations. After execution, the LLM integrates the results. This modular paradigm reduces repeated prompt concatenation and improves efficiency.
  • When to use: Use ReWOO when speed and token efficiency are critical and tasks involve independent tool calls. For example, fetching multiple unrelated pieces of information (order status, stock availability, shipping rates) can be planned simultaneously. ReWOO is riskier because incorrect assumptions in the planning stage cannot be corrected until all calls return.

Architecture

  1. Planning: The LLM generates a sequence of tool calls without executing them.
  2. Execution: An orchestration layer performs all planned calls (potentially in parallel) while the LLM remains idle.
  3. Integration: Once results are available, the LLM processes them and formulates the final answer.

A customer asks, ΓÇ£Are any replacement units available in my city? How much would expedited shipping cost, and can you verify my warranty status?ΓÇ¥ A ReWOO agent plans three calls (stock database, shipping API, order database), executes them concurrently, and then synthesizes the results into a single response. By not pausing after each call, the agent reduces response time and token usage.

SelfΓÇæConsistency + Majority Voting

Concept and intuition

What it is: SelfΓÇæconsistency (also called majority voting) addresses the stochastic nature of languageΓÇæmodel reasoning. Instead of relying on a single reasoning path, the model samples multiple reasoning trajectories independently and then votes on the final answer. The answer that appears most frequently is selected. This ensemble approach reduces the chance of errors due to early mistakes in a single chain.

When to use: Apply selfΓÇæconsistency when accuracy is paramount (e.g., legal or financial decisions) and latency and cost are less critical. It is useful for tasks with deterministic answers where reasoning variability can be averaged out.

Architecture

Parallel reasoning: Trigger the LLM to produce n independent chainΓÇæofΓÇæthought responses for the same query (with different sampling seeds).

Extraction: Extract the final answers from each response.

Voting: Count the frequency of each answer and select the majority. Optionally incorporate confidence scores.

Result: Return the consensus answer along with consistency metrics (e.g., 2/3 agreement).

Suppose the agent must compute the refund amount for a complex bundle purchase. Running three reasoning chains yields final answers: Γé╣7,500, Γé╣7,500 and Γé╣7,000. Majority voting selects Γé╣7,500. The system can display the consensus and reveal that 2/3 chains agreed, signaling confidence to the user.

Toolformer and ToolΓÇæCalling Evolution

Concept and intuition

  • What it is: Early agents relied on handΓÇæcrafted prompts to call tools. Toolformer shows that language models can be trained to decide which tools to call, when to call them, and how to incorporate results. It uses a selfΓÇæsupervised approach: starting with a preΓÇætrained model, it inserts toolΓÇæcall demonstrations into unlabeled text, lets the model practice calling those APIs, and fineΓÇætunes it to integrate the outputs. This training enables the model to choose appropriate tools during inference without manual prompting.

  • Evolution of tool calling: -
    • Sequential tool calls: The earliest functionΓÇæcalling APIs allowed an agent to call one tool, wait for the result, then decide whether to call another. This is the default in ReAct and PlanΓÇæandΓÇæExecute.
    • Parallel tool calls: More recent APIs support issuing multiple tool calls at once. An article on CodeAnt AI notes that parallel execution dramatically reduces total latency: four API calls that each take 300 ms sequentially add up to 1.2 s, whereas running them in parallel finishes in 300 ms. Parallel calls are ideal when operations are independent (e.g., fetching weather, calendar and traffic data simultaneously).
    • Hybrid (parallel + sequential): Agents can mix both strategies, running independent calls in parallel and then sequentially performing dependent calls. OpenAIΓÇÖs functionΓÇæcalling API and frameworks like LangChain or LangGraph support such orchestration.
  • When to use ΓÇô Train or fineΓÇætune agents with ToolformerΓÇæstyle data when you need native toolΓÇæselection capability and want to reduce prompt engineering overhead. Use parallel calls for latencyΓÇæsensitive workloads; use sequential calls when dependencies exist (e.g., retrieving a user ID before fetching orders).

TechVistaΓÇÖs customer assistant might need to: (1) look up the customerΓÇÖs order, (2) check local stock, (3) compute shipping cost and (4) verify warranty eligibility. A parallel toolΓÇæcalling agent executes (2)ΓÇô(3) concurrently while waiting for (1). A ToolformerΓÇætrained model learns to decide that these calls are necessary and orchestrate them efficiently.

Conclusion & Key Takeaways

Single-agent reasoning has evolved far beyond basic ReAct. Choose the style based on task characteristics:

  • Dynamic & adaptive ΓåÆ ReAct
  • Structured, long-horizon ΓåÆ Plan-and-Execute
  • High reliability & self-correction ΓåÆ Reflexion
  • Ambiguous / multi-hypothesis ΓåÆ Tree-of-Thoughts or Graph-of-Thoughts
  • Latency-sensitive with independent actions ΓåÆ ReWOO + parallel calls
  • Highest accuracy on deterministic tasks ΓåÆ Self-Consistency voting

In 2026 production systems at places like TechVista:

  • Simple warranty photo check ΓåÆ ReAct or ReWOO
  • Full claim with photo + policy + form ΓåÆ Plan-and-Execute or Graph-of-Thoughts
  • High-stakes eligibility ΓåÆ Reflexion + self-consistency voting
