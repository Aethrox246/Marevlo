02 — Contrastive

Contrastive Learning, CLIP Style

Contrastive pretraining is the king for alignment and zero-shot magic. The rule is simple: this photo matches this caption, and the others do not. Matching pairs are pulled together in embedding space while mismatches are pushed apart.

CLIP does not begin by trying to generate captions or answer questions. Instead it learns a massive matching game from internet-scale image-caption pairs.

  • For each batch, one image-caption pair is the positive match
  • All the other captions in that batch become negatives for that image
  • The model is rewarded for making the correct match stand out and the wrong ones fade away

Step-by-Step Diagram Explanation

  • Text captions go into a text encoder, usually a Transformer, which outputs one fixed-size vector for each caption
  • Images go into an image encoder, often a Vision Transformer, which cuts the image into patches and also outputs one vector
  • Those vectors are projected into the same shared space, often dimensions like 512 or 768
  • The similarity matrix compares every image to every caption in the batch
  • The bright diagonal cells are the true matches and the rest should be pushed down

The repeated puppy and hedgehog examples in the source are there to show that this is not a one-off. The same matching pattern is learned for every pair in the batch simultaneously.

InfoNCE Intuition
maximize the probability of the correct match among one positive and many negatives
This happens symmetrically: image-to-text and text-to-image both learn from the same contrastive structure.

The full intuition matters here. For each image, the model treats its real caption as the positive and the other N - 1 captions in the batch as negatives. Then it does the same thing from the caption side: the real image is the positive and all the other images are negatives. That is why people say CLIP uses a symmetric image-to-text plus text-to-image InfoNCE objective rather than a one-sided loss.

In practice, the model computes a probability distribution over the whole batch with softmax and temperature, then tries to make the probability of the true match as close to 100 percent as possible. That is what makes large batches so powerful: they give the model many more hard negatives to learn from in a single update.

That extra negative pressure is a big reason CLIP-style systems generalize so well in zero-shot settings. With more hard negatives in each batch, the model learns a sharper notion of what truly belongs together and what only looks superficially similar.

  • Embedding space: similar things should live nearby, such as golden retriever text near golden retriever photos
  • Contrastive learning: strengthen positives and weaken negatives
  • Batch size matters: more negatives create a stronger learning signal
  • CLIP scale: the source notes CLIP famously used batches of 32,768, which is why it needed huge compute
  • Zero-shot payoff: after training, brand-new phrases can retrieve relevant images even if the exact phrase never appeared during training

In a fashion app, this is what lets the model immediately connect a query like elegant blue Banarasi silk saree with zari border to the right images without extra task-specific retraining.

Real-world example: Instagram’s recommendation system can use contrastive learning to match uploaded photos with similar content and improve feed personalization.

PyTorch · Symmetric InfoNCE Loss
import torch
import torch.nn.functional as F

def info_nce_loss(image_embeds, text_embeds, temperature=0.07):
    image_embeds = F.normalize(image_embeds, dim=-1)
    text_embeds = F.normalize(text_embeds, dim=-1)

    logits = image_embeds @ text_embeds.T / temperature
    labels = torch.arange(logits.size(0), device=logits.device)

    # image -> text direction
    loss_i = F.cross_entropy(logits, labels)
    # text -> image direction
    loss_t = F.cross_entropy(logits.T, labels)

    # Final CLIP-style loss averages both directions
    return (loss_i + loss_t) / 2

# image_embeds: [batch, dim]
# text_embeds:  [batch, dim]
# temperature sharpens or softens the similarity distribution
# Larger batches mean more negatives and usually stronger alignment pressure.
03 — Other Signals

The Other Major Pretraining Signals

Captioning

Image to Text Generation

The model sees an image and predicts a natural description. This forces it to understand details deeply rather than only learning similarity structure.

Captioning is what teaches descriptive richness: not just that two examples belong together, but what is actually in the scene, what the texture looks like, what the fabric might be, and what objects or attributes deserve mention.

Masked Modeling

Guess What Is Missing

Hide parts of the image or text and train the model to reconstruct them. This is great for local structure, texture, layout, and fine visual details.

The easiest intuition is BERT-for-vision: remove a region, patch, or token and make the model infer what belongs there. That strengthens local understanding, especially for texture, spatial arrangement, and document layout.

This matters a lot for charts, tables, invoices, and fine-grained visual surfaces such as saree weave patterns, because the model is forced to recover missing local evidence rather than only learning broad global similarity.

Multi-Task

Mix Signals Together

Use contrastive training, captioning, and masking in one system. The module points to systems like Flamingo and PaLI as examples of mixing objectives for stronger representations.

The reason multi-task works so well is that each objective teaches a different skill: contrastive for alignment and zero-shot transfer, captioning for descriptive depth, masking for local detail, and the combination for better robustness and generalization.

The source’s message is not that one signal always wins. It is that different signals shape different strengths: contrastive for alignment, captioning for descriptive depth, masking for local structure, and multi-task training for breadth.

04 — Fine-Tuning

How to Adapt a Pretrained Multimodal Model Efficiently

Once pretraining is done, the next question is adaptation. Most teams cannot afford to retrain a full giant model from scratch for each new product. That is where LoRA and adapters come in.

Full Fine-Tuning vs LoRA FREEZE MOST THINGS · TRAIN SMALL PLUGINS

Full Fine-Tuning

Vision encoder update all
Projector update all
LLM backbone update all

Powerful, but expensive in memory, compute, and engineering time.

LoRA / Adapters

Vision encoder frozen
Projector tiny adapter
LLM backbone low-rank update

Freeze the base, train only roughly 1 to 5 percent of the parameters.

In multimodal systems, many teams apply LoRA only to the projector or the LLM while keeping the vision encoder frozen. In the source’s fashion example, that means you can fine-tune a model on Odisha-specific saree styles such as Pochampally ikat and Sambalpuri motifs overnight on a modest GPU.

That trade-off is the practical point: instead of paying for a full multimodal retrain, you preserve the expensive visual foundation and adapt only the smaller parts that need to learn the product-specific language and style behavior.

Preference Tuning

The next layer after basic fine-tuning is preference tuning. The rule is: helpful and accurate is better than helpful but hallucinated, which is better than rude or vague.

The module explicitly frames this through DPO and RLHF-style preference tuning: rank responses, then train the model to prefer the grounded and truthful ones.

Modern approaches like DPO are especially popular because they can optimize directly on preference pairs without needing the full complexity of classic RLHF pipelines.

This matters because without preference tuning, the model may confidently invent details like this saree is pure silk when the image actually shows a polyester blend. Preference tuning teaches the model a grounding habit: only say what you see.

That ranking logic is the key missing habit in many raw models. If you explicitly teach the system that helpful + grounded outranks helpful but guessed, then the model stops treating fluent guesses as acceptable answers. In e-commerce, that difference directly affects trust, returns, and disputes.

05 — Data

Data Curation: Garbage In Still Means Garbage Out

Multimodal data is messy. Internet images come with noisy captions, documents come with OCR errors, and tables get flattened into meaningless summaries if you are not careful.

  • Caption noise: a saree image labeled as beautiful woman in dress teaches the wrong association
  • OCR noise: scanned invoices or labels can turn Total: $100 into Tota1: $1OO
  • Chart or table labeling failures: a pricing table screenshot labeled only as some numbers never teaches row or column understanding

Practical curation steps from the source:

  • Filter data by CLIP score, for example image-text similarity above about 0.28
  • Run dedicated OCR systems like TrOCR and verify the outputs
  • Use synthetic data for clean table images with known labels
  • Generate synthetic tables, charts, and invoice-style documents with perfect labels when real-world structured data is too noisy
  • Use human-in-the-loop review in high-value domains like medical, legal, or authenticity-sensitive fashion

The workflow matters as much as the ingredients: score noisy pairs, remove the worst caption-image mismatches, clean OCR-heavy sources, generate synthetic structured examples for tables or charts, then reserve human review for the high-stakes tail. That is how messy web data turns into usable multimodal training data.

The source’s recommendation is sharp: in your app, a curated set of 5,000 high-quality saree photos with accurate descriptions is often better than 500,000 noisy examples. Quality wins.

Real-world example: datasets like LAION-5B use CLIP-based filtering to curate billions of image-text pairs for training systems such as Stable Diffusion.

Pseudo-CLIP · Filtering with Threshold and Review
def filter_pair(record, threshold=0.28):
    score = record["clip_score"]

    if score < 0.20:
        return "drop"
    if score < threshold:
        return "send_to_review"
    return "keep"

records = [
    {"id": "saree_photo_1", "clip_score": 0.41},
    {"id": "invoice_scan_2", "clip_score": 0.19},
    {"id": "table_image_3", "clip_score": 0.26},
]

for record in records:
    print(record["id"], filter_pair(record))
06 — Evaluation

Evaluation Traps and Grounding Failures

The sneaky problem in multimodal AI is that a model can sound fluent and plausible while being completely wrong about the image.

Grounding Failure vs Grounded Claim LOOK PLAUSIBLE ≠ BE CORRECT

Hallucinated Claim

“The woman is wearing a red saree with heavy embroidery.”

But the actual image shows a blue plain cotton saree.

Grounded Claim

“There is a tear on the border region near the lower right edge.”

The claim points to a visible location and can be verified in the image.

  • Looks correct but hallucinates: fluent language can hide false visual claims
  • Groundedness failure: the model ignores the image and parrots training bias
  • Generic instead of specific: it says damaged without locating the actual defect

How to catch these problems:

POPE / MME

Use grounding-sensitive benchmarks such as POPE and MME to probe whether the model is inventing objects, attributes, or unsupported details.

They matter because multimodal failure is often not random nonsense, but plausible-sounding object or attribute hallucination.

Human Checks

Ask a strict evaluation question: does the answer say only what is visible?

Visual Grounding

Require the model to point to bounding boxes or regions that support its claims, so the answer is tied to visible evidence instead of generic language.

In the fashion-return example, if a customer uploads a damaged saree photo, the model must ground the claim tear on the border to the actual spot. Grounded answers create trust and reduce disputes.

Real-world example: VQA and related grounding benchmarks are used to evaluate multimodal reasoning in systems deployed by companies such as Meta for LLaMA-based VLMs.

Pseudo-Code · IoU Grounding Validation
def claim_is_grounded(claim_box, detected_boxes, iou_threshold=0.5):
    for detected_box in detected_boxes:
        iou = compute_iou(claim_box, detected_box)
        if iou >= iou_threshold:
            return True
    return False

# If the claim “tear on the border” has no matching region,
# the answer should not be trusted as grounded.
# In practice you would also log the best IoU and attach the region to the answer.
07 — Conclusion

Key Takeaways from Module 3

Pretraining signals lay the multimodal foundation. Fine-tuning and curation are what turn that foundation into practical, grounded performance. Evaluation is what keeps fluent mistakes from becoming product failures.

Contrastive CLIP-style learning dominates for alignment and zero-shot transfer.

Captioning, masking, and multi-task signals add descriptive depth, local detail, and stronger generalization beyond simple alignment.

LoRA and adapters make efficient multimodal fine-tuning practical by updating only a small slice of the system instead of retraining everything.

Preference tuning combats hallucinations by explicitly teaching the model that grounded truth beats confident guessing.

Data curation should prioritize quality through CLIP filtering, OCR cleanup, synthetic examples, and human review in high-stakes domains.

Evaluation must focus on grounding, using POPE, MME, human checks, and region-based verification to catch fluent but wrong outputs.

In product terms, the recipe is direct: use contrastive pretraining for alignment and zero-shot matching, add captioning or masking when you need richer detail, adapt with LoRA instead of retraining the whole stack, and enforce grounding with preference tuning plus evaluation before the model reaches users.
