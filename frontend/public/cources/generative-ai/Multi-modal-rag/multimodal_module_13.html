02 — Memory Types

The Memory Stack That Stops Agents From Forgetting

Memory is not one thing. Production agents need multiple memory layers because remembering the last two chat turns is very different from recalling a similar claim from six months ago, or remembering that a tool failed earlier in the current workflow.

Short-Term Memory

What it is: short-term memory captures the recent exchanges between the user and the agent. In practice, this usually means including the last few turns directly in the prompt so the model can track pronouns, implicit references, and follow-up corrections.

When to use it: use short-term memory for immediate dialogue continuity whenever the agent must understand what “it,” “that one,” or “the earlier photo” refers to in the current session.

TechVista example: when a customer says “It won’t turn on after charging for an hour,” the short-term memory keeps that complaint active so the next question “Does it show any lights?” makes sense. Once the thread grows too long, the older turns are summarized and pushed into longer-term storage.

Long-Term Vector Memory

What it is: long-term memory lets the agent recall information beyond the current conversation by storing embeddings of documents, chat transcripts, product photos, or policy pages in a vector database.

Why it matters: this semantic memory stores factual knowledge and conceptual relationships, so the agent can retrieve general knowledge like laws, safety rules, and product details through RAG or knowledge-graph style retrieval instead of relying only on the live prompt.

TechVista example: embeddings of all past warranty claims, including damaged-phone photos, are stored. When a new claim arrives, the agent searches for similar cases and may discover that a specific model often shows the same defect pattern. Previous customer chats can also be retrieved so returning customers do not have to repeat their history.

Episodic Memory

What it is: episodic memory records specific experiences: what happened, in what order, and with what outcome. It typically stores dialogue-state snapshots, tool calls, intermediate outputs, and status changes for the current case.

When to use it: use episodic memory for long-horizon workflows where the agent must avoid repeating failed steps and must keep track of where the process stands.

TechVista example: a warranty claim may involve photo upload, receipt verification, warranty-period checking, and replacement approval. The agent logs each step, such as “asked customer to upload photo” and “photo verified as damage code D3.” If the receipt later fails OCR, the agent can look back and continue cleanly instead of asking for the same photo again.

Tool Memory

What it is: tool memory is a specialized long-term memory that records prior interactions with external tools, including inputs, outputs, failures, and the surrounding context.

Why it matters: by indexing tool behavior, the agent learns practical heuristics such as “use the image classifier when the photo is blurry” or “the warranty API requires a 9-digit order ID and purchase date.”

TechVista example: the agent learns that policy_lookup requires product ID plus purchase date, while the damage classifier works best with images above 800×600. It also remembers which version of a vision model was more reliable, so future tool choice becomes faster and less error-prone.

Memory Stack in Practice SHORT-TERM → LONG-TERM → EPISODIC → TOOL MEMORY
Layer 1

Current Chat

Hold the recent dialogue turns for immediate coherence.

Layer 2

Past Knowledge

Retrieve similar historical cases and domain facts semantically.

Layer 3

Session History

Remember what steps already happened in this specific workflow.

Layer 4

Tool Habits

Remember which tools, inputs, and parameters succeeded or failed.

The practical pattern is simple but important: short-term memory handles immediate dialogue continuity, long-term vector memory recalls domain facts and similar historical cases, episodic memory tracks progress within the current task, and tool memory reduces wasted trial-and-error when interacting with APIs, databases, classifiers, and retrieval systems.

03 — Human Oversight

Human-in-the-Loop: When the Agent Should Stop and Ask for Help

Even with strong memory, agents can still make mistakes, miss context, or act too confidently in ambiguous situations. Human-in-the-loop systems let the agent run autonomously most of the time, but pause and request human review when confidence is low, ethical judgment is required, or the cost of a wrong action is too high.

HITL is not a sign that the agent failed. It is a production feature. A trustworthy system knows when not to act alone.

Apply HITL in high-stakes domains like finance, healthcare, legal workflows, or expensive support decisions. It is also valuable for subjective judgments and for ongoing training, because the human correction becomes future supervision data.

TechVista example: most warranty claims are processed automatically. But when a damage description is ambiguous, the receipt is unusual, or the claim value is large, the system escalates to a human supervisor who inspects the photos and evidence before approving or declining the case. That lowers fraud risk, improves fairness, and creates useful correction data for future tuning.

04 — Safety and Guardrails

Why Autonomous Agents Need Boundaries

Multimodal agents introduce risks that plain chatbots do not: tool misuse, cascading action chains, unintended control amplification, prompt injection, privacy leaks, and silent schema errors. Guardrails are the system of checks that keep those risks contained.

Input Guardrails

Prompt safety filters, sensitive-data detection, prompt-injection defenses, schema checks, date/ID validation, and contextual compliance checks before tool execution.

Output Guardrails

Content moderation, fact-checking, sensitive-data filtering, structured-output validation, and self-critics or LLM-as-judge checks that inspect the generated result.

Identity and Access

Only authorized tools and services should be allowed to trigger sensitive actions like refunds, account edits, or policy overrides.

Behavioral Boundaries

Actions should be classified by risk so high-risk steps require approval, escalation, or explicit human sign-off.

Visibility and Control

All actions, tool calls, approvals, and outputs should be logged so the system can be audited, debugged, and interrupted when necessary.

Guardrail Chain INPUT CHECKS → POLICY CHECK → TOOL CALL → OUTPUT CHECKS
Step 1

Screen Input

Detect malicious prompts, irrelevant content, and bad parameters before they reach tools.

Step 2

Classify Risk

Check whether the requested action is low-risk, medium-risk, or approval-gated.

Step 3

Execute Safely

Run only the allowed tool with validated schema and approved identity scope.

Step 4

Verify Output

Validate facts, format, refund amount, and data redaction before sending or acting.

TechVista example: before the refund tool is called, a policy checker validates eligibility and a human-approval guardrail handles expensive or risky refunds. Input guardrails verify that only order IDs tied to verified purchases are processed. Output guardrails validate the refund amount, redact personal data before any email is sent, and ensure only authorized services can trigger the actual monetary action.

Guardrails are non-negotiable. Without them, multimodal agents can leak private information, call harmful tools, or confidently take the wrong action while appearing perfectly fluent.

05 — Evaluation

How We Measure Whether Agents Actually Work

AgentVista

Evaluates multimodal agents in realistic scenarios with long-horizon tool interactions across 25 sub-domains, often requiring more than 25 tool-calling turns.

Even the best cited model, Gemini-3-Pro with tools, achieved only 27.3% accuracy, showing how far current systems still are from robust human-level performance.

VisualWebArena

Tests agents on 910 tasks across Classifieds, Shopping, and Reddit-like environments, with image-text comprehension, spatial reasoning, OCR, and long interaction traces.

Baseline agents reached only about 16.4% success, highlighting how hard multimodal web interaction still is.

WebArena

Replicates diverse web environments such as e-commerce, forums, and code repositories, and scores functional correctness on tasks like navigation, search, and editing.

Humans reach around 78% success, while early GPT-4-based agents were much lower, often failing through abandonment, looping, or inconsistent planning.

AgentBench

Covers multiple environments such as math, websites, and interactive tasks, and compares many models on long-term reasoning and decision quality.

The broad lesson is that strong commercial models still struggle, and open-source models often lag further behind on long-horizon planning.

For practitioners, the lesson is not to watch a single accuracy number. Evaluate reasoning quality, tool use, long-horizon behavior, cost, latency, and safety under realistic workloads. Later work has shown that adding reflection and workflow memory can improve these scores, which means benchmark results should inform architecture decisions, not just leaderboard slides.

TechVista can build internal benchmark suites inspired by VisualWebArena and AgentVista, such as navigating shopping pages, interpreting product photos, checking policy pages, and answering warranty questions under time and cost budgets before allowing a new agent version into production.

06 — Cost and Latency

How to Keep Powerful Agents Fast Enough and Cheap Enough

Parallel Tool Calling

Run independent checks at the same time instead of sequentially whenever dependencies allow it, so latency grows with the slowest branch instead of the sum of all branches.

Semantic Caching

Reuse answers or partial results via vector similarity when the question is common or semantically close to a previous one, avoiding repeated expensive model calls.

Hybrid Model Routing

Use small language models for simple tasks and escalate to larger models only for harder reasoning, which is often the fastest path to good UX.

Context Optimization

Send only the evidence the model actually needs instead of dumping unnecessary documents, images, or long histories into the prompt.

Response Limits

Long outputs are expensive. Tight response limits lower both latency and cost, especially when the model tends to over-explain.

The source highlights a crucial systems insight: decoding output tokens can be up to 100× more expensive than processing input tokens. That means adding a little more structured context or a short summary can reduce long, rambling generation and lower total latency.

Optimization is also about avoiding hidden costs: wasted compute from hallucination loops, unnecessary tool calls, irrelevant queries, and oversized prompts. Practical strategies include input filtering, semantic caching, context optimization, response limits, and lightweight parallel guardrails that can fail fast before a costly model call completes.

TechVista example: a small, fast model first classifies user intent such as “warranty claim” versus “general question.” More complex work like policy interpretation is escalated to a larger model. Common questions such as “How long is the warranty on this laptop?” can be served from semantic cache, while prompt-injection filters and PII detectors run in parallel to cancel harmful requests early.

07 — Persistence

How Agents Remember Across Days Instead of Starting Over

Agents often need to remember facts across sessions or days. Stuffing old conversation history back into the prompt is expensive and brittle. A better design uses stateful tools: one function stores updates in an external database, another retrieves them later.

Persistent State Flow STORE → RECALL → CONTINUE → UPDATE
Step 1

Store Facts

Save claim IDs, uploaded evidence, pending documents, and tool results externally.

Step 2

Recall Later

Load the relevant state when the user returns, rather than rebuilding the case from scratch.

Step 3

Continue Work

Resume exactly where the process left off.

Step 4

Update State

Write back the new evidence and decision progress after the next step completes.

This pattern behaves like a small external memory system. The agent loads the prior state at the start of the session, continues the workflow, and writes back the updated status at the end. It reduces hallucination risk because the agent reads facts from storage instead of inventing them.

Python · Stateful Tool Pattern
def store_claim_state(claim_id, state_update):
    # Save structured state externally instead of bloating the prompt
    db.upsert({"claim_id": claim_id, **state_update})

def retrieve_claim_state(claim_id):
    # Load the latest known facts before continuing the workflow
    return db.get({"claim_id": claim_id})

state = retrieve_claim_state("TV-2026-1042")
if state and state.get("missing_document") == "receipt":
    next_step = "ask_for_receipt"
else:
    next_step = "continue_standard_flow"

TechVista example: a customer starts a warranty claim on Monday but forgets to upload the receipt. On Wednesday they return. The agent recalls the claim ID, the already uploaded photos, and the missing receipt requirement instead of making the customer restart. When the receipt arrives, a store_document style tool updates the case and a retrieve_claim style tool fetches the consolidated state for final processing.

08 — Custom Tools

Designing Multimodal Tools That Agents Can Use Safely

Multimodal agents need specialized tools for images, audio, and other inputs. Building these tools is not just a coding exercise. It also requires good schemas, logging, versioning, governance, and observability.

The ToolMaker work cited in the source makes the point sharply: agents that could synthesize and refine their own tools reached around 80% success, versus roughly 20% for agents restricted to static human-written tools in the reported setup. The lesson is not that every agent should invent tools, but that tool design quality strongly shapes agent capability.

JSON · Custom Multimodal Tool Schema
{
  "tool_name": "image_damage_classifier",
  "description": "Classify visible product damage from an uploaded image",
  "parameters": {
    "image_base64": "string",
    "product_type": "string",
    "request_id": "string"
  },
  "returns": {
    "damage_type": "string",
    "severity": "string",
    "bounding_boxes": "array",
    "tool_version": "string"
  }
}

A strong tool schema includes a name, a clear description, typed parameters, and a structured return format. Good production tools also log every call, monitor performance, expose version metadata, and define who is allowed to invoke them.

TechVista example: the company builds a custom image_damage_classifier that accepts a base64-encoded image and returns structured JSON with damage type, severity, and bounding boxes. Version metadata prevents old and new outputs from being mixed. Governance determines which agents are allowed to call it, and logging tracks every use for drift analysis, observability, and auditing.

A custom tool without schema discipline and governance can be as dangerous as no tool at all. Agents need clean parameter contracts, explicit outputs, and strong access controls.
