01 · Motivation

Why Linear Chains Break

LangGraph is used in production by companies like Klarna for millions of customer interactions, Replit for AI coding assistance, and Elastic for search intelligence. Learning LangGraph means learning the same orchestration framework behind some of the most sophisticated AI products currently in production.

The course frames the core problem with a workplace analogy: if you hired an employee and forced them to do step 1, then step 2, then step 3, with no ability to go back, clarify, retry, or adapt, they would fail at most real work. Real work needs judgment, correction, branching, and iteration.

That is exactly where linear AI chains break. LangChain LCEL chains flow in one direction. They work for simple tasks like summarization or translation, but for anything requiring iteration or judgment, they are too rigid.

Key positioning: LangGraph does not replace LangChain. It extends it. LangChain gives you the LEGO bricks such as LLMs, tools, and prompts. LangGraph gives you the blueprint that arranges those bricks into a machine that can branch, loop, and coordinate.

02 · Mental Model

What Is a Graph?

The module asks you to forget the jargon for a moment and think of a graph as a company org chart with superpowers.

Nodes = Employees

Each node has a specific job such as classifying emails, looking up orders, or writing responses.

Edges = Reporting Lines

Edges define who passes work to whom.

State = Shared Project Folder

Every employee reads from it and adds their work to it.

Conditional Edges = Manager Decisions

The manager decides whether work goes to Sales, Support, or somewhere else.

Cycles = Redo Capability

This is the ability to say “this is not good enough, go back and redo it.”

START / END

START is the front desk where work arrives. END is the outbox where completed work gets delivered.

In a traditional company or DAG, work only flows one direction: receptionist to manager to specialist to delivery. In a LangGraph company, the manager can send work back for revision, and specialists can ask other specialists for help. That is the power of the graph model.

05 · Core Concepts

The Three Building Blocks

Everything in LangGraph is built from exactly three things. If you deeply understand State, Nodes, and Edges, you can build anything.

Component 1: State

State is the shared whiteboard. Every node can read from it, and nodes return updates that get merged back into it.

Component 2: Nodes

Nodes are Python functions with one responsibility. They do one job, update state, and hand control to the next part of the graph.

Component 3: Edges

Edges are the routing layer. They decide where work flows next, including static and conditional routes.

Component 1: State, The Shared Whiteboard

Think of State as the whiteboard in a conference room. The key insight is the reducer behavior: do updates replace the old value or add to it?

ReducersWithout reducer vs with reducer
# WITHOUT reducer: each write REPLACES the previous value
# Like erasing the whiteboard and writing new
class State(TypedDict):
    current_step: str    # Node returns "step2" → old value gone

# WITH reducer: each write ADDS to the existing value
# Like writing on the whiteboard without erasing
from typing import Annotated
from langgraph.graph.message import add_messages

class State(TypedDict):
    messages: Annotated[list, add_messages]  # New messages APPEND
    current_step: str                        # Still replaces (no reducer)

Common Mistake #1

Forgetting the reducer on your messages field. Without it, every node response deletes the entire conversation history. The course calls this out bluntly: your agent literally gets amnesia after every step.

Wrong vs CorrectMessages field
# WRONG — messages will be overwritten every time
class BadState(TypedDict):
    messages: list          # No reducer! Each node replaces the entire list!

# CORRECT — messages accumulate
class GoodState(TypedDict):
    messages: Annotated[list, add_messages]  # Reducer appends!

Component 2: Nodes, The Employees

Each node is a Python function that does one job. A node should classify intent, or look up orders, or generate a response, but never all three at once.

Pro tip from the source: not every node needs an LLM. Logic nodes are faster, cheaper, and more predictable. Use LLMs only when you need reasoning or language generation.

The 4 Types of NodesSource-preserved
# TYPE 1: LLM Node — calls an AI model
def classify_intent(state):
    response = llm.invoke([...])  # Call GPT/Claude
    return {"intent": response.content}

# TYPE 2: Tool Node — calls an external API or database
def lookup_order(state):
    order = database.get_order(state["order_id"])  # External call
    return {"order_data": order}

# TYPE 3: Logic Node — pure Python, no LLM needed
def check_priority(state):
    if state["intent"] == "complaint" and state["is_vip"]:
        return {"priority": "urgent"}
    return {"priority": "normal"}

# TYPE 4: Response Node — generates final output
def compose_response(state):
    response = llm.invoke([SystemMessage(content=f"Priority: {state['priority']}"), ...])
    return {"messages": [response]}

Component 3: Edges, The Routing Manager

Edges decide who gets the task next. The module emphasizes conditional edges as the place “where the magic happens.”

Conditional EdgesRouting example
def route_by_intent(state) -> str:
    """The routing manager: decides who handles the task."""
    intent = state["intent"]
    if intent == "complaint":  return "complaint_handler"
    if intent == "order":      return "order_handler"
    return "general_handler"   # Default path

# Connect it to the graph:
graph.add_conditional_edges(
    "classify",          # AFTER this node finishes...
    route_by_intent,     # ...run this function to decide next node
    {                    # Map return values to node names:
        "complaint_handler": "handle_complaint",
        "order_handler": "handle_order",
        "general_handler": "handle_general"
    }
)
06 · Master Project

ARIA v1: Basic Intent Classifier

Time to build the first version of ARIA. In this module, ARIA learns to understand what users want, classify intent, and route requests to the right handler. No tools yet, no memory, no fancy stuff, just the foundation.

ARIA v1 ArchitectureInteractive graph view

Guided Exercise — Build ARIA v1 Step by Step

Step 1 of 5: Define ARIA's State

ARIA needs the conversation messages, the detected intent, and whether the request has been handled.

State DefinitionStep 1
from typing import TypedDict, Annotated
from langgraph.graph import StateGraph, START, END
from langgraph.graph.message import add_messages
from langchain_openai import ChatOpenAI
from langchain_core.messages import SystemMessage, HumanMessage, AIMessage

class AriaState(TypedDict):
    messages: Annotated[list, add_messages]  # Conversation history
    intent: str                               # What the user wants
    handled: bool                             # Is the request resolved?

Step 2 of 5: Create the LLM and Define Nodes

Node DefinitionsStep 2
llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)

def classify_intent(state: AriaState) -> dict:
    """ARIA's brain: figures out what the user wants."""
    response = llm.invoke([
        SystemMessage(content=(
            "You are an intent classifier for a company assistant named ARIA. "
            "Classify the user's message as EXACTLY one of: "
            "greeting, order_inquiry, complaint, general_question. "
            "Respond with ONLY the intent label, nothing else."
        )),
        state["messages"][-1]
    ])
    return {"intent": response.content.strip().lower()}

def handle_greeting(state: AriaState) -> dict:
    return {
        "messages": [AIMessage(content=(
            "Hi there! I'm ARIA, your company assistant. "
            "I can help with orders, answer questions, or handle any concerns. "
            "What can I do for you today?"
        ))],
        "handled": True
    }

def handle_order(state: AriaState) -> dict:
    response = llm.invoke([
        SystemMessage(content=(
            "You are ARIA, a company assistant specializing in orders. "
            "Help the customer with their order inquiry. "
            "If they haven't provided an order number, ask for it politely."
        )),
        *state["messages"]
    ])
    return {"messages": [response], "handled": True}

def handle_complaint(state: AriaState) -> dict:
    response = llm.invoke([
        SystemMessage(content=(
            "You are ARIA. A customer is unhappy. "
            "1. Acknowledge their frustration sincerely "
            "2. Apologize for the inconvenience "
            "3. Offer a specific solution or escalation path"
        )),
        *state["messages"]
    ])
    return {"messages": [response], "handled": True}

def handle_question(state: AriaState) -> dict:
    response = llm.invoke([
        SystemMessage(content="You are ARIA, a helpful company assistant. Answer accurately."),
        *state["messages"]
    ])
    return {"messages": [response], "handled": True}

Step 3 of 5: Define the Routing Logic

Routing LogicStep 3
def route_by_intent(state: AriaState) -> str:
    """ARIA's routing manager: sends work to the right handler."""
    routing_table = {
        "greeting": "greet",
        "order_inquiry": "order",
        "complaint": "complaint",
        "general_question": "question"
    }
    return routing_table.get(state["intent"], "question")  # Default: question

Step 4 of 5: Assemble the Graph

Graph AssemblyStep 4
graph = StateGraph(AriaState)

# Add all nodes (ARIA's team)
graph.add_node("classify", classify_intent)
graph.add_node("greet", handle_greeting)
graph.add_node("order", handle_order)
graph.add_node("complaint", handle_complaint)
graph.add_node("question", handle_question)

# Wire the edges (who reports to whom)
graph.add_edge(START, "classify")  # Every request starts with classification
graph.add_conditional_edges("classify", route_by_intent, {
    "greet": "greet",
    "order": "order",
    "complaint": "complaint",
    "question": "question"
})
# All handlers lead to END (request complete)
for handler in ["greet", "order", "complaint", "question"]:
    graph.add_edge(handler, END)

Step 5 of 5: Compile and Test

Compile and TestStep 5
aria = graph.compile()

# Test with different inputs
test_cases = [
    "Hello!",
    "Where is my order #12345?",
    "Your service is absolutely terrible!",

for msg in test_cases:

07 · Walkthrough

ARIA v1 Execution Trace

The source traces a concrete example: "Where is my order #12345?"

Step 1: START receives state

Messages = ["Where is my order #12345?"], intent = "", handled = False.

Step 2: Route to classify

The LLM analyzes the message and returns {intent: "order_inquiry"}. State is updated with that intent.

Step 3: Conditional edge runs

route_by_intent sees order_inquiry, returns order, and routes execution to the order handler.

Step 4: Order node runs

The order specialist prompt generates the response and returns updated messages plus handled=True. State now contains two messages.

Step 5: Static edge to END

The final state is returned to the caller.

Module transition: At the end of Module 1, ARIA can understand four kinds of requests and respond appropriately. Module 2 gives her tools so she can actually search, look up, and act.

09 · Understanding Check

Mini Quiz — Test Your Understanding

Q1: What is the difference between a DAG and LangGraph's graph model?

A1: DAGs have no cycles and cannot loop back. LangGraph supports cycles, enabling iteration and self-correction.

Q2: If you have messages: list with no reducer and two nodes both write messages, what happens?

A2: The second node overwrites the first node’s messages, so conversation history is lost.

Q3: Can a node access state fields it doesn't update?

A3: Yes. Nodes receive the full state and only need to return the fields they update.

Q4: What does graph.compile() do? Can you use a graph without compiling?

A4: It validates graph structure and creates a runnable app. Uncompiled graphs cannot be invoked.

Q5: In the ARIA example, what would happen if we forgot the add_messages reducer?

A5: Each handler would erase the classify node’s messages, so the response would lose the original user input.

10 · Failure Patterns

Common Mistakes — Learn From Others' Failures

Mistake #1

Forgetting the reducer on message lists.

Symptom: the agent “forgets” what the user said.

Fix: add Annotated[list, add_messages].

Mistake #2

Putting too much logic in one node.

Symptom: hard to debug, hard to test.

Fix: split into focused single-responsibility nodes.

Mistake #3

No path from a node to END.

Symptom: graph compilation fails because of orphaned nodes.

Fix: every node must have at least one outgoing edge.

Mistake #4

Conditional edge returns a value not in the mapping.

Symptom: runtime error for invalid routing key.

Fix: ensure the routing function returns only keys present in the mapping dict.
