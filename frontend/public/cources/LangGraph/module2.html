01 · Shift

This Changes Everything

Up to now, ARIA can only talk. After this module, she can do things: search the web, look up orders, send emails, and run calculations. The course frames this as the difference between a chatbot and an agent.

This is also the same core pattern behind serious production systems and products like ChatGPT’s Code Interpreter and Claude’s tool use.

The breakthrough: in 2023, OpenAI introduced function calling, where an LLM can output a structured request to call a specific function with specific arguments. Instead of only generating text, the model can say it wants to call something like search_web(query="quantum computing"), then your application executes it and feeds the result back into the model.

This creates a feedback loop: LLM reasons → calls tool → gets result → reasons more → maybe calls another tool → generates final response. That loop is ReAct: Reason + Act.

02 · Loop Logic

The ReAct Loop

The model decides when to use tools and which tools to use. You do not hardcode “always search first, then calculate.” The model’s reasoning picks the sequence.

Example

Simple Tool Use

For “What is 15% of 247?”, the model should call calculate.

For “What is the weather in Tokyo?”, it should call get_weather.

Chained Tool Use

Multi-Step Reasoning

For “What is the capital of the country with the tallest building?”, the chain is: search("tallest building") → get Dubai → search("capital of UAE") → get Abu Dhabi.

03 · Tool Design

Creating Tools

A tool is just a Python function with an @tool decorator. The decorator extracts the function name, docstring, and parameter types, and that metadata is what tells the LLM what the tool does and how to call it.

The docstring is not just for humans. The LLM reads it to decide when to use the tool. A bad docstring means the LLM may use the tool at the wrong time, or not use it at all.

Tool DefinitionsSource-preserved
from langchain_core.tools import tool

@tool
def search_orders(order_id: str) -> str:
    """Look up an order by its ID. Returns order status, items, and delivery date.
    Use this when a customer asks about a specific order."""
    # In production: call your actual order database
    fake_db = {
        "12345": "Order #12345: 2 items, shipped, arriving March 22",
        "67890": "Order #67890: 1 item, processing, est. March 25"
    }
    return fake_db.get(order_id, f"Order {order_id} not found")

@tool
def search_knowledge_base(query: str) -> str:
    """Search the company knowledge base for information about products,
    policies, and procedures. Use this for general questions."""
    return f"KB result for '{query}': Our return policy is 30 days..."

@tool
def calculate(expression: str) -> str:
    """Evaluate a mathematical expression. Use for any calculations.
    Examples: '15 * 0.18' or '(500 + 200) * 0.08'"""
    try:
        allowed = set('0123456789+-*/.() ')
        if all(c in allowed for c in expression):
            return str(round(eval(expression), 2))
        return "Error: Invalid characters in expression"
    except Exception as e:
        return f"Error: {e}"

@tool
def create_support_ticket(subject: str, description: str, priority: str = "normal") -> str:
    """Create a support ticket when an issue cannot be resolved immediately.
    Priority: 'low', 'normal', 'high', 'urgent'."""
    ticket_id = "TK-" + str(hash(subject))[:6]
    return f"Ticket {ticket_id} created: {subject} (priority: {priority})"

tools = [search_orders, search_knowledge_base, calculate, create_support_ticket]
04 · Agent Graph

Building the ReAct Agent

The central technical move in Module 2 is binding tools to the LLM and then creating a graph with an agent node, a tool node, and a loop edge back to the agent.

ReAct Agent Graphbind_tools + ToolNode + loop
from langgraph.graph import StateGraph, START, END
from langgraph.graph.message import add_messages
from langgraph.prebuilt import ToolNode
from langchain_openai import ChatOpenAI
from typing import TypedDict, Annotated

class AgentState(TypedDict):
    messages: Annotated[list, add_messages]

# KEY: bind_tools tells the LLM about available tools
llm_with_tools = ChatOpenAI(model="gpt-4o").bind_tools(tools)

def agent(state: AgentState):
    """The brain: reasons and optionally calls tools."""
    response = llm_with_tools.invoke(state["messages"])
    return {"messages": [response]}

def should_continue(state: AgentState) -> str:
    """Check if the LLM wants to use tools or is done."""
    last = state["messages"][-1]
    if hasattr(last, "tool_calls") and last.tool_calls:
        return "tools"   # LLM wants to use tools → execute them
    return "end"        # LLM is done → return response

# Build the graph
graph = StateGraph(AgentState)
graph.add_node("agent", agent)           # The brain
graph.add_node("tools", ToolNode(tools)) # The hands

graph.add_edge(START, "agent")
graph.add_conditional_edges("agent", should_continue, {
    "tools": "tools",
    "end": END
})
graph.add_edge("tools", "agent")  # THE LOOP: after tools, go back to agent!

aria_v2 = graph.compile()

graph.add_edge("tools", "agent") is what makes this an agent instead of a chain. That edge creates the cycle that lets the LLM call a tool, inspect the result, decide whether more information is needed, and continue until it is satisfied.

06 · Production Reality

Error Handling — Because Tools Fail

In production, APIs time out, databases crash, and rate limits fire. The agent needs to handle failures gracefully rather than crash or loop forever.

Robust Tool NodeCustom error handling
from langchain_core.messages import ToolMessage

def robust_tool_node(state: AgentState):
    """Execute tools with error handling."""
    results = []
    last = state["messages"][-1]
    tool_map = {t.name: t for t in tools}

    for call in last.tool_calls:
        try:
            fn = tool_map.get(call["name"])
            if not fn:
                results.append(ToolMessage(
                    content=f"Error: Unknown tool '{call['name']}'",
                    tool_call_id=call["id"]
                ))
                continue
            result = fn.invoke(call["args"])
            results.append(ToolMessage(content=str(result), tool_call_id=call["id"]))
        except Exception as e:
            results.append(ToolMessage(
                content=f"Tool error: {str(e)}. Please try a different approach.",
                tool_call_id=call["id"]
            ))
    return {"messages": results}
07 · Understanding Check

Mini Quiz

Q1: What does .bind_tools(tools) do to the LLM?

A1: It tells the LLM about available tools, including their names, descriptions, and parameter schemas, so the model can generate structured tool calls.

Q2: In the ReAct loop, what prevents infinite tool calling?

A2: The LLM decides to stop. When it has enough information, it returns a response without tool calls and should_continue routes to END.

Q3: Can the LLM call multiple tools in a single turn?

A3: Yes. Modern LLMs can generate multiple tool_calls in one response, and ToolNode executes all of them.

Q4: What happens if a tool returns an error message?

A4: The error message goes back to the LLM, which can try a different approach or explain the problem to the user.

08 · Failure Patterns

Common Mistakes

Mistake #1: Tool Docstrings Too Vague

Bad: "Search for stuff."

Good: "Search the web for current information. Returns top 5 results with titles and snippets."

Mistake #2: Not Handling Tool Errors

Symptom: the agent crashes when an API times out.

Fix: wrap tool calls in try/except and return error messages.

Mistake #3: Agent Calls Tools Forever

Symptom: 50+ tool calls for a simple question.

Fix: add a max_tool_calls counter in state with a circuit breaker.

09 · Research

ReAct: Synergizing Reasoning and Acting

The module cites Yao et al. (2023) and explains that interleaving reasoning with actions beats both reasoning-only and action-only approaches.

On HotpotQA, ReAct outperformed chain-of-thought by 6 points. On ALFWorld, it achieved 71% success versus 45% for act-only approaches.

The key insight is exactly the one this module operationalizes: reasoning helps plan tool use, and tool outputs ground further reasoning.
