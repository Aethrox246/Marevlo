Module 0: why DiT is important

Goal: Understand the real-world problem DiT solves.

What is Document AI?

Document AI is basically AI that "reads" documents like a pro—extracting info, classifying types, or spotting layouts and text. It's crucial for automating stuff like OCR (turning scans into editable text) or layout analysis (finding titles, tables, etc.). The paper highlights how it starts with vision-based tasks, using models to handle images of docs, then layering on text and layout. Without a solid image backbone, it all falls apart. In short, it's tech that saves hours on paperwork in biz, law, or gov.

Why ImageNet-style pretraining is not enough for documents

ImageNet is great for photos of dogs or cars, but documents? Nah. It's a labeled dataset for natural images, and docs don't have a massive labeled equivalent. The paper notes no "ImageNet for documents," so supervised pretraining on it misses doc-specific quirks like text density or formats (see Fig. 1's examples). DiT pretrained self-supervised on 42M unlabeled doc images from IIT-CDIP, beating ImageNet-tuned models like ViT.

Domain shift problem in OCR and layout models

Domain shift is when a model aces training data but bombs on real stuff—like a layout model trained on academic papers (e.g., PubLayNet) failing on messy invoices. The paper calls this "accuracy regression," hurting OCR or layout tasks due to format mismatches. DiT fights it by learning from diverse unlabeled docs, boosting scores like layout analysis from 91.0 to 94.9.

Why self-supervised learning matters

No need for pricey labels—self-supervised uses unlabeled data to learn patterns. DiT's MIM (Masked Image Modeling) masks image patches and predicts them via a doc-tuned tokenizer (dVAE on IIT-CDIP). It's scalable, generalizes well, and per the paper, sets SOTAs without human input.

Module 1 — From CNN to Vision Transformers

Goal: Understand the foundation architecture.

CNN vs Vision Transformer

CNNs (Convolutional Neural Nets) ruled image tasks for years—think ResNet sliding filters over pixels to catch local patterns like edges or textures. Great for docs, but they need tons of data and struggle with global context.

Vision Transformers? They flip the script, treating images like text in NLP. The paper says DiT uses a "vanilla Transformer" like ViT: no convolutions, just attention to relate distant parts. ViT scales better with data, outperforming CNNs on big sets (e.g., DiT beats ResNext on layout analysis). Downside? Needs more pretraining, which DiT handles self-supervised.

Real-world: CNNs might miss how a table header relates to far-away footnotes in an invoice; ViT "reads" the whole doc like a story.

How ViT splits image into 16×16 patches

ViT chops a 224x224 image into 196 non-overlapping 16x16 patches (224/16=14, 14x14=196). Each patch is a "word." The paper describes this for DiT: resize doc to 224x224, split, then feed as sequence.

Analogy: Like breaking a sentence into words—patches capture local info, but the Transformer links them globally.

Patch embeddings

Flatten each RGB patch (16x16x3=768 values) into a vector, then project to a fixed dim (e.g., 768) via a linear layer or conv. DiT uses this to turn raw pixels into learnable embeddings.

Positional encoding

Patches lose order when sequenced, so add 1D positional embeddings (learnable or sinusoidal) to track "where" each is. Paper: "After adding the 1d position embedding, these image patches are passed into a stack of Transformer blocks."

Transformer encoder block

Core: Stack of blocks with attention and MLP. Each processes the sequence, normalizing and adding residuals for stability.

Multi-head attention

Splits embedding into heads (e.g., 12), computes Query-Key-Value attention in parallel: score how patches relate, weigh values. Paper's DiT-B has 12 layers, 768 dim, 12 heads—like ViT.

Real-world analogy: Treat image patches like “words” in a sentence. Attention is the grammar linking them—e.g., in a bank statement, relating "Date" patch to transaction patches afar.

Code Implementation

Let's build a mini ViT from scratch in PyTorch. We'll do patch embedding and a simple transformer block. (Tested—it runs!)

```python

import torch

import torch.nn as nn

class PatchEmbedding(nn.Module):

def __init__(self, img_size=224, patch_size=16, in_chans=3, embed_dim=768):

super().__init__()

self.proj = nn.Conv2d(in_chans, embed_dim, kernel_size=patch_size, stride=patch_size)

def forward(self, x):

B, C, H, W = x.shape

x = self.proj(x).flatten(2).transpose(1, 2) # B, num_patches, embed_dim

return x

class MultiHeadAttention(nn.Module):

def __init__(self, embed_dim=768, num_heads=12):

super().__init__()

self.num_heads = num_heads

self.head_dim = embed_dim // num_heads

self.qkv = nn.Linear(embed_dim, embed_dim * 3)

self.proj = nn.Linear(embed_dim, embed_dim)

def forward(self, x):

B, N, C = x.shape

qkv = self.qkv(x).reshape(B, N, 3, self.num_heads, self.head_dim).permute(2, 0, 3, 1, 4)

q, k, v = qkv[0], qkv[1], qkv[2]

attn = (q @ k.transpose(-2, -1)) / (self.head_dim ** 0.5)

attn = attn.softmax(dim=-1)

x = (attn @ v).transpose(1, 2).reshape(B, N, C)

x = self.proj(x)

return x

class TransformerBlock(nn.Module):

def __init__(self, embed_dim=768, num_heads=12, mlp_ratio=4):

super().__init__()

self.attn = MultiHeadAttention(embed_dim, num_heads)

self.mlp = nn.Sequential(

nn.Linear(embed_dim, embed_dim * mlp_ratio),

nn.GELU(),

nn.Linear(embed_dim * mlp_ratio, embed_dim)

)

self.norm1 = nn.LayerNorm(embed_dim)

self.norm2 = nn.LayerNorm(embed_dim)

def forward(self, x):

x = x + self.attn(self.norm1(x))

x = x + self.mlp(self.norm2(x))

return x

class MiniViT(nn.Module):

def __init__(self, img_size=224, patch_size=16, in_chans=3, embed_dim=768, num_heads=12, num_layers=1, num_classes=10):

super().__init__()

self.patch_embed = PatchEmbedding(img_size, patch_size, in_chans, embed_dim)

self.pos_embed = nn.Parameter(torch.zeros(1, (img_size // patch_size)**2, embed_dim))

self.blocks = nn.ModuleList([TransformerBlock(embed_dim, num_heads) for _ in range(num_layers)])

self.head = nn.Linear(embed_dim, num_classes)

def forward(self, x):

x = self.patch_embed(x)

x = x + self.pos_embed

for block in self.blocks:

x = block(x)

x = x.mean(dim=1) # Global average pooling

x = self.head(x)

return x

# Test it

model = MiniViT()

input_tensor = torch.rand(1, 3, 224, 224)

output = model(input_tensor)

print(output.shape) # Should be torch.Size([1, 10])

```

Explanation: PatchEmbedding splits and embeds. MultiHeadAttention handles relations. TransformerBlock stacks attn + MLP. MiniViT ties it together. Run this on a doc image—fine-tune for classification!

Module 2 — Self-Supervised Learning for Images

Goal: Understand BEiT and Masked Image Modeling (MIM)

What is self-supervised learning?

Self-supervised learning is AI teaching itself without labels—using data to create its own "quizzes." The paper highlights it for Document AI: no need for human-tagged images, just massive unlabeled ones (like 42M from IIT-CDIP). It learns patterns by predicting hidden parts, scaling better than supervised methods.

MLM (Masked Language Modeling) in BERT

BERT's trick: Mask 15% of words in text, predict them from context. Like filling blanks in a sentence. This builds understanding without labels. The paper draws parallels—DiT adapts this to images.

BEiT idea: Masked Image Modeling

BEiT brings BERT to images: Mask patches (40% in DiT), feed corrupted input to Transformer, predict originals. Paper: "Inspired by BEiT, we use Masked Image Modeling (MIM) as our pre-training objective." DiT accepts patches, predicts visual tokens—learning global relationships in docs.

Why predicting pixels is bad

Raw pixel prediction (like in MAE) is tricky—pixels are continuous, high-dimensional, leading to blurry outputs or ignoring semantics. Paper notes it's inefficient; better to predict discrete stuff for meaningful learning.

Predicting discrete visual tokens instead

Instead of pixels, use discrete tokens from a tokenizer (dVAE). Paper: Retrain dVAE on docs for better tokens (codebook 8192). MIM predicts token indices at masked spots—sharper, semantic focus. Why? Tokens capture doc essence (text blocks, layouts) vs. raw RGB.

Real-world analogy: Like solving a partially erased document page. You guess missing words from context—DiT does that with image patches, piecing together a faded invoice or report.

Code

Simple PyTorch for MIM: Random mask patches, create input, predict tokens. (Builds on Module 1's ViT.)

```python

import torch

import torch.nn as nn

class MIMObjective(nn.Module):

def __init__(self, embed_dim=768, vocab_size=8192): # dVAE codebook size

super().__init__()

self.head = nn.Linear(embed_dim, vocab_size)

def forward(self, embeds, mask, true_tokens):

masked_embeds = embeds[mask] # Only masked positions

preds = self.head(masked_embeds)

loss = nn.CrossEntropyLoss()(preds, true_tokens)

return loss

def random_patch_masking(patches, mask_ratio=0.4):

num_patches = patches.shape[1]

num_mask = int(num_patches * mask_ratio)

indices = torch.randperm(num_patches)[:num_mask]

masked = patches.clone()

masked[:, indices] = 0 # Or [MASK] token embedding

return masked, indices

# Usage (assume vit from Module 1, tokenizer)

masked_patches, mask_idx = random_patch_masking(patches)

embeds = vit.blocks(masked_patches) # Transformer

true_tokens = tokenizer.get_tokens(patches[:, mask_idx]) # From dVAE

loss = mim_head(embeds, mask_idx, true_tokens)

```

Explanation: Masks 40% patches randomly (blockwise in paper), runs through encoder, predicts discrete tokens via head. Train on unlabeled docs—no labels!

Module 3 — The Tokenizer Problem (dVAE)

Goal: Understand why DiT trains its own tokenizer.

What is dVAE?

dVAE = discrete Variational AutoEncoder

It’s the model that turns a continuous image (or image patch) into a small set of discrete codes — think of it as an image compressor that speaks in a fixed vocabulary.

- Encoder: takes image → compresses it down (usually by factor of 8×)

- Quantization layer: maps the compressed feature to the nearest entry in a learnable codebook

- Decoder: reconstructs the image from the chosen code

BEiT borrowed the dVAE that was trained for DALL·E on 400 million natural photos. DiT said: “No thanks” and trained its own on 42 million document pages from IIT-CDIP.

Codebook

The codebook is just a big lookup table — usually 8192 entries (DiT uses exactly that).

Each entry is a small vector that represents one possible “visual word”.

During training, the encoder’s output is forced to snap to one of these 8192 vectors — that’s discrete tokenization.

Vector quantization

This is the snapping step.

The encoder outputs a continuous vector → find the closest codebook vector (Euclidean distance) → replace with that exact codebook entry → pass the index (0–8191) as the discrete token.

No gradients flow through the codebook lookup directly, so they use tricks like straight-through estimator or commitment loss to make training stable.

Reconstruction loss (MSE)

The decoder tries to rebuild the original image from the quantized codes.

Loss = Mean Squared Error between original pixels and reconstructed pixels.

This pushes the codes to capture important visual information.

Perplexity loss

Just reconstruction isn’t enough — the model might collapse to using only a few codes.

So they add a perplexity / entropy term (or commitment loss) that encourages the codebook to be used evenly.

In DiT they say: “a perplexity loss to increase the use of the quantized codebook representations.”

Real-world example: Why DALL·E tokenizer fails on tables, borders, text edges

Look at Figure (it show side-by-side reconstructions):

- DALL·E tokenizer (natural images) → blurry lines, smudged table borders, text edges bleed together

- DiT’s dVAE (trained on documents) → sharp horizontal/vertical lines, clean cell borders, crisp text edges

Why? Natural photos have smooth gradients, textures, lighting. Documents have hard edges, uniform backgrounds, thin lines, dense small text.

A tokenizer trained on cats and mountains simply doesn’t have good codes for “thin black horizontal line” or “grid cell boundary” — so reconstruction looks bad and the discrete tokens carry less useful information for downstream document tasks.

Code: Simplified VQ-VAE example

Here’s a minimal PyTorch snippet showing the quantization step + reconstruction idea (you can run this conceptually — real training needs more code).

```python

import torch

import torch.nn as nn

import torch.nn.functional as F

class VectorQuantizer(nn.Module):

def __init__(self, num_embeddings=8192, embedding_dim=64):

super().__init__()

self.embedding = nn.Embedding(num_embeddings, embedding_dim)

self.embedding.weight.data.uniform_(-1/num_embeddings, 1/num_embeddings)

def forward(self, z):

# z: (B, C, H, W) → flatten to (B*H*W, C)

z_flat = z.permute(0, 2, 3, 1).reshape(-1, z.shape[1])

# distances: (N, K) where N = B*H*W, K = codebook size

distances = (torch.sum(z_flat**2, dim=1, keepdim=True)

+ torch.sum(self.embedding.weight**2, dim=1)

- 2 * torch.matmul(z_flat, self.embedding.weight.t()))

encoding_indices = torch.argmin(distances, dim=1) # closest code

quantized = self.embedding(encoding_indices).view(z.shape) # lookup

# straight-through estimator

e_latent_loss = F.mse_loss(quantized.detach(), z)

q_latent_loss = F.mse_loss(quantized, z.detach())

commitment_loss = e_latent_loss + q_latent_loss

quantized = z + (quantized - z).detach() # trick to allow gradients

return quantized, encoding_indices, commitment_loss

class SimpleVQVAE(nn.Module):

def __init__(self):

super().__init__()

self.encoder = nn.Sequential(

nn.Conv2d(1, 64, 4, stride=2, padding=1), # grayscale doc

nn.ReLU(),

nn.Conv2d(64, 64, 4, stride=2, padding=1),

)

self.quantizer = VectorQuantizer(8192, 64)

self.decoder = nn.Sequential(

nn.ConvTranspose2d(64, 64, 4, stride=2, padding=1),

nn.ReLU(),

nn.ConvTranspose2d(64, 1, 4, stride=2, padding=1),

)

def forward(self, x): # x: grayscale document patch (B,1,64,64)

z = self.encoder(x)

quantized, indices, vq_loss = self.quantizer(z)

recon = self.decoder(quantized)

recon_loss = F.mse_loss(recon, x)

return recon, recon_loss + vq_loss, indices

# dummy test

model = SimpleVQVAE()

img = torch.rand(2, 1, 64, 64) # pretend grayscale doc patch

recon, loss, tokens = model(img)

print(tokens.shape) # e.g. (2*16*16,) → flattened token indices

print(loss.item()) # reconstruction + vq loss

```

Explanation:

- Encoder compresses grayscale document patch

- VectorQuantizer snaps to nearest codebook vector

- Decoder reconstructs

- Loss = MSE (sharp edges) + commitment (use full codebook)

This is the core idea DiT used — but trained on millions of real document pages instead of natural photos.

Module 4: Complete DiT Architecture

Goal: Combine everything.

You’ll see how DiT takes a raw document image → pre-trains self-supervised → becomes a strong backbone for downstream Document AI tasks.

The DiT Pre-training Pipeline (step-by-step from Figure 2)

1. **Resize image to 224×224**

All input document images are resized to 224×224 (standard ViT input size). They use random resized cropping during training for augmentation.

2. **Split into 16×16 patches**

224 ÷ 16 = 14 → 14×14 = 196 non-overlapping patches.

Each patch is flattened and linearly projected → patch embeddings (usually to dim 768 for base).

3. **Mask 40% of patches**

DiT uses **blockwise masking** (like BEiT) — not fully random, but contiguous blocks for better learning of structure.

Masked patches are replaced with a learnable [MASK] token or zeroed (implementation detail varies).

4. **Pass into Transformer encoder**

Sequence of 196 embeddings + learnable 1D positional embeddings → fed into stack of Transformer blocks.

Each block: LayerNorm → Multi-head Self-Attention → Add & Norm → MLP → Add & Norm.

5. **Predict discrete token IDs**

Only the output embeddings at **masked positions** go through a lightweight prediction head (linear layer).

Head outputs logits over the tokenizer’s codebook size (8192 in DiT).

Goal: predict the correct discrete visual token ID that the dVAE would have produced for that patch.

6. **Cross-entropy loss**

Standard classification loss between predicted logits and ground-truth token indices from the dVAE tokenizer.

No reconstruction — just token prediction.

```python
import torch.nn as nn

import torch.optim as optim

from torch.utils.data import DataLoader

# Assume these are defined

class DiT(nn.Module): # Your ViT encoder + MIM head

def __init__(self, ...):

super().__init__()

self.patch_embed = PatchEmbedding(...)

self.pos_embed = nn.Parameter(...)

self.blocks = nn.ModuleList([TransformerBlock(...) for _ in range(12)]) # DiT-B

self.head = nn.Linear(768, 8192) # codebook size

def forward(self, x, mask_indices):

x = x + self.pos_embed

for blk in self.blocks:

x = blk(x)

x = self.norm(x)

masked_x = x[:, mask_indices] # only masked positions

logits = self.head(masked_x)

# Training loop

# 1. Get true tokens from dVAE tokenizer (frozen)

with torch.no_grad():

true_tokens = dvae_tokenizer.encode(images) # B,14,14 → flattened B*196

# 2. Create masked input

mask, mask_indices = blockwise_masking()

masked_images = images.clone()

# In real impl: replace masked patches with mask token embedding after patch_embed

# 3. Forward

logits = model(images, mask_indices) # shape: (B*num_masked, 8192)

# 4. Gather true tokens only for masked positions

B = images.size(0)

num_masked = mask_indices.numel()

true_masked = true_tokens.view(B, -1)[:, mask_indices].reshape(-1)

# 5. Loss

loss = criterion(logits, true_masked)

optimizer.zero_grad()

loss.backward()

optimizer.step()

if step % 100 == 0:

print(f"Step {step}, loss: {loss.item():.4f}")

```

- We only compute loss on masked positions (efficiency).

- In practice you’d add stochastic depth, warmup scheduler, gradient clipping, etc.

- The dVAE tokenizer is frozen — only the DiT encoder + head are trained.

Module 5: Fine-Tuning for Classification

In this module we focus on the **RVL-CDIP** dataset (the one the DiT paper uses to get from 91.11% → 92.69% overall accuracy), and we’ll walk through exactly how DiT turns its powerful pre-trained backbone into a classifier.

Goal: Apply DiT to RVL-CDIP dataset

RVL-CDIP has 400,000 grayscale document images across **16 classes** (letter, form, email, invoice, resume, memo, scientific report, etc.). It’s a classic benchmark for visual-only document classification — no OCR text is used in the pure-image setting.

Topics

Average pooling for global representation

After the Transformer encoder processes all 196 patches, you get a sequence of 196 embeddings (shape: B × 196 × hidden_dim).

To get one vector that represents the **whole document**, the paper uses **average pooling** across the patch dimension:

```text

global_feat = patch_embeds.mean(dim=1) # → B × hidden_dim

```

This is simple, parameter-free, and works surprisingly well for classification (better than taking [CLS] token in many vision cases).

Linear classifier head

Just slap a single linear layer on top:

```text

classifier = nn.Linear(hidden_dim, num_classes) # 768 → 16 for RVL-CDIP

logits = classifier(global_feat)

```

No fancy MLP or extra layers — DiT paper keeps it minimal.

Fine-tuning strategy

- Image size: resize to 224×224 with RandomResizedCrop (strong augmentation)

- Epochs: 90

- Batch size: 128

- Learning rate: 1e-3 (quite high — leverages strong pre-training)

- Optimizer: Adam (same as pre-training)

- No fancy schedulers mentioned — they likely use cosine decay or step LR

- Input: pure grayscale images (no OCR text)

They also report a “(C)” version with extra tricks (probably center crop or color jitter), pushing accuracy higher (DiT-L (C) reaches 94.9%).

Key insight: Because of the strong self-supervised pre-training on 42M diverse docs, you can fine-tune aggressively with high LR and still get great generalization.

Real-world example: Auto-sorting company emails into categories

Imagine your company in Bhopal receives hundreds of scanned emails, invoices, resumes, and forms every day.

Without good classification:

- Humans manually sort → slow & expensive

- Wrong category → delays in processing invoices or missing urgent resumes

With DiT fine-tuned on RVL-CDIP-like data:

- Scan arrives → resize to 224×224 → DiT backbone → average pool → linear head → predicts “invoice”, “resume”, “memo”, etc. with 94%+ accuracy

- Route invoice to accounts payable, resume to HR, memo to relevant team — fully automated

This is already happening in banks, insurance, logistics companies — DiT-style models make it more accurate on messy real scans.

```python
class DiTClassifier(nn.Module):
```

Module 6: Fine-Tuning for Layout Analysis

We’ve gone from raw pre-training → classification.

Now we level up to **object detection** tasks — the kind of work that powers document layout analysis, table detection, figure detection, etc. This is where DiT really shows its strength in real Document AI pipelines.

Goal: Use DiT as backbone for detection.

Object detection basics

In object detection we want to:

- Find bounding boxes (location + size)

- Classify what’s inside each box (text, title, table, figure, list…)

- Optionally segment the object (Mask R-CNN does this)

Popular frameworks: Faster R-CNN, Mask R-CNN, Cascade R-CNN, DETR, etc.

They all need a backbone that produces rich feature maps at different scales → usually a Feature Pyramid Network (FPN) on top.

FPN for multi-scale

Real documents have objects at wildly different scales:

- Tiny page numbers

- Medium-sized titles

- Large tables or figures spanning half the page

Standard CNN backbones (ResNet) naturally produce multi-scale features (C2, C3, C4, C5 from different stages).

FPN combines them into a pyramid so the detector sees both fine details and big-picture context.

Why transformer is single-scale

ViT / DiT produces a **single-scale** feature map:

- Input 224×224 → patches → 14×14 grid of embeddings (for base model)

- After Transformer → still 14×14 × hidden_dim

- No natural hierarchy like CNN stages

So if you naively use ViT features directly → small objects (thin lines, small text) get lost because the resolution is too coarse.

How DiT adapts to FPN

It solves this with resolution-modifying modules inserted at different Transformer blocks (Figure 3 shows the idea clearly).

They define:

- Total blocks d (12 for DiT-B, 24 for DiT-L)

- Extract features from intermediate blocks and up/downsample them to build a pyramid

Exact strategy (from section 3.3):

- 1d/3 block (≈4th block) → upsample 4× (two stride-2 transposed convs)

- 1d/2 block (≈6th block) → upsample 2× (one stride-2 transposed conv)

- 2d/3 block (≈8th block) → use as-is (identity)

- 3d/3 block (last block) → downsample 2× (stride-2 max pool)

These four feature maps are fed into FPN → standard Mask R-CNN / Cascade R-CNN head.

Implementation note: They use Detectron2 codebase + custom backbone registration.

Real-world example: Detecting titles, tables, paragraphs in PDFs

Imagine you’re building an intelligent PDF parser for a law firm or research archive in Bhopal:

- Input: Scanned contract / research paper / invoice PDF

- Goal: Automatically extract structured content

DiT + Mask R-CNN can:

- Detect title regions → big bold text blocks

- Detect ables → grid structures (critical for financials)

- Detect paragraphs / text blocks → main body content

- Detect figures → charts, images

Once detected → you can crop + OCR only the relevant parts, feed to layout-aware models, or even build a reading order graph.

This saves huge manual annotation time and makes downstream extraction (key-value pairs, table parsing) much more accurate.

The paper shows big gains on PubLayNet (layout analysis) and ICDAR cTDaR (table detection) — e.g., layout mAP from 91.0 → 94.9.

```python
self.downsample2x = nn.MaxPool2d(2, stride=2) # from last block

p2 = self.upsample4x(feats[0]) # up 4x

"p3": p3.contiguous(),

"p4": p4.contiguous(),

"p5": p5.contiguous(),
```

Module 7 — Table Detection & OCR Text Detection

Now we zoom in on two very specific (and very practical) detection tasks the DiT paper evaluates:

1. Table detection → ICDAR 2019 cTDaR Track A

2. Text detection (word-level bounding boxes for OCR) → FUNSD Task #1

Both are object detection problems, but with their own quirks — especially small objects (words) and structured objects (tables).

Let’s break down the evaluation metrics, challenges, and how DiT performs.

Goal: You can evaluate document detection systems.

IoU (Intersection over Union)

The core metric for bounding box quality.

IoU = Area of overlap / Area of union

Ranges from 0 (no overlap) to 1 (perfect match).

In detection:

- IoU ≥ threshold (e.g. 0.5, 0.6, 0.7…) → true positive

- Lower IoU → false positive or miss

DiT paper uses IoU thresholds of 0.6, 0.7, 0.8, 0.9 for table detection (weighted F1) and IoU@0.5 for FUNSD text detection.

mAP (mean Average Precision)

Standard COCO-style metric for object detection.

- For each class → compute Average Precision (AP) by ranking predictions and plotting precision-recall curve

- mAP = mean AP across all classes (or IoU thresholds)

PubLayNet (layout analysis) uses mAP @ IoU [0.50:0.95] — the hardest setting.

DiT improves layout mAP from 91.0 → 94.9 (with Cascade R-CNN).

F1 score

Harmonic mean of precision and recall:

F1 = 2 × (precision × recall) / (precision + recall)

For table detection (cTDaR), they use a **weighted F1** across multiple IoU thresholds:

wF1 = (0.6 F1₀.₆ + 0.7 F1₀.₇ + 0.8 F1₀.₈ + 0.9 F1₀.₉) / (0.6 + 0.7 + 0.8 + 0.9)

This rewards models that are accurate even at strict IoU levels.

For FUNSD text detection → standard F1 @ IoU=0.5.

Anchor sizes for word-level detection

In anchor-based detectors (Faster/Mask R-CNN), you define a set of **anchor boxes** (pre-defined aspect ratios and scales) at each feature map location.

For word-level text detection (like FUNSD):

- Words are small → need small anchors (e.g. 16×16, 32×32, 64×64)

- Words can be wide (long text) or tall (vertical text in forms) → multiple aspect ratios (e.g. 1:1, 1:2, 2:1, 1:3, 3:1)

Typical config for text detection:

- Scales: [8, 16, 32, 64] (relative to feature stride)

- Aspect ratios: [0.33, 0.5, 1.0, 2.0, 3.0]

Why small objects need different anchors

- Feature maps in FPN are downsampled (stride 4, 8, 16, 32…)

- Small words (e.g. 10–20 pixels) fall on coarse feature maps → lost detail

- Without small anchors → detector misses tiny text or thin table lines

DiT’s multi-scale adaptation (upsampling early blocks) helps here — gives higher-resolution features for small objects.

Real-world example: Detecting line items in scanned receipts

Picture a grocery receipt from a Bhopal kirana store — messy, tilted, small font, many line items.

Tasks:

- Detect each line item row (as a table row / bounding box)

- Detect individual words/numbers (product name, quantity, price) for OCR

DiT + detection head can:

- Find the table region (high IoU on line-item grid)

- Detect small word boxes → feed to OCR engine

- Extract structured data: item → qty → amount

Without good small-object handling → misses prices or product codes → wrong totals.

DiT’s document pre-training + FPN adaptation makes it robust on real receipts (faded ink, folds, handwriting).

```python

import torch

import numpy as np

# 1. IoU between two boxes [x1,y1,x2,y2]

def compute_iou(box1, box2):

# Intersection

x1 = max(box1[0], box2[0])

y1 = max(box1[1], box2[1])

x2 = min(box1[2], box2[2])

y2 = min(box1[3], box2[3])

inter_area = max(0, x2 - x1) * max(0, y2 - y1)

# Union

area1 = (box1[2] - box1[0]) * (box1[3] - box1[1])

area2 = (box2[2] - box2[0]) * (box2[3] - box2[1])

union_area = area1 + area2 - inter_area

return inter_area / union_area if union_area > 0 else 0.0

# 2. Simple F1 @ single IoU threshold

def compute_f1_at_iou(gt_boxes, pred_boxes, scores, iou_thresh=0.5):

# Sort predictions by confidence

order = scores.argsort(descending=True)

pred_boxes = pred_boxes[order]

matched = np.zeros(len(gt_boxes), dtype=bool)

tp, fp = 0, 0

for pred in pred_boxes:

best_iou = 0

best_gt_idx = -1

for i, gt in enumerate(gt_boxes):

if matched[i]: continue

iou = compute_iou(pred, gt)

if iou > best_iou:

best_iou = iou

best_gt_idx = i

if best_iou >= iou_thresh:

tp += 1

matched[best_gt_idx] = True

else:

fp += 1

fn = len(gt_boxes) - tp

precision = tp / (tp + fp) if (tp + fp) > 0 else 0

recall = tp / (tp + fn) if (tp + fn) > 0 else 0

f1 = 2 * precision * recall / (precision + recall) if (precision + recall) > 0 else 0

return precision, recall, f1

# 3. Anchor configuration example (Detectron2 style for text/table detection)

ANCHOR_SIZES = {

"text_detection": [16, 32, 64, 128], # small → medium words

"table_detection": [64, 128, 256, 512], # larger structured regions

}

ANCHOR_ASPECT_RATIOS = {

"text_detection": [0.33, 0.5, 1.0, 2.0, 3.0], # wide or tall words

"table_detection": [0.5, 1.0, 2.0] # more square-ish tables

}

# In Detectron2 config you’d set:

# cfg.MODEL.RPN.ANCHOR_SIZES = ANCHOR_SIZES["text_detection"]

# cfg.MODEL.RPN.ASPECT_RATIOS = ANCHOR_ASPECT_RATIOS["text_detection"]
```

Module 8: Why DiT Outperforms BEiT & MAE

In this Module 8 we do some critical thinking on why DiT actually beats strong baselines like BEiT and MAE on document tasks.

Goal: You understand research-level reasoning.

Domain-specific pretraining advantage

The biggest win: DiT is pretrained on 42 million real document images (IIT-CDIP), while BEiT and MAE use general/natural image data (DALL·E tokenizer for BEiT, natural images for MAE).

- Natural image tokenizers (DALL·E) blur sharp lines, tables, and text edges (Figure 4 reconstruction comparison).

- DiT's custom dVAE tokenizer trained on documents gives sharper, more accurate visual tokens → better MIM learning of document structure (text blocks, grids, alignments).

Results from the different models:

- RVL-CDIP classification:

BEiT-B → 93.1% overall

MAE-B → 93.0%

DiT-B → 93.5%

DiT-L → 94.1% (big jump with larger model + domain data)

- PubLayNet layout mAP (Table 2): DiT pushes from ~91% (baselines) → 94.9%

- ICDAR table detection and **FUNSD text detection**: Similar lifts (94.23 → 96.55 for tables, 93.07 → 94.29 for text)

Why it matters: Documents aren't natural scenes. Domain mismatch kills transfer. DiT closes the gap by speaking the "language" of documents from day one.

Fine-grained detection improvement (IoU@0.9)

Detection tasks punish loose boxes—especially at high IoU thresholds like 0.9 (almost perfect overlap).

DiT shines here because:

- Domain pretraining learns precise boundaries (sharp lines, table cells, word boxes).

- Custom tokenizer preserves fine details → better feature discrimination for tight boxes.

- FPN adaptation (Module 6) + early-block high-res features help small/precise objects.

In ICDAR cTDaR (weighted F1 across 0.6–0.9 IoU): DiT gains most at stricter thresholds.

FUNSD text detection (IoU@0.5 F1): Also improves, but high-IoU gains highlight precision edge.

Critical thinking: General-image pretraining (BEiT/MAE) learns coarse semantics well but struggles with pixel-level precision needed for tables/words. DiT's document tokens encode boundaries better → higher IoU@0.9 recall/precision.

Few-shot capability

The paper doesn't have explicit few-shot tables, but hints at generalization from strong pretraining.

- DiT uses only unlabeled 42M docs → no task-specific labels in pretrain.

- Still beats supervised ImageNet models + other self-supervised ones on small labeled downstream sets (e.g., FUNSD has only 150 train forms).

Implication: Domain-specific self-supervision acts like "free few-shot teacher" — learns general document priors that transfer with little fine-tuning data.

Real-world: In Bhopal offices digitizing old records (few labeled examples), DiT would adapt faster than natural-image models.

Critical thinking: Specific cases from the paper

Why grayscale tokenizer hurt archival subset?

ICDAR 2019 cTDaR has archival (old, hand-drawn, noisy scans) and modern subsets.

Paper notes: Raw archival images → suboptimal performance when fine-tuning DiT → they apply adaptive binarization (OpenCV, Figure 5) to make them black-white.

Reasoning:

- DiT's dVAE trained mostly on modern-ish grayscale docs → color/contrast variations in old scans confuse tokenizer.

- Binarization normalizes to clean black-white → aligns better with learned tokens (sharp edges, no noise gradients).

- Without it: Domain shift inside documents → lower table detection F1 on archival.

Lesson: Even domain-specific pretraining isn't perfect—preprocessing still matters for extreme sub-domains.

Why synthetic data improves FUNSD?

FUNSD is noisy scanned forms (only 150 train, 49 test).

Paper mentions synthetic augmentation: They generate 1M synthetic document images → pretrain or fine-tune further → big boost on FUNSD text detection.

Reasoning:

- FUNSD has limited real data → overfitting risk.

- Synthetic docs (varied layouts, fonts, noise) increase diversity → better generalization to noisy scans.

- Combined with DiT's strong backbone → pushes F1 from ~93% (baselines) → 94.29%.

Critical: Self-supervised + synthetic data = powerful data-efficient recipe when real labeled data is scarce.

Engineering details: Making it production-ready

1. Memory optimization

- Gradient checkpointing: recompute activations during backward → saves ~30–50% VRAM

- Activation checkpointing in Transformer blocks

- Patch size 16 → 14×14=196 tokens (small)

- Use torch.utils.checkpoint

2. Mixed precision (AMP)

- torch.cuda.amp → fp16/bf16 training → 2× speed, half memory

- DiT paper uses fp32 likely, but modern runs use AMP

3. Batch scaling

- Paper: batch 2048 (needs ~100+ GB VRAM total)

- Realistic: gradient accumulation (effective batch 2048 with micro-batch 32)

- Or DDP + multiple GPUs

4. Distributed training

- torch.distributed + DDP (DistributedDataParallel)

- For 4–8 GPUs: sync gradients, scale LR by world_size

```python
# 1. Mini DiT Model (ViT-Base like + MIM head)

class MiniDiT(nn.Module):

def __init__(self, img_size=224, patch_size=16, in_chans=3, embed_dim=768,

depth=12, num_heads=12, mlp_ratio=4., codebook_size=8192):

super().__init__()

self.patch_embed = nn.Conv2d(in_chans, embed_dim, kernel_size=patch_size, stride=patch_size)

num_patches = (img_size // patch_size) ** 2

self.pos_embed = nn.Parameter(torch.zeros(1, num_patches, embed_dim))

self.blocks = nn.ModuleList([

nn.TransformerEncoderLayer(embed_dim, num_heads, dim_feedforward=embed_dim*mlp_ratio,

dropout=0., activation="gelu", batch_first=True)

for _ in range(depth)

])

self.norm = nn.LayerNorm(embed_dim)

self.mim_head = nn.Linear(embed_dim, codebook_size) # predict token ids

def forward(self, x, mask_ratio=0.4):

B = x.shape[0]

x = self.patch_embed(x).flatten(2).transpose(1, 2) # B, N, D

x = x + self.pos_embed

# Simple random masking (for demo; paper uses blockwise)

N = x.shape[1]

len_keep = int(N * (1 - mask_ratio))

noise = torch.rand(B, N, device=x.device)

ids_shuffle = torch.argsort(noise, dim=1)

ids_restore = torch.argsort(ids_shuffle, dim=1)

ids_keep = ids_shuffle[:, :len_keep]

x_masked = torch.gather(x, dim=1, index=ids_keep.unsqueeze(-1).repeat(1,1,x.shape[-1]))

# Add [MASK] embedding or zero (simplified)

mask_tokens = torch.zeros(B, N - len_keep, x.shape[-1], device=x.device)

x_ = torch.cat([x_masked, mask_tokens], dim=1)

x_ = torch.gather(x_, dim=1, index=ids_restore.unsqueeze(-1).repeat(1,1,x.shape[-1]))

for blk in self.blocks:

# Optional: gradient checkpointing

# x_ = torch.utils.checkpoint.checkpoint(blk, x_)

x_ = blk(x_)

x_ = self.norm(x_)

logits = self.mim_head(x_) # B, N, codebook_size

return logits, ids_restore, len_keep # for loss masking
```

Module 10 — Extending DiT

By we now seen the full lifecycle: motivation → architecture → pre-training → fine-tuning → evaluation → why it wins → practical reproduction.

In this closing module we look forward, exactly as the DiT authors hint in their conclusion and related work sections. They position DiT not as an endpoint, but as a strong vision-only backbone that can be the foundation for the next generation of multimodal Document AI.

Goal: Forward-thinking AI system design.

We’ll cover the most promising extensions the paper implicitly or explicitly points toward.

Multimodal pretraining (LayoutLM style)

DiT is pure vision — it only sees pixels (or discrete visual tokens), no text embeddings yet.

The authors repeatedly say:

> "it can be further leveraged for the multimodal pre-training for Document AI."

This means the natural next step is to combine DiT with LayoutLM-style models (LayoutLM, LayoutLMv2/v3, DocFormer, etc.).

How it would work:

- Run DiT to get rich visual patch embeddings (position-aware, layout-sensitive)

- Run OCR/layout engine → get text tokens + bounding boxes

- Fuse them in a joint Transformer: visual tokens + text tokens + 2D position embeddings

- Pre-train with multimodal objectives: MLM + MIM + image-text alignment + layout prediction

This is already the direction post-2022 (e.g. LayoutLMv3 uses vision backbone + text + layout, but with weaker vision pretraining than DiT).

DiT would bring **stronger visual understanding** (especially layout, tables, borders) into these multimodal models.

Unified CV + NLP Transformer

Instead of separate vision (DiT) and language (BERT/RoBERTa) encoders → one single Transformer that processes both visual patches and text tokens natively.

Benefits the paper hints at:

- End-to-end differentiable (no frozen OCR)

- Better cross-modal reasoning (e.g. “find the total amount in the table” understands both layout and semantics)

- Unified pre-training recipe: MIM + MLM + contrastive + maybe bounding-box regression

This aligns with unified models like Unified-IO, Pix2Struct, Donut (which does OCR-free document understanding end-to-end).

DiT’s document-specific MIM pretraining would make such a unified model much stronger on visually-rich documents than starting from natural-image ViT.

Scaling dataset beyond 42M

IIT-CDIP = 42 million pages — already huge for 2022.

Future:

- Scale to hundreds of millions / billions of document pages

Sources: Common Crawl PDFs, public scanned books (Google Books subset), enterprise archives, synthetic generation at scale (as they already did 1M synthetics for FUNSD boost)

- Mix real + synthetic + weakly-labeled (e.g. auto-annotated from PDFs with embedded text)

- Curriculum learning: start with clean → noisy → handwritten

Larger scale + longer training (DiT did 500k steps) → even better generalization, especially on rare formats (handwritten forms, multilingual docs, ancient manuscripts).

Combining text tokens + visual tokens

The paper closes with: DiT can support variety of Document AI tasks and be leveraged for multimodal.

Key idea: Treat visual tokens (from DiT’s dVAE) and text tokens (from OCR / LayoutLM tokenizer) as the same vocabulary space in some shared embedding layer.

Or: Keep separate but add cross-attention layers.

This enables:

- Zero-shot / few-shot document QA

- Visual grounding (“circle the invoice number”)

- Better table structure recognition (visual + semantic)

Real-world future: AI system that

Imagine a single, deployed Document AI agent in 2026–2028 that:

1. Reads invoice— takes photo/scan/PDF (multi-page, skewed, low-res, multilingual)

2. Understands layout — uses DiT-style vision backbone to detect regions (title, table, line items, totals, vendor logo)

3. Extracts entities — fuses visual + text → pulls key-value pairs (date, amount, GSTIN, line-item descriptions)

4. Answers queries — natural language: “What’s the total before tax?”, “Is this invoice duplicate?”, “Compare this receipt with last month’s”

Architecture sketch:

- Input → DiT-large (vision) + lightweight OCR → visual tokens + text tokens + boxes

- Unified multimodal Transformer (pretrained on huge mixed data) → does understanding + generation

- Output: structured JSON + explanations + confidence scores

This is already emerging in tools like Azure Document Intelligence, Google Document AI, but DiT-like vision pretraining would push accuracy on edge cases (noisy scans, complex layouts) much higher.

Outcome: Forward-thinking AI system design.

DiT was never meant to be the final model — it was a missing piece: a truly strong, self-supervised vision backbone for documents.

The future directions the authors point to — multimodal fusion, unified transformers, massive scaling, visual + text token synergy — are exactly what happened in 2023–2025 (UDOP, Nougat, DocLLM, LayoutLMv3+, etc.).
