Module 2 — The Three Pillars of Document Understanding

Text + Vision + Spatial Intelligence

Module Goal:

By the end of this module, you will understand exactly how DocFormer (and modern Document AI models) solve the “reading” puzzle. You’ll see how Text + Vision + Spatial features work together — the same way your brain instantly understands an invoice, receipt, or form. No more treating documents like plain text files.

1. Textual Features: The “What”

This is the semantic layer — the actual meaning of the words.

DocFormer uses a powerful Transformer tokenizer (based on BERT/RoBERTa) to turn raw text into rich embeddings.

The Problem

If you only had text:

"Total:" and "₹75" are just two random strings with no relationship.

The DocFormer Solution

The model learns that "Total:" is a financial keyword and "₹75" is a currency value. It understands context, synonyms, and financial language.

LIGHTBULB Tip

Text features answer “What does it say?”

But alone, they are blind to layout and appearance — exactly why plain BERT fails on documents.

2. Spatial Features: The “Where” (The Secret Sauce)

This is what makes Document AI magical.

Every single word/token on the page gets a physical address — a bounding box.

The Coordinate System

DocFormer sees a map, not a list:

  • x_min, y_min → Top-left corner
  • x_max, y_max → Bottom-right corner
  • Width & Height → Size/scale of the text

Why this matters

  • “Date” at top-right + “12/05/2026” right next to it → related (Key-Value pair)
  • Same words but on opposite sides of the page → unrelated

Real-World Analogy

Think of a restaurant menu.

“Chef’s Special” at the top in big font + price right beside it = you instantly know they belong together.

Your brain uses spatial intelligence automatically. DocFormer does the same with coordinates.

3. Visual Features: The “Look”

Text tells you content. Visuals tell you the importance.

DocFormer runs a CNN (ResNet50 backbone) directly on the document image to extract:

  • Boldness / font weight → headers or key fields
  • Lines & grids → tables
  • Logos, stamps, colors → “PAID” stamp, company logo, red warning text

INFO Note

Visual features are extracted from the actual pixels — so the model can “see through” light stamps or understand that bold + larger text = more important.

2. The Core Problem: The "Fusion" Gap

Multi-modal learning fails without proper alignment.

  • Text → Discrete (words/tokens)
  • Images → Continuous (pixels)
  • Spatial → Geometric (coordinates)

Older models used Late Fusion (mix at the end — like adding ingredients after cooking).

DocFormer’s breakthrough: Early & Continuous Fusion (mix in every Transformer layer, so modalities "talk" throughout).

Why Fusion Matters

Without it, the model can’t answer: “Is this bold red ‘Total’ at bottom-right related to the number nearby?”

DocFormer uses cross-attention to let text “query” visuals via spatial coords — deep, interactive understanding.

INFO Note

This gap explains why classic models flop on "in-the-wild" docs: stamps overlap text, tables break sequences, layouts vary wildly.

from transformers import LayoutLMTokenizer, LayoutLMModel # For old way

# --- THE OLD WAY (LayoutLM v1 Style) ---

# Image features pre-extracted & simply "added" to text

bbox = torch.tensor([[800, 900, 850, 920], [860, 900, 920, 920]]) # Spatial

text_embeddings = model.embeddings(input_ids=input_ids)

spatial_embeddings = model.embeddings.position_embeddings_2d(bbox) # Simplified

# Image was separate (e.g., from Faster R-CNN crops)

image_crops = torch.randn(2, 3, 224, 224) # Placeholder crops

image_features = torch.randn(2, 768) # Pre-extracted

old_input = text_embeddings + spatial_embeddings + image_features # Simple addition! Weak fusion.

# --- THE DOCFORMER WAY (Simplified Concept) ---

# Uses Multi-modal Self-Attention (MSA) for deep fusion

class DocFormerFusion(torch.nn.Module):

def __init__(self):

super().__init__()

self.query_layer = torch.nn.Linear(768, 768)

self.key_layer = torch.nn.Linear(768, 768)

def forward(self, text_feat, visual_feat, spatial_feat):

# Cross-attention: Text "queries" visuals via spatial

fused_query = self.query_layer(text_feat + spatial_feat) # Text guided by position

fused_key = self.key_layer(visual_feat + spatial_feat) # Visuals guided by position

# Attention score: "Does this word match bold pixels at these coords?"

attention_score = torch.matmul(fused_query, fused_key.transpose(-1, -2))

return attention_score

# Example usage

visual_feat = torch.randn(2, 768) # From ResNet CNN

spatial_feat = torch.randn(2, 768) # From bbox embeddings

fused_output = fusion_model(text_feat, visual_feat, spatial_feat)

print(f"Fused Attention Shape: {fused_output.shape}") # torch.Size([2, 2])

Line-by-Line Explanation

  • Old Way → Simple + addition. Hopes text ("Apple") aligns with image (logo) by chance — no real interaction.
  • DocFormer Way → Cross-attention creates a "shared space." Text queries: "Is 'Total' in a bold visual box at these spatial coords?" This enables true multi-modal reasoning.

Prep bounding boxes for shared use across modalities.

import torch.nn as nn

class SharedSpatialEmbeddings(nn.Module):

def __init__(self, hidden_size=768, max_position=1000):

super().__init__()

# Shared embeddings for ALL coords — text or visual

self.x_embeddings = nn.Embedding(max_position, hidden_size)

self.y_embeddings = nn.Embedding(max_position, hidden_size)

self.w_embeddings = nn.Embedding(max_position, hidden_size)

self.h_embeddings = nn.Embedding(max_position, hidden_size)

def forward(self, bboxes):

# bboxes: [batch, seq_len, 4] (x1, y1, x2, y2)

w = bboxes[:, :, 2] - bboxes[:, :, 0]

h = bboxes[:, :, 3] - bboxes[:, :, 1]

# Pull from SHARED tables — same math for words/patches

emb_x1 = self.x_embeddings(bboxes[:, :, 0])

emb_y1 = self.y_embeddings(bboxes[:, :, 1])

emb_x2 = self.x_embeddings(bboxes[:, :, 2])

emb_y2 = self.y_embeddings(bboxes[:, :, 3])

emb_w = self.w_embeddings(w)

emb_h = self.h_embeddings(h)

# Sum to unique spatial "signature"

return emb_x1 + emb_y1 + emb_x2 + emb_y2 + emb_w + emb_h

Line-by-Line Explanation

  • max_position=1000 → Normalize coords to 1000x1000 grid (resolution-independent, handles any doc size)
  • Width/Height calc → Adds scale info (big text = headers)
  • Sum embeddings → Rich, fused position vector for attention

Module 5 — Multi-Modal Self-Attention

The Real Innovation

Document AI Masterclass

Module Goal:

By the end of this module, you will understand why standard Transformer attention fails on documents and exactly how DocFormer’s multi-modal attention solves it. You’ll see the leap from single-modality QKᵀ to a true multi-lingual debate between text, vision, and spatial intelligence — the “consciousness” that makes DocFormer read documents like a human.

1. From QKᵀ to Multi-Modal Attention

Standard Transformer attention (BERT, GPT, etc.):

This works beautifully for 1D sequences — but documents are 2D layouts.

The Blind Spot

It only asks: “How semantically similar are word A and word B?”

It never asks: “How close are they on the page? Are they both bold? Is one inside a table cell?”

DocFormer’s Upgrade

It injects spatial bias and visual context directly into the attention scores.

Now the question becomes:

“How similar are word A and word B given they are 20 pixels apart, share the same y-coordinate, and both appear bold in the image?”

LIGHTBULB Tip

Standard attention = conversation between words.

DocFormer attention = debate between words, pixels, and coordinates — every participant gets a vote.

2. The “Discrete Multi-modal” Strategy

Most earlier models (LayoutLMv2, etc.) used Joint Modeling:

  • Concatenate text tokens + image patch tokens into one long sequence
  • Feed to a single Transformer
  • Hope the model figures out relationships

Problems with Joint Modeling

  • Very long sequences → memory & speed explosion
  • Modalities are “sitting next to each other” but not deeply interacting
  • Hard to preserve modality-specific information

DocFormer’s Discrete Multi-modal Approach (much smarter)

  1. Text Branch — self-attention only on text tokens + shared spatial bias injected every layer
  2. Vision Branch — self-attention only on image patch tokens + same shared spatial bias
  3. Fusion — sum (or weighted combine) the outputs of both branches

Why this wins

  • Each modality gets clean, focused computation
  • Shared spatial bias acts as the “anchor” that keeps branches aligned
  • Final fusion is simple but powerful because both branches already “speak geometry”

Real-World Analogy

Joint modeling = forcing English speakers and French speakers to sit in one long line and talk.

Discrete multi-modal = two separate rooms (text & vision), but both rooms use the same map (shared spatial embeddings) — then combine the best insights from both rooms.

3. The Power of Relative Position Bias (RPB)

Absolute coordinates are fragile.

If the entire document is shifted 10 pixels right (common in phone scans), absolute (x,y) values change — but relationships stay identical.

Solution: Relative Position Bias

Instead of absolute positions, DocFormer adds a learned bias based on distance & direction between any two tokens.

Examples:

  • Token A directly above Token B → positive bias (likely Key-Value pair)
  • Token A far left of Token B → negative or neutral bias
  • Same row, close x-distance → strong positive bias (column alignment)

Result

The model becomes translation-invariant — it understands layout structure even if the scan is rotated slightly, cropped, or shifted.

INFO Note

Relative bias is added directly to the QKᵀ matrix before softmax — subtle but extremely powerful for tables, forms, invoices.

🚀 The Engineering Deep Dive: The Code

Here’s a simplified but faithful version of DocFormer-style multi-modal attention

import torch

import torch.nn as nn

import torch.nn.functional as F

class MultiModalAttention(nn.Module):

def __init__(self, d_model, num_heads):

super().__init__()

self.num_heads = num_heads

self.d_k = d_model // num_heads

# Separate projections — keep modalities distinct

self.q_text = nn.Linear(d_model, d_model)

self.k_text = nn.Linear(d_model, d_model)

self.v_text = nn.Linear(d_model, d_model)

self.q_vis = nn.Linear(d_model, d_model)

self.k_vis = nn.Linear(d_model, d_model)

self.v_vis = nn.Linear(d_model, d_model)

def forward(self, text_feat, vis_feat, spatial_bias):

# 1. Text branch: self-attention + spatial bias

q_t = self.q_text(text_feat) # [B, L_text, D]

k_t = self.k_text(text_feat)

v_t = self.v_text(text_feat)

scores_t = torch.matmul(q_t, k_t.transpose(-2, -1)) / (self.d_k ** 0.5)

attn_t = F.softmax(scores_t + spatial_bias, dim=-1)

out_text = torch.matmul(attn_t, v_t) # [B, L_text, D]

# 2. Vision branch: self-attention + same spatial bias

q_v = self.q_vis(vis_feat) # [B, L_vis, D]

k_v = self.k_vis(vis_feat)

v_v = self.v_vis(vis_feat)

scores_v = torch.matmul(q_v, k_v.transpose(-2, -1)) / (self.d_k ** 0.5)

attn_v = F.softmax(scores_v + spatial_bias, dim=-1)

out_vis = torch.matmul(attn_v, v_v) # [B, L_vis, D]

# 3. Fusion: sum outputs (DocFormer style — simple & effective)

# In practice: modalities often have same sequence length after patching

fused_output = out_text + out_vis

return fused_output

Key Takeaways from the Code

  • Separate Q/K/V projections per modality → clean, modality-specific attention
  • spatial_bias added to both branches → shared geometry anchors everything
  • Final + fusion → modalities vote together after reasoning independently
  • In real DocFormer: cross-attention layers and more sophisticated fusion exist, but this pattern is the core idea

Why Self-Supervised Pretraining?

Labeling every invoice, receipt, form, and table in the world is impossible.

Instead, DocFormer plays self-supervised games — tasks where the data itself provides the supervision.

The model has to “guess” hidden parts of the document using the other visible parts → forcing multi-modal reasoning.

1. Multi-Modal Masked Language Modeling (MM-MLM)

Classic BERT: mask 15% of words → predict from context.

DocFormer: mask text tokens but leave the image untouched.

Setup Example

Text: “Item … Subtotal … Total … ₹1,250”

→ Mask “Total”

Image: still shows bold text at bottom-right + underline

The Intelligence

Model sees:

  • Bold, dark patch + underline in image
  • Spatial position: bottom-right, large font scale → Even without the word “Total”, it predicts “Total” / “Grand Total” / “Amount Due”

LIGHTBULB Tip

This teaches the text branch to trust visual and spatial cues when OCR fails — exactly what happens in real scanned/handwritten documents.

2. Learn To Reconstruct (LTR) — DocFormer Original

Most models ignore pixels after feature extraction.

DocFormer asks: “Can you redraw the document from your understanding?”

The Task

  • Take fused features (Text + Vision + Spatial)
  • Pass through a lightweight decoder
  • Reconstruct original image pixels (pixel regression)

Why it’s powerful

To redraw lines, boxes, logos, tables accurately, the model must deeply understand:

  • Exact placement of every element
  • Visual style (boldness, thickness)
  • Alignment between digital concept and visual reality

INFO Note

This is like forcing the model to “explain the document back to you in pixels” — extremely strong supervision for layout understanding.

3. Text Describes Image (TDI) — Global Intelligence

Local tasks (MM-MLM, LTR) are great for details.

TDI looks at the big picture.

Setup

  • Positive pair: Receipt text + Receipt image → label = 1 (match)
  • Negative pair: Receipt text + Loan Agreement image → label = 0 (mismatch)

The Goal

Binary classification: Does this text belong to this image?

Why it works

Forces the model to learn global characteristics:

  • Receipts → narrow, list-heavy, many small numbers
  • Contracts → dense paragraphs, signatures at bottom
  • Invoices → tables + totals at bottom-right

Real-World Analogy

You can tell a grocery receipt from a legal contract even if you cover all the words — just from layout “vibe”. TDI teaches the model the same skill.

🚀 The Engineering Deep Dive: The Code

Simplified multi-task pretraining step in PyTorch style.

import torch.nn.functional as F

def pretraining_step(model, batch):

"""

One forward + loss computation for all three objectives.

batch contains: input_ids, image, bboxes, labels (masked), is_match

"""

# Forward pass through DocFormer

outputs = model(

input_ids=batch['input_ids'],

image=batch['image'],

bboxes=batch['bboxes']

)

# 1. MM-MLM: only compute loss on masked tokens

mlm_logits = outputs.text_logits # [B, seq_len, vocab_size]

mlm_loss = F.cross_entropy(

mlm_logits.view(-1, mlm_logits.size(-1)),

batch['labels'].view(-1),

ignore_index=-100 # non-masked tokens

)

# 2. LTR: pixel reconstruction (MSE)

reconstructed = model.reconstruction_head(outputs.fused_features)

ltr_loss = F.mse_loss(reconstructed, batch['image']) # [B, 3, H, W]

# 3. TDI: global match (binary classification)

tdi_logits = model.tdi_head(outputs.global_pooled) # [B, 1]

tdi_loss = F.binary_cross_entropy_with_logits(

tdi_logits.squeeze(-1),

batch['is_match'].float()

)

# Multi-task total loss (simple sum — can be weighted)

total_loss = mlm_loss + ltr_loss + tdi_loss

return {

'total_loss': total_loss,

'mlm_loss': mlm_loss,

'ltr_loss': ltr_loss,

'tdi_loss': tdi_loss

}

Key Takeaways from the Code

  • Each task has its own head → modular and easy to balance
  • Losses computed independently but gradients flow through shared backbone
  • ignore_index=-100 in MLM → standard masked modeling trick
  • Simple sum works surprisingly well; production often adds task weighting

import torch

import torch.nn as nn

from transformers import DocFormerModel, DocFormerConfig

class DocFormerForTokenClassification(nn.Module):

def __init__(self, num_labels: int = 4): # e.g., Header, Question, Answer, Other

super().__init__()

# Load pretrained backbone

self.docformer = DocFormerModel.from_pretrained("microsoft/docformer-base")

# Task-specific head

self.dropout = nn.Dropout(0.1)

self.classifier = nn.Linear(self.docformer.config.hidden_size, num_labels)

def forward(

self,

input_ids,

image,

bboxes,

attention_mask=None,

labels=None

):

# 1. Get fused multi-modal features

outputs = self.docformer(

input_ids=input_ids,

image=image,

bbox=bboxes,

attention_mask=attention_mask

)

sequence_output = outputs.last_hidden_state # [batch, seq_len, hidden_size]

# 2. Classify every token

sequence_output = self.dropout(sequence_output)

logits = self.classifier(sequence_output) # [batch, seq_len, num_labels]

loss = None

if labels is not None:

loss_fct = nn.CrossEntropyLoss(ignore_index=-100)

loss = loss_fct(

logits.view(-1, self.classifier.out_features),

labels.view(-1)

)

return {"loss": loss, "logits": logits} if loss is not None else {"logits": logits}

2. Step 1: Shared Spatial Embeddings

The foundation of layout understanding: one embedding space for coordinates — whether they come from OCR words or image patches.

import torch

import torch.nn as nn

class SharedSpatialEmbeddings(nn.Module):

def __init__(self, config):

super().__init__()

max_pos = config['max_2d_position'] # usually 1000

dim = config['coord_dim'] # usually 64–128

hidden = config['hidden_size'] # usually 768

self.x_emb = nn.Embedding(max_pos, dim)

self.y_emb = nn.Embedding(max_pos, dim)

# Project concatenated [x1,y1,x2,y2] → model dimension

self.spatial_projection = nn.Linear(dim * 4, hidden)

def forward(self, bboxes):

# bboxes: [batch, seq_len, 4] → (x1, y1, x2, y2)

# All coords already normalized [0, 1000)

x1, y1, x2, y2 = bboxes.unbind(dim=-1) # each [B, L]

emb_x1 = self.x_emb(x1.long())

emb_y1 = self.y_emb(y1.long())

emb_x2 = self.x_emb(x2.long())

emb_y2 = self.y_emb(y2.long())

# Concat → project

spatial = torch.cat([emb_x1, emb_y1, emb_x2, emb_y2], dim=-1)

return self.spatial_projection(spatial) # [B, L, hidden_size]

LIGHTBULB Tip

Using the same embedding tables for x and y coordinates across text and vision is what allows the model to treat a word and a visual patch at the same location as related concepts.

3. Step 2: Custom Multi-Modal Self-Attention with Spatial Bias

Standard attention lacks layout awareness.

We inject a spatial bias directly into the attention scores.

class DocFormerAttention(nn.Module):

def __init__(self, d_model, n_heads):

super().__init__()

self.n_heads = n_heads

self.d_k = d_model // n_heads

self.q_proj = nn.Linear(d_model, d_model)

self.k_proj = nn.Linear(d_model, d_model)

self.v_proj = nn.Linear(d_model, d_model)

def forward(self, x, spatial_features):

B, L, D = x.shape

# Project to Q, K, V

q = self.q_proj(x).view(B, L, self.n_heads, self.d_k).transpose(1, 2)

k = self.k_proj(x).view(B, L, self.n_heads, self.d_k).transpose(1, 2)

v = self.v_proj(x).view(B, L, self.n_heads, self.d_k).transpose(1, 2)

# Standard attention scores

scores = torch.matmul(q, k.transpose(-2, -1)) / (self.d_k ** 0.5)

# Spatial bias: dot-product similarity between spatial embeddings

# → tokens that are close in layout get boosted

spatial_bias = torch.matmul(spatial_features, spatial_features.transpose(-1, -2))

spatial_bias = spatial_bias.unsqueeze(1) # [B, 1, L, L] for broadcasting

# Inject bias before softmax

scores = scores + spatial_bias

attn_probs = torch.softmax(scores, dim=-1)

out = torch.matmul(attn_probs, v) # [B, n_heads, L, d_k]

out = out.transpose(1, 2).contiguous().view(B, L, D)

return out

INFO Note

The spatial bias here is very simple (dot-product). Real DocFormer uses more sophisticated relative position biases (like T5 or LayoutLMv3), but this pattern already captures most of the layout magic.

4. Step 3: Simplified DocFormer Model

Putting it all together — vision path + text path + shared spatial + custom attention.

import torchvision.models as models

class SimplifiedDocFormer(nn.Module):

def __init__(self, config):

super().__init__()

# Vision backbone: ResNet50 without final layers

resnet = models.resnet50(pretrained=True)

self.visual_backbone = nn.Sequential(*list(resnet.children())[:-2])

self.vis_proj = nn.Linear(2048, config['hidden_size'])

# Text embedding (in production → use LayoutLM tokenizer + weights)

self.text_emb = nn.Embedding(config['vocab_size'], config['hidden_size'])

# Shared spatial

self.spatial_layer = SharedSpatialEmbeddings(config)

# Custom attention (one layer — stack many in real model)

self.attention = DocFormerAttention(

config['hidden_size'],

config['n_heads']

)

def forward(self, text_ids, image, bboxes):

# Vision path

vis = self.visual_backbone(image) # [B, 2048, H/32, W/32]

vis = vis.flatten(2).transpose(1, 2) # [B, num_patches, 2048]

vis = self.vis_proj(vis) # [B, num_patches, D]

# Text path

text = self.text_emb(text_ids) # [B, text_len, D]

# Spatial for both

spatial = self.spatial_layer(bboxes) # [B, total_len, D]

# Simple concat (real models use better fusion)

combined = torch.cat([text, vis], dim=1) # [B, text_len + patches, D]

# Apply attention with spatial awareness

output = self.attention(combined, spatial)

return output

Important

This is one attention layer.

In real DocFormer you stack 6–12 such layers + feed-forward + layer-norm after each.

def training_step(model, batch, lambdas=(1.0, 0.5, 0.1)):

2. Memory Optimization: Training at Scale

Going from 1,000 labeled documents to 1,000,000+ unlabeled ones turns VRAM into the main bottleneck.

DocFormer teams use these three battle-tested techniques:

A. Mixed Precision Training (FP16 / BF16)

  • Use 16-bit instead of 32-bit floats
  • Memory usage ↓ ~50%
  • Training speed ↑ 2–3× on Ampere+ GPUs (A100, H100, RTX 30/40 series)

DocFormer Context

Visual branch (ResNet) + text branch + spatial embeddings → many large activations.

FP16/BF16 keeps everything in cache longer → huge throughput gain.

B. Gradient Checkpointing

  • Don’t store all intermediate activations during forward pass
  • Re-compute them on-the-fly during backward pass
  • Memory ↓ 60–70% for deep networks (ResNet + 12-layer transformer)
  • Tradeoff: ~20–40% slower forward/backward, but enables 4–8× larger batch size

C. Gradient Accumulation

  • Simulate large batches on small GPUs
  • Accumulate gradients over 4–16 micro-steps → effective batch size 128–512 even on 24 GB cards

Real-World Analogy

Mixed precision = using half the ingredients but still cooking the same meal.

Gradient checkpointing = forgetting the recipe midway and re-reading it — slower but uses way less counter space.

🚀 Implementation: Scaling the Loss & Precision

Production-grade training loop with mixed precision + gradient scaling.

import torch

from torch.cuda.amp import GradScaler, autocast

from torch.nn.utils import clip_grad_norm_

# 1. Mixed precision infrastructure

scaler = GradScaler()

optimizer = torch.optim.AdamW(model.parameters(), lr=5e-5)

max_grad_norm = 1.0

def train_step(model, batch):

optimizer.zero_grad(set_to_none=True) # faster than zero_grad()

# 2. Forward in mixed precision (FP16/BF16)

with autocast():

outputs = model(

input_ids=batch['input_ids'],

image=batch['image'],

bboxes=batch['bboxes']

)

# 3. Weighted multi-task loss (dynamic or fixed weights)

loss = (

1.00 * outputs.mlm_loss + # Text understanding — highest priority

0.70 * outputs.ltr_loss + # Layout & visual alignment

0.15 * outputs.tdi_loss # Global document type

)

# 4. Backward with loss scaling

scaler.scale(loss).backward()

# 5. Unscale → clip gradients → prevent explosion

scaler.unscale_(optimizer)

clip_grad_norm_(model.parameters(), max_norm=max_grad_norm)

# 6. Optimizer step & update scaler

scaler.step(optimizer)

scaler.update()

return loss.item()

Key Production Notes

  • set_to_none=True in zero_grad() → faster memory release
  • Gradient clipping essential — visual branch gradients can be 10–50× larger than text
  • Start with weights [1.0, 0.7, 0.15] → tune via validation loss curves
  • Use torch.compile(model) (PyTorch 2.0+) for extra 15–40% speedup on modern GPUs
